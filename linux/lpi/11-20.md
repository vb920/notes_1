Chapter 11 - System Limits

UNIX systems expose many **implementation-dependent limits and options**, so portable programs must not assume fixed values for things like open files, pathname length, argument size, realtime signals, or page size.

Some limits are fixed by **hardware/compiler architecture**, such as the maximum value of an `int`, and are usually available through headers like `<limits.h>`.

Other limits are controlled by the **kernel configuration or runtime resource limits**, so they must be queried while the program is running.

SUSv3 defines minimum guaranteed values using constants like `_POSIX_OPEN_MAX`, `_POSIX_NAME_MAX`, and `_POSIX_ARG_MAX`.

These `_POSIX_*` values are portability baselines, not necessarily the real limits of the current machine.

The actual implementation limit may be higher, such as Linux allowing many more message queue priorities than the POSIX minimum.

For system-wide or process-related limits, applications use:

sysconf(_SC_OPEN_MAX);
sysconf(_SC_ARG_MAX);
sysconf(_SC_PAGESIZE);

`sysconf()` asks the running system for limits such as max open file descriptors, page size, realtime signal count, or argument/environment size.

For filesystem-dependent limits, applications use `pathconf()` or `fpathconf()`.

This matters because limits like `NAME_MAX`, `PATH_MAX`, and `PIPE_BUF` may vary between filesystems.

For example, one filesystem may allow 14-byte names, while Linux-native filesystems commonly allow 255-byte filenames.

SUSv3 divides limits into **runtime invariant**, **pathname variable**, and **runtime increasable** limits.

Runtime invariant limits are stable for an implementation, but may still be queried using `sysconf()` if not known at compile time.

Pathname variable limits depend on the filesystem and are queried using a path or file descriptor.

Runtime increasable limits have a POSIX minimum but may be raised by kernel configuration or resource-limit mechanisms.

On Linux, some values returned by `sysconf()` can change if the process changes resource limits using `setrlimit()`.

Important examples include `RLIMIT_NOFILE` affecting `_SC_OPEN_MAX`, and `RLIMIT_STACK` affecting `_SC_ARG_MAX`.

If a limit is indeterminate, robust software should either fall back to POSIX minimums, retry the failing syscall, or dynamically allocate and grow buffers.

At kernel depth, this chapter teaches that limits are not abstract constants; they reflect **kernel resource accounting, filesystem semantics, process limits, ABI design, and POSIX portability contracts**.


Chapter 12 - System and Process Information

The chapter explains how Linux exposes **system and process information** mainly through the **`/proc` virtual filesystem** and the `uname()` system call.

`/proc` is not a disk-backed filesystem; it is generated dynamically by the **kernel** when processes read or write its files.

This avoids older UNIX approaches where privileged tools had to inspect raw kernel memory and depend on unstable kernel data structures.

For every process, Linux creates a directory `/proc/PID`, containing kernel-maintained state about that process.

Files like `/proc/PID/status` expose process metadata such as PID, PPID, UID/GID, memory usage, signal masks, capabilities, CPU affinity, and context-switch counts.

Because `/proc/PID/status` changes across kernel versions, programs should parse it defensively by field name, not by fixed line number.

`/proc/PID/fd` exposes each open file descriptor as a symbolic link, making process file usage inspectable from user space.

`/proc/self` is a convenience symlink that lets a process access its own `/proc/PID` directory without knowing its PID.

Linux also represents threads under `/proc/PID/task/TID`, reflecting the kernel’s thread-group model used for POSIX threads.

System-wide kernel information is available under `/proc`, including memory, mounts, locks, networking, IPC, and kernel tunables.

Directories like `/proc/sys/kernel`, `/proc/sys/vm`, `/proc/sys/net`, and `/proc/sys/fs` expose configurable kernel parameters.

Some `/proc` files are read-only, while others can be written by privileged users to change kernel behavior, such as `/proc/sys/kernel/pid_max`.

A key kernel detail is that `/proc/PID` entries are volatile: they appear when a process is created and disappear immediately when it exits.

Therefore, tools using `/proc` must handle races where a process vanishes between discovering and opening its files.

The `uname()` system call returns system identity information such as kernel name, release, version, machine architecture, hostname, and Linux-specific NIS domain name.

Overall, `/proc` acts as a kernel-user-space introspection interface, exposing live kernel state through ordinary file I/O while hiding internal kernel data structures.

Chapter 13 - File I/O buffering

This chapter explains that Linux file I/O performance is dominated by **buffering at two layers**: the **kernel page/buffer cache** and the **stdio user-space buffer**.

For disk files, `read()` and `write()` usually do **not directly access disk**; they copy data between user space and the kernel’s page cache.

A `write()` returns after copying data into kernel memory, while the actual disk write happens later through kernel writeback.

Similarly, `read()` often returns data already present in the page cache instead of performing immediate disk I/O.

This makes system calls fast and reduces expensive disk operations, but it also means file data may not be durable immediately after `write()`.

Modern Linux uses the **page cache**, not a separate historical buffer cache, to cache file data and memory-mapped file pages.

Large user buffers improve performance because they reduce the number of `read()`/`write()` system calls.

Writing 1 byte at a time causes huge syscall overhead, while writing in filesystem-block-sized chunks, such as 4096 bytes, approaches optimal performance.

The kernel may also perform **read-ahead** for sequential file access, preloading future blocks into cache before the process requests them.

The C stdio library adds another buffering layer in user space for functions like `printf()`, `fprintf()`, `fgets()`, and `fputc()`.

For disk files, stdio is normally **fully buffered**, meaning data is accumulated in a user-space buffer before `write()` is invoked.

For terminals, stdio is usually **line buffered**, flushing output when a newline appears.

`stderr` is normally **unbuffered**, so error messages appear immediately.

`setvbuf()`, `setbuf()`, and `setbuffer()` allow programs to control stdio buffering behavior.

`fflush()` flushes stdio buffers into the kernel, but it does **not guarantee disk persistence**.

To force kernel buffers to disk, Linux provides `fsync()`, `fdatasync()`, and `sync()`.

`fsync(fd)` flushes both file data and all associated metadata, giving **file integrity completion**.

`fdatasync(fd)` flushes file data and only the metadata needed to retrieve that data, giving **data integrity completion**.

`fdatasync()` can be faster than `fsync()` because it may avoid unnecessary metadata writes such as timestamp updates.

`sync()` flushes dirty buffers system-wide, but it is coarse-grained and affects more than one file.

Opening a file with `O_SYNC` makes every `write()` synchronous, forcing data and metadata toward disk before returning.

`O_SYNC` provides stronger durability but can destroy throughput because each write may block on physical I/O.

For performance-sensitive durable writes, it is usually better to use **large buffers plus occasional `fsync()` or `fdatasync()`**, rather than `O_SYNC` on every write.

Linux background writeback threads periodically flush dirty page-cache pages to reduce crash-loss windows.

Kernel writeback behavior is tunable through `/proc/sys/vm`, such as dirty expiration settings.

`posix_fadvise()` lets applications tell the kernel expected access patterns, such as sequential, random, will-need, or don’t-need.

This does not change program semantics, but it can help the kernel optimize page-cache behavior and read-ahead.

For example, `POSIX_FADV_SEQUENTIAL` increases read-ahead, while `POSIX_FADV_RANDOM` disables it.

`POSIX_FADV_DONTNEED` tells the kernel cached pages are no longer useful and can be reclaimed.

Linux also supports **direct I/O** using `O_DIRECT`, which bypasses the page cache and transfers data directly between user buffers and disk/device.

Direct I/O is not automatically faster; it often loses kernel optimizations like caching, clustering, and read-ahead.

It is mainly useful for systems like databases that implement their own caching and I/O scheduling.

`O_DIRECT` requires strict alignment: buffer address, file offset, and I/O length must align to device block-size boundaries.

Violating direct-I/O alignment rules usually causes `EINVAL`.

Mixing stdio functions and raw system calls on the same file is dangerous unless buffering is carefully handled.

For example, `printf()` may keep data in a stdio buffer while `write()` immediately sends data to the kernel, causing output reordering.

`fileno()` converts a `FILE *` stream to a file descriptor, while `fdopen()` wraps a file descriptor in a stdio stream.

At kernel depth, the key idea is that Linux I/O performance and correctness depend on understanding **where data currently lives**: user buffer, stdio buffer, page cache, disk cache, or stable storage.

Chapter 14 - File Systems
This chapter explains Linux file systems from the kernel’s point of view: **devices, partitions, file-system layout, i-nodes, VFS, journaling, mounting, and filesystem statistics**.

A **device special file** is a filesystem object that represents a kernel device driver, usually under `/dev`.

Linux uses device drivers to hide hardware-specific details behind common operations like `open()`, `read()`, `write()`, `mmap()`, and `ioctl()`.

This is what enables UNIX’s “everything is a file” style of I/O abstraction.

Devices are mainly of two types: **character devices** and **block devices**.

Character devices, such as terminals and keyboards, handle data as byte streams.

Block devices, such as disks, handle data in fixed-size blocks and are the normal backing store for file systems.

Each device file has a **major number** and a **minor number** stored in its i-node.

The **major number** tells the kernel which driver should handle the device.

The **minor number** identifies a specific device instance handled by that driver.

The filename under `/dev` is not what binds the device to the driver; the major/minor numbers do.

Modern Linux uses **udev** and **sysfs** to dynamically create device nodes based on actual hardware detected by the kernel.

A disk is divided into one or more **partitions**, and each partition appears to the kernel as a separate block device.

A partition may contain a file system, raw database storage, or swap space.

Swap partitions are enabled using `swapon()` and disabled using `swapoff()`.

Linux exposes partition and swap information through `/proc/partitions` and `/proc/swaps`.

A **file system** is an organized structure for storing files and directories on a block device.

Linux supports many file systems, including ext2, ext3, ext4, XFS, Btrfs, Reiserfs, FAT, NTFS, ISO9660, NFS, SMB, and FUSE-based user-space filesystems.

The kernel’s supported filesystem types can be inspected through `/proc/filesystems`.

A filesystem is created using `mkfs`, which initializes metadata structures on a block device or partition.

In a traditional filesystem layout, the main regions are the **boot block**, **superblock**, **i-node table**, and **data blocks**.

The boot block may contain bootloader-related data and is not normally used by the filesystem itself.

The **superblock** stores global filesystem metadata such as block size, filesystem size, and i-node table size.

The **i-node table** contains one i-node per file or directory.

The **data blocks** store actual file contents and directory entries.

The filesystem allocation unit is the **logical block**, commonly 1024, 2048, or 4096 bytes.

In ext2, the disk is divided into **block groups**, each containing local metadata and data blocks.

This block-group design improves locality by trying to keep a file’s i-node and data blocks near each other.

An **i-node** is the kernel’s core metadata object for a file.

It stores file type, UID, GID, permissions, timestamps, link count, file size, allocated blocks, and pointers to file data blocks.

Importantly, filenames are not stored in the i-node; filenames live in directories and map names to i-node numbers.

Linux filesystems usually track access time, modification time, and status-change time, but traditionally not creation time.

In ext2, each i-node contains direct, indirect, double-indirect, and triple-indirect block pointers.

The first 12 pointers directly reference data blocks, making small-file access fast.

Indirect pointers allow large files without making every i-node huge.

This design keeps i-nodes fixed-size while supporting both small files efficiently and very large files flexibly.

Sparse files are possible because missing regions can be represented by null block pointers instead of real disk blocks full of zeros.

The **Virtual File System**, or VFS, is the kernel abstraction layer above concrete filesystems.

Applications call generic syscalls like `open()`, `read()`, `write()`, `stat()`, `mkdir()`, and `rename()`.

The VFS dispatches those operations to filesystem-specific implementations such as ext4, XFS, Btrfs, VFAT, or NFS.

This allows applications to use one API without knowing the internal layout of each filesystem.

If a filesystem does not support an operation, such as symlinks on VFAT, the VFS returns an appropriate error.

Traditional filesystems like ext2 require `fsck` after crashes because metadata updates may be left incomplete.

A **journaling filesystem** solves this by recording metadata updates in a journal before applying them to the main filesystem.

After a crash, the kernel can replay or discard incomplete journal transactions quickly.

This avoids scanning the entire filesystem and greatly improves availability after crashes.

Journaling usually protects metadata consistency; data journaling is optional and may reduce performance.

Linux journaling filesystems include ext3, ext4, XFS, JFS, Reiserfs, and Btrfs.

Linux presents all mounted filesystems under a single directory hierarchy rooted at `/`.

A filesystem is attached to this hierarchy at a **mount point** using `mount`.

The `mount()` system call requires privilege and takes a source device, target directory, filesystem type, mount flags, and filesystem-specific data.

Mount flags control behavior such as read-only mounts, no device access, no executable files, no setuid programs, no atime updates, synchronous updates, and bind mounts.

`/proc/mounts` reflects the kernel’s live mount table and is more reliable than `/etc/mtab`.

With mount namespaces, each process can have its own view of mounted filesystems.

`umount()` detaches a filesystem, but it fails with `EBUSY` if processes still hold files or directories inside it.

`umount2()` adds options such as lazy unmount with `MNT_DETACH`.

Linux supports advanced mount behavior such as mounting the same filesystem at multiple points.

It also supports stacking multiple mounts on the same mount point, where newer mounts hide older ones until removed.

A **bind mount** makes an existing file or directory visible at another location in the hierarchy.

Bind mounts are useful for chroot jails, containers, namespace construction, and controlled filesystem views.

Recursive bind mounts replicate submounts as well as the top-level directory.

`tmpfs` is a memory-backed virtual filesystem that stores data in RAM and swap.

It is fast because it avoids disk I/O, but its contents disappear after unmount or crash.

`tmpfs` is commonly used for temporary storage and also underlies mechanisms such as POSIX shared memory via `/dev/shm`.

`statvfs()` and `fstatvfs()` return filesystem-level information such as block size, free blocks, free i-nodes, mount flags, and maximum filename length.

At kernel depth, the key idea is that Linux file I/O is built as layered abstractions: **device drivers expose block devices, filesystems organize blocks into i-nodes and directories, VFS unifies filesystem operations, and mounts compose many filesystems into one process-visible namespace**.

Chapter 15 - File Attributes

This chapter focuses on **file metadata**, mostly stored in the file’s **i-node**, and explains how Linux retrieves and modifies attributes such as type, ownership, permissions, timestamps, and special i-node flags.

The main metadata retrieval APIs are `stat()`, `lstat()`, and `fstat()`.

`stat(pathname, ...)` returns metadata for a named file and follows symbolic links.

`lstat(pathname, ...)` returns metadata about the symbolic link itself rather than its target.

`fstat(fd, ...)` returns metadata for a file already opened through a file descriptor.

The `stat` structure is mostly a user-space view of the kernel’s i-node metadata.

Important fields include `st_dev`, `st_ino`, `st_mode`, `st_nlink`, `st_uid`, `st_gid`, `st_size`, `st_blocks`, and file timestamps.

The pair `(st_dev, st_ino)` uniquely identifies a file across mounted filesystems.

`st_dev` identifies the device/filesystem containing the i-node.

`st_ino` identifies the file’s i-node number inside that filesystem.

For device special files, `st_rdev` stores the major and minor device numbers of the represented device.

The `st_mode` field encodes both **file type** and **permission bits**.

File type is tested using macros like `S_ISREG()`, `S_ISDIR()`, `S_ISLNK()`, `S_ISCHR()`, `S_ISBLK()`, `S_ISFIFO()`, and `S_ISSOCK()`.

The file type tells the kernel whether the object is a regular file, directory, symlink, device, FIFO, or socket.

The lower permission bits of `st_mode` encode read, write, and execute permissions for owner, group, and others.

`st_nlink` stores the number of hard links pointing to the same i-node.

`st_size` gives logical file size in bytes, while `st_blocks` gives actual allocated disk blocks.

This distinction matters for sparse files, where `st_size` may be large but `st_blocks` may be small.

`st_blksize` is not the filesystem block size; it is the recommended I/O block size for efficient reads and writes.

Linux tracks three classic timestamps: `st_atime`, `st_mtime`, and `st_ctime`.

`st_atime` records last file access time.

`st_mtime` records last file content modification time.

`st_ctime` records last i-node/status change time, such as permission, owner, link count, or metadata changes.

`ctime` is not creation time; most traditional UNIX filesystems do not store creation time.

Modern Linux supports nanosecond timestamp resolution on filesystems such as ext4, XFS, JFS, and Btrfs.

File timestamps can be changed using `utime()`, `utimes()`, `utimensat()`, and `futimens()`.

`utime()` works with second precision, while `utimes()` supports microsecond precision.

`utimensat()` and `futimens()` support nanosecond precision and allow independent timestamp updates.

`UTIME_NOW` sets a timestamp to the current time.

`UTIME_OMIT` leaves a timestamp unchanged without needing a prior `stat()` call.

A successful timestamp update also changes `st_ctime`, because the file’s i-node metadata changed.

File ownership is represented by `st_uid` and `st_gid`.

New files usually inherit the creator’s effective UID as owner.

The group ID of a new file may come either from the process effective GID or from the parent directory’s GID.

On ext-family filesystems, this behavior depends on mount options such as `grpid`, `bsdgroups`, `nogrpid`, and the parent directory’s set-group-ID bit.

`chown()`, `lchown()`, and `fchown()` change file UID/GID ownership.

`chown()` follows symlinks, `lchown()` changes the link itself, and `fchown()` operates through a file descriptor.

Only privileged processes can arbitrarily change file ownership.

An unprivileged owner may change a file’s group only to a group of which the process is a member.

Changing file ownership clears set-user-ID and set-group-ID bits as a security protection.

This prevents users from creating privileged executables by changing ownership after enabling setuid/setgid bits.

File permissions divide access into three classes: owner, group, and other.

Each class has read, write, and execute bits.

For regular files, read means file contents may be read, write means contents may be modified, and execute means the file may be executed.

For directories, read means filenames may be listed.

Directory write permission allows creating, deleting, and renaming entries.

Directory execute permission means search permission: the process may traverse the directory and access named entries.

To access `/a/b/c`, the process needs execute permission on `/`, `/a`, and `/a/b`.

Deleting a file depends mainly on permissions of the containing directory, not the file itself.

The kernel checks permissions using effective IDs, or more precisely on Linux, filesystem UID/GID.

Permission checks happen during pathname-based operations such as `open()`.

After a file descriptor is obtained, later operations like `read()`, `write()`, `fstat()`, and `mmap()` do not repeat pathname permission checks.

The permission algorithm checks owner first, then group, then other.

This means owner permissions can be more restrictive than group permissions, and the owner will still be denied if owner bits deny access.

Privileged processes bypass most permission checks, but Linux still requires at least one execute bit to be set before a privileged process may execute a regular file.

`access()` checks permissions using real UID/GID rather than effective UID/GID.

`access()` is dangerous in privileged programs because it creates a time-of-check/time-of-use race.

A safer pattern is to attempt the real operation directly and inspect failure through `errno`.

The special permission bits are set-user-ID, set-group-ID, and sticky bit.

Set-user-ID makes an executable run with the file owner’s effective UID.

Set-group-ID makes an executable run with the file group’s effective GID.

On directories, set-group-ID causes newly created files/subdirectories to inherit the directory’s group.

The sticky bit on directories restricts deletion: users may delete only files they own or files in directories they own.

This is why `/tmp` is world-writable but protected by the sticky bit.

`umask()` controls which permission bits are removed from newly created files and directories.

The final permissions are essentially: `requested_mode & ~umask`.

A common umask is `022`, which removes write permission for group and others.

`chmod()` and `fchmod()` change file permissions using pathname or file descriptor.

Symbolic-link permissions are normally ignored; `chmod()` changes the target file, not the symlink itself.

Linux also supports nonstandard **i-node flags**, sometimes called ext2 extended file attributes.

These flags are viewed and changed using `lsattr` and `chattr`, or programmatically using `ioctl()` with `FS_IOC_GETFLAGS` and `FS_IOC_SETFLAGS`.

Important i-node flags include append-only, immutable, no-atime, synchronous updates, data journaling, no-dump, and directory-sync.

`FS_APPEND_FL` allows writes only in append mode and is useful for log-like files.

`FS_IMMUTABLE_FL` prevents even metadata-changing operations such as unlink, rename, chmod, chown, truncate, and write.

`FS_NOATIME_FL` avoids updating access time and improves read-heavy workload performance.

`FS_SYNC_FL` makes file updates synchronous, similar to always opening with `O_SYNC`.

`FS_DIRSYNC_FL` makes directory updates synchronous and is useful when directory-entry durability matters.

Many i-node flags are inherited by new files created inside flagged directories.

At kernel depth, this chapter shows that a Linux file is not just bytes: it is an i-node-backed object with identity, ownership, permissions, timestamps, link count, allocation state, and kernel-enforced behavioral flags.

Chapter 16 - Extendend Attributes

Extended attributes, or **EAs**, let Linux attach arbitrary **name-value metadata** directly to a file’s i-node.

They were added in Linux 2.6 and are used internally for features such as **ACLs**, **file capabilities**, and security labels.

Unlike normal file contents, EAs describe the file rather than being part of the file data stream.

An EA name has the form:namespace.name

The namespace separates attributes into different kernel/security domains.

Linux supports four main EA namespaces: **user**, **trusted**, **system**, and **security**.

`user.*` attributes are meant for normal applications and can be accessed according to ordinary file permissions.

Reading a `user.*` EA requires read permission on the file, while modifying it requires write permission.

`trusted.*` attributes are similar to user attributes but require `CAP_SYS_ADMIN`.

`system.*` attributes are mainly used by the kernel, especially for storing access control lists.

`security.*` attributes store security-module metadata, such as SELinux labels and file capabilities.

EAs require filesystem support; Linux filesystems such as ext2, ext3, ext4, XFS, JFS, Reiserfs, and Btrfs support them.

On some filesystems, user EAs require mounting with the `user_xattr` option.

From the shell, EAs can be manipulated using `setfattr` and `getfattr`.

Programmatically, Linux provides `setxattr()`, `getxattr()`, `listxattr()`, and `removexattr()`.

Each has pathname, symlink-aware, and file-descriptor variants, such as `setxattr()`, `lsetxattr()`, and `fsetxattr()`.

`setxattr()` can either create or replace attributes, with `XATTR_CREATE` and `XATTR_REPLACE` controlling exact behavior.

`getxattr()` retrieves an attribute value, while `listxattr()` returns all EA names as null-terminated strings.

A key systems detail is that EA operations are still mediated by **VFS, filesystem support, namespace rules, file permissions, and Linux capabilities**, making EAs a flexible but kernel-controlled metadata extension to the traditional i-node model.

Chapter 17 - Access Control Lists

ACLs extend the traditional UNIX permission model by allowing permissions to be assigned to **specific users and groups**, not just owner/group/other.

Traditional permissions are compact but coarse: each file has only owner, group, and other permission classes.

Linux ACLs provide finer-grained access control from kernel 2.6 onward.

ACL support is filesystem-dependent and may require mount options such as `acl` on ext2/ext3/ext4/Reiserfs.

Linux ACLs are based on the withdrawn POSIX.1e draft, so they are widely used but not fully standardized across UNIX systems.

An ACL is a list of **ACL entries**, and each entry defines permissions for a user, group, or special class.

Each ACL entry has three parts: **tag type**, optional **tag qualifier**, and **permission set**.

The permission set contains the familiar read, write, and execute bits.

`ACL_USER_OBJ` represents permissions for the file owner and corresponds to traditional owner permissions.

`ACL_GROUP_OBJ` represents permissions for the file’s owning group.

`ACL_OTHER` represents permissions for users who match no other ACL entry.

These three entries form a **minimal ACL**, equivalent to ordinary UNIX permissions.

`ACL_USER` entries grant permissions to specific named users.

`ACL_GROUP` entries grant permissions to specific named groups.

An ACL containing `ACL_USER` or `ACL_GROUP` entries is called an **extended ACL**.

Extended ACLs are stored on Linux as extended attributes, specifically `system.posix_acl_access`.

The `ACL_MASK` entry is required when named user or named group entries exist.

The mask limits the effective permissions of `ACL_USER`, `ACL_GROUP_OBJ`, and `ACL_GROUP` entries.

Conceptually, `ACL_MASK` is the maximum permission ceiling for the ACL “group class.”

This mask exists so ACL-aware and ACL-unaware programs can coexist safely.

When `chmod()` changes traditional group permissions on a file with an ACL, it actually changes the `ACL_MASK`, not necessarily `ACL_GROUP_OBJ`.

Likewise, `stat()` reports the ACL mask in the group permission bits when an ACL mask exists.

This is why `ls -l` may show group permissions that reflect the mask rather than the raw owning-group ACL entry.

ACL permission checking starts with privileged-process checks, similar to traditional permissions.

If the process owns the file, the kernel uses `ACL_USER_OBJ`.

If the process matches a named `ACL_USER` entry, that entry is used after applying `ACL_MASK`.

If the process matches the file group or a named group entry, the kernel checks matching group-class entries and applies the mask.

If no user or group entry matches, the kernel falls back to `ACL_OTHER`.

For group matching, having multiple matching groups does not simply union all permissions arbitrarily; the algorithm checks whether a matching group entry grants the requested access.

From the shell, `getfacl` displays ACLs and `setfacl` modifies them.

`setfacl -m` adds or changes ACL entries.

`setfacl -x` removes selected ACL entries.

`setfacl -b` removes extended ACL entries and leaves only a minimal ACL.

Files with extended ACLs are shown by `ls -l` with a trailing `+` after the permission string.

Directories can also have **default ACLs**, stored as `system.posix_acl_default`.

A default ACL does not control access to the directory itself.

Instead, it determines the initial ACL inherited by new files and subdirectories created inside that directory.

New subdirectories inherit the parent directory’s default ACL as their own default ACL.

New files inherit the parent default ACL as their access ACL, with relevant entries masked by the `mode` argument of `open()`, `mkdir()`, etc.

When a directory has a default ACL, the process `umask` does not determine the new file’s access ACL in the normal way.

Internally, Linux implements ACL operations using the ACL library API layered over extended attributes.

Programs use `<sys/acl.h>` and link with `-lacl`.

`acl_get_file()` retrieves an access or default ACL into memory.

`acl_set_file()` writes an in-memory ACL back to the file.

`acl_get_entry()` iterates over ACL entries.

`acl_get_tag_type()`, `acl_get_qualifier()`, and `acl_get_permset()` inspect each entry.

`acl_calc_mask()` computes the mask as the union of group-class permissions.

`acl_valid()` verifies that an ACL has required entries and no duplicate named users or groups.

At kernel depth, ACLs are best understood as a **VFS-enforced, extended-attribute-backed expansion of the i-node permission model**, preserving UNIX compatibility while enabling per-user and per-group authorization.

Chapter 18 - Directories and Links

Directories are special files whose i-node type is **directory**, but whose data is organized as a table mapping **filename → i-node number**.

A file’s i-node does **not** store its filename; names live inside directories.

This separation allows multiple directory entries to point to the same i-node, creating **hard links**.

A hard link is simply another filename for the same underlying file object.

All hard links to a file are equal; there is no “original” name at the kernel level.

The i-node’s link count records how many directory entries point to it.

`ln old new` creates a hard link, increasing the i-node link count.

`rm name` or `unlink()` removes one directory entry and decrements the link count.

The file’s i-node and data blocks are freed only when the link count reaches zero **and** no process still has the file open.

This means an unlinked file can continue consuming disk space until the last file descriptor is closed.

This behavior is useful for temporary files: create, open, unlink immediately, and let the kernel delete it automatically on close/exit.

Hard links cannot normally cross filesystem boundaries because i-node numbers are meaningful only within one filesystem.

Hard links to directories are disallowed to prevent cycles in the directory tree.

A symbolic link is a separate file whose contents are a pathname.

Unlike hard links, symlinks point to a **name**, not directly to an i-node.

Therefore, symbolic links can cross filesystems and can point to directories.

If the target pathname disappears, the symlink remains but becomes a **dangling link**.

Symbolic links may be absolute or relative; relative symlinks are interpreted relative to the directory containing the link.

Many syscalls follow symlinks by default and operate on the target file.

Some calls, such as `lstat()`, `readlink()`, `lchown()`, and `unlink()`, operate on the symlink itself.

In pathname resolution, symlinks in intermediate directory components are always followed.

Linux limits symlink dereferencing to prevent infinite loops and kernel stack exhaustion.

`link(oldpath, newpath)` creates a new hard link to an existing file.

`unlink(pathname)` removes a filename; if it names a symlink, the symlink itself is removed, not its target.

`rename(oldpath, newpath)` changes a directory entry name or moves it within the same filesystem.

`rename()` does not move file data; it only updates directory mappings.

If `newpath` already exists, `rename()` atomically replaces it.

Open file descriptors are unaffected by `rename()` because descriptors refer to open file descriptions, not filenames.

`rename()` cannot move a file across filesystems; that fails with `EXDEV`.

Cross-filesystem moves require copy-then-unlink, as done by `mv`.

`symlink(target, linkpath)` creates a symbolic link whose contents are `target`.

The symlink target does not need to exist when the symlink is created.

`readlink()` reads the raw contents of a symlink but does not append a null terminator.

Therefore, callers must manage buffer size and manually terminate strings if needed.

`mkdir(pathname, mode)` creates a new directory.

A new directory initially contains `.` pointing to itself and `..` pointing to its parent.

`mkdir()` creates only the final pathname component; parent directories must already exist unless higher-level tools implement recursive creation.

Directory permissions are affected by the process `umask`.

If the parent directory has set-group-ID set, new subdirectories inherit that bit and usually the parent’s group behavior.

`rmdir(pathname)` removes an empty directory.

`rmdir()` fails if the directory is not empty.

`remove(pathname)` is a C library wrapper that calls `unlink()` for files and `rmdir()` for directories.

Directories cannot be read portably using `read()`.

Programs scan directories using `opendir()`, `readdir()`, and `closedir()`.

`opendir()` returns a directory stream handle of type `DIR *`.

`readdir()` returns one directory entry at a time, usually containing at least `d_name` and `d_ino`.

The order returned by `readdir()` is filesystem order, not sorted order.

Programs should explicitly skip `.` and `..` when recursively scanning directories.

If a directory changes during scanning, POSIX does not guarantee whether newly added or removed entries will be observed.

`rewinddir()` resets a directory stream to the beginning.

`dirfd()` returns the file descriptor underlying a directory stream.

`opendir()` sets close-on-exec on the directory file descriptor to avoid leaking it across `exec()`.

`nftw()` walks an entire directory tree and calls a user-supplied function for every visited file.

By default, `nftw()` performs preorder traversal: directory first, then contents.

`FTW_DEPTH` changes traversal to postorder, processing children before their parent directory.

`FTW_PHYS` prevents following symbolic links.

`FTW_MOUNT` prevents crossing into another mounted filesystem.

Each process has a **current working directory**, used to resolve relative pathnames.

A child process inherits its parent’s current working directory.

`getcwd()` returns the absolute pathname of the current working directory.

`chdir()` changes the current working directory using a pathname.

`fchdir()` changes the current working directory using an open directory file descriptor.

`fchdir()` is useful because it avoids depending on a pathname that may change.

Linux provides `*at()` syscalls such as `openat()`, `fstatat()`, `unlinkat()`, and `renameat()`.

These interpret relative pathnames relative to a supplied directory file descriptor.

The `*at()` APIs reduce race conditions in pathname-based filesystem operations.

They are also useful for multithreaded programs that need per-thread logical working directories.

Each process also has a **root directory**, used to resolve absolute pathnames.

`chroot()` changes a process’s root directory, creating a restricted filesystem view.

A chroot jail is not a complete security container by itself.

After `chroot()`, programs should also `chdir("/")`, otherwise relative paths may still escape if the old working directory was outside.

Open directory file descriptors outside the jail can also allow escape through `fchdir()`.

`realpath()` resolves symlinks, `.` and `..`, producing a canonical absolute pathname.

`dirname()` and `basename()` split a pathname into directory and final filename components.

These functions may modify their input strings, so callers should pass copies if the original path must be preserved.

At kernel depth, this chapter shows that Linux pathname handling is fundamentally about **directory-entry-to-i-node mappings, link counts, symlink resolution, process-relative namespace state, and race-safe directory-relative operations**.

Chapter 19 - Monitoring File Events

`inotify` is a Linux-specific kernel mechanism, introduced in **Linux 2.6.13**, for monitoring file and directory events.

It allows applications to learn when files are opened, modified, deleted, renamed, or when metadata changes.

Typical users include file managers, backup tools, editors, daemons, and configuration reload systems.

The basic object is an **inotify instance**, created using `inotify_init()`.

`inotify_init()` returns a file descriptor representing a kernel-maintained event queue.

Applications then add watches using `inotify_add_watch()`.

Each watch contains a pathname and an event mask describing what should be monitored.

The kernel returns a **watch descriptor**, which identifies that watched object within the inotify instance.

A process can monitor many files or directories through a single inotify file descriptor.

Events are obtained by calling `read()` on the inotify file descriptor.

Each `read()` returns one or more `struct inotify_event` records.

The event structure contains the watch descriptor, event mask, rename cookie, optional filename, and name length.

When watching a directory, events are reported both for the directory itself and for entries inside it.

For events inside a watched directory, the `name` field identifies the affected child file or subdirectory.

For events on the watched object itself, the `name` field is unused.

Important event masks include `IN_ACCESS`, `IN_OPEN`, `IN_MODIFY`, `IN_ATTRIB`, and close events.

`IN_ATTRIB` covers metadata changes such as permissions, ownership, link count, extended attributes, UID, or GID.

`IN_CREATE` and `IN_DELETE` report creation and deletion of entries inside a watched directory.

`IN_DELETE_SELF` reports that the watched file or directory itself was deleted.

`IN_MOVE_SELF` reports that the watched object itself was renamed or moved.

`IN_MOVED_FROM` and `IN_MOVED_TO` report movement out of and into watched directories.

Rename-related move events are linked using the `cookie` field.

This cookie allows applications to associate the old name and new name of a renamed file.

If the affected object is a directory, the returned event mask includes `IN_ISDIR`.

`IN_IGNORED` tells the application that a watch was removed.

A watch can be removed explicitly by `inotify_rm_watch()` or implicitly by the kernel.

Implicit removal happens when the watched object is deleted or its filesystem is unmounted.

`IN_UNMOUNT` reports that the filesystem containing the watched object was unmounted.

`IN_Q_OVERFLOW` reports that the kernel event queue overflowed and events were lost.

`inotify` monitoring is **not recursive**.

To monitor an entire directory tree, applications must add a watch for every directory in the tree.

If a new subdirectory is created, the application must add a new watch for it manually.

`inotify_add_watch()` normally replaces the old mask if the pathname is already watched.

With `IN_MASK_ADD`, the new mask is ORed into the existing watch mask.

`IN_ONESHOT` makes a watch automatically disappear after a single event.

`IN_ONLYDIR` ensures that the watched pathname is a directory, avoiding race conditions.

`IN_DONT_FOLLOW` prevents symlink dereferencing, allowing the symlink itself to be watched.

By default, reads from an inotify file descriptor block until events are available.

With `IN_NONBLOCK` or `O_NONBLOCK`, reads fail with `EAGAIN` if no events are queued.

`inotify_init1()` adds useful creation flags such as `IN_CLOEXEC` and `IN_NONBLOCK`.

`IN_CLOEXEC` prevents leaking the inotify fd across `exec()`.

The inotify fd can be integrated with `select()`, `poll()`, `epoll`, and signal-driven I/O.

This makes it suitable for event-loop based programs.

The kernel stores inotify events in an ordered queue.

For rename, `IN_MOVED_FROM` is guaranteed to appear before the matching `IN_MOVED_TO`.

However, identical queued events may be coalesced by the kernel.

Therefore, inotify is reliable for detecting that an event occurred, but not for counting exactly how many identical events occurred.

Applications must maintain their own mapping from watch descriptors to pathnames.

Kernel memory use is controlled by `/proc/sys/fs/inotify`.

`max_queued_events` limits queued events per inotify instance.

`max_user_instances` limits inotify instances per real user ID.

`max_user_watches` limits watch items per real user ID.

If queue overflow occurs, applications may need to rescan watched directories to rebuild correct state.

`inotify` replaced the older `dnotify` mechanism.

`dnotify` used signals, monitored only directories, consumed directory file descriptors, and reported less precise information.

`inotify` is superior because it uses file-descriptor event queues, supports individual files and directories, integrates with polling APIs, and provides detailed event metadata.

At kernel depth, `inotify` is a **VFS-level notification system** where path-based watches feed ordered event records into a per-instance kernel queue, allowing user space to react to filesystem state changes without polling.

Chapter 20 - Signals

Signals are Linux/UNIX **software interrupt notifications** delivered to a process when some asynchronous event occurs.

They interrupt normal control flow, so the process may suddenly stop executing ordinary code and run signal-related logic.

Signals can be generated by the **kernel**, by another process, or by the process itself.

Kernel-generated signals commonly arise from hardware exceptions, terminal input, timers, child termination, I/O readiness, or resource-limit violations.

Examples include `SIGSEGV` for invalid memory access, `SIGFPE` for arithmetic errors, and `SIGCHLD` when a child process terminates or stops.

Terminal-generated signals include `SIGINT` from Control-C and `SIGTSTP` from Control-Z.

Each signal is represented by a small integer, but portable programs use symbolic names such as `SIGINT`, `SIGTERM`, and `SIGKILL`.

Linux standard signals occupy signal numbers 1–31, while realtime signals are a separate later mechanism.

A signal is first **generated**, then becomes **pending**, and is later **delivered** to the process.

If the signal is not blocked, delivery normally occurs when the process next runs or immediately if it is already executing.

Each process has a **signal mask**, which is the set of signals currently blocked from delivery.

If a blocked signal is generated, it remains pending until the process unblocks it.

Signals are therefore not just events; they interact with scheduler timing, process state, and kernel-maintained pending masks.

On delivery, the default action depends on the signal.

Some signals are ignored by default, such as `SIGCHLD` in many cases.

Some terminate the process, such as `SIGTERM`.

Some terminate and produce a core dump, such as `SIGSEGV`, `SIGABRT`, `SIGQUIT`, and `SIGILL`.

Some stop the process, such as `SIGSTOP`, `SIGTSTP`, `SIGTTIN`, and `SIGTTOU`.

`SIGCONT` resumes a stopped process.

A process can change a signal’s **disposition**, meaning how it reacts when that signal is delivered.

The disposition can be default action, ignore, or execute a signal handler.

A signal handler is a programmer-defined function invoked asynchronously by the kernel when the signal is delivered.

After the handler returns, the process usually resumes at the instruction where it was interrupted.

`SIGKILL` and `SIGSTOP` are special: they cannot be caught, ignored, or blocked.

This gives the kernel and administrators unconditional kill/stop mechanisms.

`SIGTERM` is the polite termination signal and should normally be tried before `SIGKILL`.

Well-designed programs catch `SIGTERM` to clean up temporary files, release resources, and exit gracefully.

`SIGKILL` bypasses cleanup and should be a last resort.

`signal()` is the older API for changing signal disposition.

Although simpler, `signal()` has historical portability problems and should not be preferred in serious portable programs.

`sigaction()` is the modern and reliable API for installing signal handlers.

`sigaction()` can retrieve existing dispositions, set handler flags, and define extra signals to block while a handler runs.

The `sa_mask` field specifies signals automatically blocked during handler execution.

By default, the signal currently being handled is also blocked during its own handler, preventing recursive handler reentry.

`SA_NODEFER` disables this automatic self-blocking.

`SA_RESETHAND` resets the signal disposition to default after the handler is entered.

`SA_RESTART` asks the kernel/libc to restart certain interrupted system calls automatically.

`SA_SIGINFO` enables richer signal handlers with extra information about the signal source.

Signals may be sent with `kill(pid, sig)`.

Despite the name, `kill()` can send any signal, not just terminating ones.

If `pid > 0`, the signal is sent to that specific process.

If `pid == 0`, it is sent to all processes in the caller’s process group.

If `pid < -1`, it is sent to the process group whose ID is `abs(pid)`.

If `pid == -1`, it broadcasts to every process the caller is permitted to signal, except special exclusions such as `init`.

Signal-sending permission is controlled by credentials and capabilities.

A privileged process with `CAP_KILL` may signal arbitrary processes.

An unprivileged process can signal another process only if its real/effective user IDs match permitted IDs of the target.

`SIGCONT` has special relaxed rules within the same session to support shell job control.

Passing signal number `0` to `kill()` sends no signal but performs permission and existence checks.

This is often used to test whether a PID exists, though PID reuse means it cannot prove the original program is still alive.

`raise(sig)` sends a signal to the calling process, or specifically to the calling thread in threaded programs.

`killpg()` sends a signal to every process in a process group.

Signal sets are represented by `sigset_t`.

Portable code must initialize them using `sigemptyset()` or `sigfillset()`.

Signals can be added or removed with `sigaddset()` and `sigdelset()`.

Membership can be tested with `sigismember()`.

`sigprocmask()` changes or retrieves the process signal mask.

`SIG_BLOCK` adds signals to the blocked set.

`SIG_UNBLOCK` removes signals from the blocked set.

`SIG_SETMASK` replaces the current mask entirely.

Attempts to block `SIGKILL` or `SIGSTOP` are silently ignored.

`sigpending()` returns the set of currently pending blocked signals.

A crucial kernel detail is that **standard signals are not queued**.

If the same standard signal is generated multiple times while blocked, the pending set records only one instance.

When later unblocked, that signal is delivered once, not once per generation.

Even unblocked standard signals may be coalesced in practice if they arrive faster than the receiver is scheduled.

Realtime signals differ because they are queued, but standard signals behave like bits in a mask.

Changing the disposition of a pending signal affects how it will be handled when unblocked.

If a pending signal’s disposition is changed to ignore, the pending signal may effectively disappear.

`pause()` suspends the process until a signal is delivered and a handler runs.

After the handler returns, `pause()` fails with `EINTR`.

At kernel depth, signals are a per-process/per-thread asynchronous control mechanism built from **signal generation, pending masks, blocking masks, dispositions, handlers, and scheduler-time delivery semantics**.
