
Chapter 2 - Fundamental Concepts

The kernel is the core part of an operating system that manages computer resources like CPU, memory, and devices, making it easier and more flexible for programs to run.

The Linux kernel manages core system resources by scheduling processes, handling memory, providing filesystems, creating/terminating processes, and controlling access to hardware devices .Typically resides at /boot/vmlinuz .

The Linux kernel handles networking and system calls while enforcing protection through user/kernel modes and managing hardware conflicts transparently for processes and users.

A process sees only an isolated, simplified view of the system, while the kernel centrally controls CPU scheduling, memory, files, devices, process creation, and interprocess communication.

On UNIX/Linux, the shell is a user-space command interpreter, with variants like sh, csh, ksh, and bash, used both interactively and for writing shell scripts.

Linux identifies users by unique usernames/UIDs, organizes them into groups for access control, and gives the root superuser UID 0 with special administrative privileges.

Linux organizes all files under a single root directory /, where directories are special files containing links to files or other directories, forming one unified hierarchy.

Linux supports symbolic links as special files pointing to other pathnames, while filenames and pathnames follow rules that help uniquely locate files within the single directory hierarchy.

Linux pathnames can be absolute or relative, each process has a current working directory, and file access is controlled using ownership plus read/write/execute permissions for user, group, and others.

UNIX/Linux uses universal byte-stream I/O through common system calls like open(), read(), write(), and close(), with file descriptors representing open files and stdio providing a higher-level C library interface.

A program becomes a process when executed, and the kernel manages its resources, memory segments, IDs, and lifecycle, including creation of child processes using fork().

After fork(), the child process is a copy of the parent, can replace itself with a new program using exec(), has unique PID/PPID values, and terminates with a status that the parent can inspect.

A process’s permissions depend on its user/group IDs, privileged processes gain special powers through root or capabilities, and init is the first system process that creates and monitors other processes.

Daemon processes run long-term in the background without a terminal, while each process has environment variables for inherited configuration and resource limits to control its system usage.

Resource limits control how much system resource a process can use, while mmap() creates memory mappings for files or anonymous memory that can support efficient I/O, sharing, and process memory management.

Static libraries copy needed code into each executable, while shared libraries are loaded at runtime so multiple programs can use one common copy, saving disk space, memory, and update effort.

Linux provides many IPC mechanisms—such as signals, pipes, sockets, locks, message queues, semaphores, and shared memory—while signals act like software interrupts that notify processes about events or exceptional conditions.

Signals can be ignored, handled, blocked, or cause termination/suspension, while threads allow multiple execution paths inside one process sharing memory but using separate stacks for concurrent work.

Shell job control groups processes into jobs within sessions, manages foreground/background execution through controlling terminals, and pseudoterminals provide virtual terminal interfaces for terminal-oriented programs.

Linux measures real time from the UNIX Epoch, tracks each process’s user and system CPU time, and commonly uses client-server architecture where clients request services from servers.

Client-server systems let clients request shared services from servers, while realtime systems focus on guaranteeing responses within strict deadlines.

The Linux /proc file system is a virtual, Linux-specific interface that exposes kernel and process information as readable/writable files and directories.
Chapter 3 - System Programming Concepts

System calls are controlled entry points into the kernel, while library functions like glibc often wrap or abstract them; correct system programming requires checking every return value and handling errors explicitly.

A syscall wrapper loads arguments and syscall number into registers, traps into kernel mode, and the kernel dispatches via sys_call_table after validating the call and user-space arguments.

After the syscall routine completes, the kernel restores state and returns to user mode; libc converts negative kernel error returns into errno + -1, making syscalls costlier than normal C calls due to mode switch, validation, dispatch, and copy overhead.

strace observes the user/kernel boundary by tracing syscalls, arguments, return values, and errno, making it useful for debugging ABI-level behavior.

glibc version/location can be found by executing libc.so.6 directly or using ldd on a dynamically linked binary to locate the active GNU C library.

glibc version can be checked at compile time via __GLIBC__ macros or runtime via gnu_get_libc_version(), and every syscall/library call must have its return status checked because failures are common and usually signaled by -1 with details in errno.

On failure, syscalls set errno but never reliably clear it, so first check the return value, then inspect errno; special calls that can return -1 successfully require clearing errno before invocation.`strerror()` and `perror()` convert `errno` into locale-sensitive messages, but library functions differ: some use `-1`+`errno`, some use other failure values like `NULL`+`errno`, and some don’t use `errno` at all.

Error routines like errMsg(), errExit(), err_exit(), and errExitEN() standardize syscall/library error reporting using errno or explicit error numbers.

Syscall numbers are architecture-specific ABI identifiers; user code normally calls names, while libc maps them to numbers.

A key kernel-level distinction is exit() vs _exit(): _exit() avoids flushing stdio buffers and running parent-installed exit handlers, important after fork().

The ename table maps architecture-specific errno numbers to symbolic names like EAGAIN/EWOULDBLOCK, helping connect runtime errors to man-page semantics.

Numeric CLI arguments are parsed via strtol()-based getInt()/getLong(), giving validation, base control, and range checks unlike unsafe atoi().

Feature test macros decide which standards-specific definitions headers expose, so portable programs must define macros like _POSIX_C_SOURCE or _XOPEN_SOURCE before including headers.

SUSv3 defines portable system data types like pid_t, uid_t, off_t, size_t, so programs should avoid raw int/long because sizes and representations vary across UNIX versions and architectures.
When printing these types, cast to a stable large type such as long and use %ld; for off_t, use long long + %lld when needed.
Standard structures should be initialized by named field assignment, not positional initializers, because field order and extra implementation fields are not portable.
Binary-writing standard structs directly is nonportable; write individual fields in a defined order instead.
For portability, guard optional macros with #ifdef, include headers carefully across implementations, and often include <sys/types.h> early even if Linux doesn’t require it.

Chapter 4 - File I/O Model

File I/O in Linux is built around file descriptors, per-process small integers that refer to open files.
The universal I/O model uses open(), read(), write(), and close() for regular files, pipes, sockets, terminals, devices, and more.

Standard descriptors 0/1/2 are inherited from the shell as stdin/stdout/stderr, and redirection works by modifying these descriptors before exec().
Core file I/O uses open() to get an fd, read()/write() for byte transfer, and close() to release the descriptor and kernel resources.

copy.c demonstrates universal I/O: open() input/output files, loop with read() into a buffer and write() to output, then close() both descriptors.
It checks all syscall results, handles partial/failed writes, detects EOF when read() returns 0, and creates/truncates output using O_CREAT | O_WRONLY | O_TRUNC.

open() opens an existing file or creates a new one, returning a file descriptor or -1 with errno; symbolic links are normally dereferenced.
flags selects exactly one access mode like O_RDONLY, O_WRONLY, or O_RDWR; O_RDWR is not O_RDONLY | O_WRONLY, and mode is needed only with O_CREAT to set new-file permissions.

open() access mode must be exactly one of O_RDONLY, O_WRONLY, or O_RDWR, with creation permissions further filtered by umask/default ACLs.
On success, open() returns the lowest-numbered unused file descriptor, which enables controlled redirection by closing a target fd before opening another file.`open()` always allocates the lowest unused fd, so closing `STDIN_FILENO` before `open()` forces the new file onto fd `0`.
`flags` combines one access mode with options like `O_CREAT`, `O_TRUNC`, `O_APPEND`, `O_EXCL`, `O_NONBLOCK`, `O_CLOEXEC`, etc., controlling creation, truncation, append atomicity, blocking, symlink behavior, and exec inheritance.`open()` flags fall into three groups: access mode flags, creation flags, and open-file status flags; only status flags can later be retrieved/changed with `fcntl()`.
`O_CLOEXEC` atomically sets close-on-exec at `open()` time, preventing fd-leak races in multithreaded `fork()` + `exec()` programs.`O_CREAT` creates the file if missing and **requires `mode`**, while `O_EXCL | O_CREAT` makes creation atomic and fails with `EEXIST` if the file already exists.`O_DIRECT` bypasses page cache, `O_DIRECTORY` requires a directory, `O_DSYNC` makes data writes synchronous, `O_LARGEFILE` enables large files on 32-bit systems, and `O_NOATIME` avoids access-time updates with permission restrictions.

O_NOFOLLOW prevents symlink dereference and fails with ELOOP, useful for privileged programs avoiding symlink attacks.
O_NOCTTY prevents terminal takeover, O_NONBLOCK enables nonblocking I/O, O_SYNC makes writes synchronous, and O_TRUNC destroys existing regular-file data by truncating to zero length

open() can fail for many kernel-enforced reasons: permissions/path traversal (EACCES, ENOENT), descriptor limits (EMFILE, ENFILE), read-only filesystems (EROFS), directories opened for writing (EISDIR), or executing binaries opened for modification (ETXTBSY).
creat(path, mode) is the old UNIX shortcut for creating/opening a file for writing, equivalent to open(path, O_WRONLY | O_CREAT | O_TRUNC, mode)

read(fd, buffer, count) reads up to count bytes into a caller-allocated buffer, returning bytes read, 0 on EOF, or -1 on error.
Reads may be short, and read() does not null-terminate data, so treating the buffer as a C string without adding '\0' is unsafe.

write(fd, buffer, count) writes caller-provided bytes and may write fewer than requested; success only means data reached the kernel buffer, not necessarily disk.
close(fd) releases the file descriptor for reuse, though all open descriptors are automatically closed when the process exits.

close() should be checked because it frees a consumable fd resource and may report delayed filesystem errors, such as NFS commit failure.
Each open file has a kernel-maintained file offset, advanced by read()/write(), and lseek() explicitly repositions it for nonsequential I/O.

lseek() changes only the kernel-maintained file offset using SEEK_SET, SEEK_CUR, or SEEK_END; it returns the new offset and performs no physical I/O.
Offsets may move beyond EOF, enabling later writes to create sparse-file holes.

lseek() is valid only for seekable objects like regular/device files; pipes, FIFOs, sockets, and terminals fail with ESPIPE.
Seeking past EOF then writing creates a file hole: reads return zero bytes logically, but disk blocks are not allocated until data is actually written.

seek_io.c demonstrates how read(), write(), and lseek() operate on the same kernel-maintained file offset.
File holes still count in the file’s logical size, but they don’t consume disk blocks until written; the filesystem treats unwritten hole bytes as zero-filled on reads .

ioctl() is the escape hatch outside universal I/O, used for device/file-specific control operations that don’t fit open/read/write/close.
It takes an fd, a device-specific request, and optional typed argument pointer; most ioctl() operations are nonportable and request-dependent.

Chapter introduces deeper file I/O by focusing on atomicity of syscalls, the fcntl() interface for managing file descriptor flags, and kernel data structures behind open files.
It also covers advanced I/O: descriptor duplication, positional I/O (offset-independent), vectored I/O, nonblocking I/O, large file support, and safe temporary file creation.

Atomicity ensures each system call executes as an indivisible kernel operation, preventing interleaving by other processes/threads.
Using O_CREAT | O_EXCL makes file creation atomic, eliminating race conditions that occur with separate “check-then-create” sequences.

A race condition arises when separate open() calls (check‑then‑create) are interleaved by the scheduler, causing multiple processes to falsely believe they created the file.
O_CREAT | O_EXCL fixes this by making existence check + creation atomic, eliminating the scheduling window where races occur.

Atomicity prevents race conditions: separate lseek() + write() can interleave across processes, causing overwrites when appending.
Using O_APPEND makes the kernel perform seek-to-end + write atomically, ensuring correct concurrent appends even under scheduling interleavings.

fcntl() is a multipurpose syscall used to control file descriptors, with behavior determined by the cmd argument.
With F_GETFL and F_SETFL, it retrieves/modifies open-file status flags (like O_APPEND, O_NONBLOCK), while access modes require masking with O_ACCMODE since they aren’t simple bit flags.

fcntl() is useful when modifying flags on descriptors not opened by the program (e.g., inherited fds, pipe(), socket()), by reading with F_GETFL, updating bits, then writing back with F_SETFL.
A file descriptor refers to an entry in the per-process fd table, which points to a shared open file description (holding file offset and status flags), separating descriptor metadata from underlying file state.

Kernel model splits file state across three layers: per-process fd table → system-wide open file descriptions → inode table.
Multiple fds (even across processes) can point to the same open file description, sharing file offset and status flags, while the inode stores persistent metadata (type, permissions, size, timestamps).

When multiple file descriptors point to the same open file description, they share the file offset and status flags, so changes via one fd affect all others (even across processes).
Descriptor duplication (dup, dup2) ensures this shared state, which is why shell redirection like 2>&1 works correctly without offset inconsistencies.

dup() creates a new fd pointing to the same open file description, using the lowest unused fd, thus sharing file offset and status flags.
dup2(oldfd, newfd) atomically duplicates onto a specific fd (closing it first if needed), ensuring controlled redirection like dup2(1, 2) for stdout → stderr.

dup-family calls create new descriptors pointing to the same open file description (shared offset/status), with dup3/F_DUPFD_CLOEXEC allowing atomic setting of FD_CLOEXEC.
pread()/pwrite() perform I/O at a specified offset without modifying the shared file offset, enabling race-free concurrent access to different file regions.

pread()/pwrite() perform atomic offset-based I/O without modifying the shared file offset, avoiding race conditions in multithreaded or multi-process scenarios.
readv()/writev() implement scatter-gather I/O, transferring data across multiple buffers in a single syscall for efficiency and reduced syscall overhead.

readv() performs atomic scatter input, reading a contiguous byte stream into multiple buffers while preserving ordering and preventing interleaving with concurrent offset changes.
System limits like IOV_MAX bound the number of buffers, but glibc may internally fall back to a single read() if the vector is too large.

readv() reads into multiple buffers (iovec) in a single atomic operation, filling them sequentially while preserving contiguous file semantics.
The return value must be checked since fewer bytes than requested may be read, leaving some buffers partially or completely unfilled.

writev() performs atomic gather output, concatenating multiple buffers into a single contiguous write, preventing interleaving with other writers.
preadv()/pwritev() extend this by combining scatter-gather I/O with offset-based access, enabling concurrent, race-free multi-buffer I/O without changing the shared file offset.

O_NONBLOCK replaced older O_NDELAY to avoid ambiguity (e.g., 0 vs EOF), providing consistent nonblocking semantics with EAGAIN/EWOULDBLOCK.

The transitional LFS API (open64(), lseek64(), etc.) provides explicit 64-bit file offsets and sizes using types like off64_t, enabling access to files >2GB on 32-bit systems.
However, the preferred modern approach is defining _FILE_OFFSET_BITS=64, which transparently upgrades standard syscalls (open, lseek) to 64-bit without changing source code.

Defining _FILE_OFFSET_BITS=64 transparently upgrades standard APIs (open, lseek, off_t) to 64-bit, enabling large-file support without code changes.
The older transitional API (open64, lseek64, off64_t) explicitly uses 64-bit types/functions but is now largely obsolete in favor of macro-based portability.

For LFS, printing off_t requires casting to a wider type like long long with %lld, since it may exceed long size under 64-bit offsets.

mkstemp() securely creates a unique temporary file atomically from a template (XXXXXX) and returns an fd, avoiding race conditions in temp file creation.

mkstemp() securely creates a unique temp file using O_EXCL, returning an fd; typical usage unlinks the file immediately so it exists only via the descriptor.
tmpfile() creates a temporary file already unlinked, returning a stdio FILE* that is automatically deleted on close(), avoiding filename races altogether.

Chapeter 6 - Processes

A program is a passive executable file (ELF) containing metadata and instructions, while a process is its runtime instance with allocated resources and execution state.
The kernel interprets the program’s binary format (e.g., ELF) to construct the process’s virtual address space and execution context at runtime.

A program file (ELF) contains machine instructions, entry point, initialized data, symbol tables, and dynamic linking info needed to construct a process.
A process is the kernel-managed runtime instance of that program, with its own virtual memory, kernel state (fds, signals, limits), and unique PID retrievable via getpid().

A process has a unique PID (kernel-assigned, reusable after wraparound) and a PPID, forming a parent-child tree rooted at init, with orphaned processes adopted by it.
A program file (ELF) becomes a process with kernel-managed state (memory, fds, signals), and its memory layout includes segments like the read-only text segment holding executable instructions.

The process virtual memory is divided into logical segments: text (read-only, shared code), initialized data, uninitialized (BSS) data, heap (dynamic allocation), and stack (function frames).
The text segment is shared and protected, data segments hold global/static variables, heap grows dynamically, and stack manages function execution state; static memory may persist across calls and affect reentrancy.

The example maps C variables to process memory: globals/statics go to initialized or BSS segments, local (automatic) variables live on the stack, and malloc() allocates from the heap.
Compilers/ABIs may optimize (e.g., registers instead of stack), but the core model remains: stack (execution), heap (dynamic), data/BSS (static), text (code).

An ABI defines low-level runtime conventions (register usage, calling conventions, memory layout) enabling compiled binaries to interact with the kernel/libraries and remain portable across systems with the same ABI.
Virtual memory gives each process an isolated address space, using locality (spatial + temporal) so the kernel efficiently maps virtual addresses to physical memory while optimizing CPU and RAM usage.

Virtual memory maps each process into an isolated address space where heap grows upward, stack grows downward, and kernel space is mapped but protected, separating user and kernel access.
Memory is divided into fixed-size pages, with only a subset (resident set) in RAM; nonresident pages are stored on disk (swap) and loaded on demand via page faults, leveraging locality for efficiency.

Each process has a page table mapping virtual pages to physical frames or disk, enforcing isolation and enabling on-demand loading via page faults.
Processes access only mapped regions; invalid accesses trigger SIGSEGV, while memory regions dynamically grow/shrink via stack, heap (brk/malloc), and mappings (mmap, shared memory).

Virtual memory provides isolation, controlled sharing, and fine-grained protection via page tables mapping virtual pages to physical memory, enabling multiple processes to safely coexist.
The stack grows/shrinks with function calls using stack frames (holding locals, args, return state), typically growing downward, while efficient memory use relies on paging, locality, and demand loading.

The stack grows via allocation of stack frames per function call, storing local variables and call linkage (saved registers, return address), and is logically freed on return but often not physically shrunk.
Kernel uses a separate kernel stack per process during syscalls (since user stack is untrusted), maintaining isolation between user-space execution and kernel execution.

argc gives the number of command-line arguments, while argv is an array of null-terminated strings (ending with NULL), with argv[0] conventionally holding the program name.
Programs can vary behavior based on argv[0], enabling multiple command names to link to the same executable with different functionality.

argv is a NULL-terminated array, so it can be iterated without argc, but portability requires passing it explicitly since it’s only guaranteed in main().
Command-line arguments and environment are stored contiguously above the stack, limited by ARG_MAX, with Linux tying this limit to RLIMIT_STACK for larger argument space.

The environment is a per-process list of name=value strings, inherited as a copy from the parent at process creation, enabling one-time parent→child information transfer.
Changes after fork() are independent, and shells commonly use environment variables (via export) or temporary assignments (NAME=value cmd) to control program behavior.

The environment is a NULL-terminated list of name=value strings accessible via the global environ, and its order is not meaningful.
It can be inspected via /proc/PID/environ or tools like printenv, and is inherited per process but independent after creation.

The environment list is a NULL-terminated array (environ) with no size variable, so it is traversed via pointers until NULL.

setenv() safely adds/modifies variables by copying name=value, avoiding lifetime issues unlike putenv() which directly reuses the provided string.

setjmp() saves the current execution context (registers, stack state), and longjmp() restores it, enabling a nonlocal jump across function calls.

setjmp() saves execution context (program counter + stack pointer), and longjmp() restores it to unwind the stack and jump back across multiple function calls.
This enables structured nonlocal error handling, where control returns to a known checkpoint and execution resumes as if setjmp() returned with a specified nonzero value.

setjmp() establishes a jump target storing execution context, and longjmp() restores it to unwind the stack and resume from that point with a distinct return value.
Its usage is restricted to specific control-expression contexts to ensure well-defined behavior under compiler optimizations and correct state restoration.

setjmp() is restricted to simple control-expression contexts because it must reliably capture execution state; using it in complex expressions (e.g., assignments) can lead to undefined behavior due to compiler/register optimizations.

Chapeter 7 - Memory

Heap allocation grows the program break into the heap segment, which is a contiguous region above BSS used for dynamic memory.
Low-level syscalls brk()/sbrk() adjust the program break, forming the basis for higher-level allocators like malloc().

brk()/sbrk() directly adjust the program break to grow/shrink the heap, with actual physical pages allocated lazily on first access (demand paging).
malloc()/free() build on these primitives, providing portable, safer, thread-friendly dynamic allocation compared to low-level heap manipulation.

malloc() allocates uninitialized heap memory with proper alignment and returns a generic pointer, while free() releases it for reuse via internal free lists rather than shrinking the heap.
Heap allocation is efficient because memory is recycled and avoids frequent sbrk() syscalls, but allocation failure (NULL) must always be checked.

free(NULL) is a safe no-op, but using a pointer after free() or freeing it twice leads to undefined behavior (memory corruption).
free() typically does not reduce the program break; it just recycles memory internally, so heap size may stay unchanged even after many deallocations.

Experiment shows free() only lowers the program break when freeing memory at the top of the heap; otherwise it keeps the break unchanged and reuses freed blocks internally.
Thus, heap size typically stays at its high-water mark, shrinking only when contiguous top regions are released.

free() coalesces adjacent free blocks to reduce fragmentation and only lowers the program break when a sufficiently large contiguous block exists at the heap top.
malloc() reuses freed blocks (splitting if needed) or extends the heap via sbrk() in larger chunks to minimize expensive system calls.

malloc() stores metadata (like block size) just before the returned pointer, and free() uses this hidden header plus in-block pointers to manage a doubly linked free list.
Memory bugs (e.g., buffer overflow, wrong pointer writes) can corrupt this metadata, causing heap corruption, overlapping allocations, or crashes, making such errors hard to debug.

Key safe-usage rules for heap memory:

Never access memory outside allocated bounds (avoid overflow/underflow).
Never free the same pointer twice (double free → corruption).
Only pass pointers to free() that were returned by malloc-family functions.
Always free unused memory in long-running programs to avoid memory leaks.
Use debugging tools (mtrace, mcheck, MALLOC_CHECK_) to detect heap errors early.

MALLOC_CHECK_ provides lightweight runtime detection of common heap errors (print warnings or abort), while advanced tools like Valgrind or Electric Fence give deeper but slower debugging.
mallopt() and mallinfo() allow tuning and inspecting allocator behavior, and functions like calloc() (zero-initialized allocation) and realloc() (resize existing blocks) extend heap management beyond basic malloc()/free().

calloc() allocates memory for arrays and initializes all bytes to 0, unlike malloc() which leaves memory uninitialized.
realloc() resizes an existing block (possibly moving it), preserving old data but leaving any newly added space uninitialized, and may incur copying cost when relocation is required

After realloc(), you must use the returned pointer, since the block may be moved; direct assignment to the original pointer risks losing the reference on failure.
Aligned allocation (memalign, posix_memalign) returns memory at specific power-of-two boundaries, useful for performance or hardware constraints, with posix_memalign being the standardized safer option.

posix_memalign() returns aligned memory safely via an output pointer, and the allocated block must be freed with free().
alloca() allocates memory on the stack frame, automatically freed on function return, making it fast but unsuitable for large allocations or use beyond the calling function.

alloca() allocates memory on the stack (fast, automatic cleanup on function return) but can cause stack overflow with no safe error indication and must not be used inside argument lists.
It’s useful for short-lived allocations and even safer with longjmp() since memory is automatically freed during stack unwinding, unlike malloc() which can leak.

Effective UID/GID determine the permissions used for access control checks, especially important for privileged operations and setuid/setgid programs.

Effective UID/GID control permission checks, and if euid = 0 (root), the process becomes privileged with full access.
SetUID/SetGID programs temporarily set a process’s effective IDs to the executable’s owner/group, allowing controlled privilege escalation for specific tasks.

SetUID/SetGID programs allow controlled privilege escalation by setting the process’s effective UID/GID to the owner/group of the executable at exec() time.
This enables ordinary users to perform restricted operations (e.g., passwd, mount) without full root access, but requires careful design to avoid security vulnerabilities due to elevated privileges.

Saved set-user-ID and set-group-ID store the effective IDs at exec time, enabling a setuid/setgid program to temporarily drop and later regain its elevated privileges safely.
This allows programs to switch between privileged (saved ID) and unprivileged (real ID) modes, which is critical for secure design—using privilege only when necessary.

On Linux, file-system UID/GID are used for file permission checks, rather than effective IDs, though they usually mirror effective IDs.
They exist mainly for historical reasons (e.g., NFS server needs), and only differ when explicitly changed via setfsuid()/setfsgid(), otherwise Linux behaves like standard UNIX.

getuid()/getgid() return the real UID/GID, while geteuid()/getegid() return the effective IDs used for permission checks.
These calls are always successful and are the primary way to inspect a process’s core credentials in user space.

setuid()/setgid() change process credentials, but behavior depends on privilege:
Unprivileged process: can only set effective ID to its real or saved ID (restore/drop privileges in setuid programs).
Privileged process (euid=0): sets real, effective, and saved IDs all at once, permanently dropping privilege if changed to non-root.

seteuid()/setegid() are preferred for temporary privilege switching, letting a process drop to real UID/GID and later regain saved privileges safely.
Privileged processes can set effective IDs arbitrarily, while unprivileged ones are limited to real or saved IDs, ensuring controlled privilege escalation/reduction.

seteuid()/setegid() are glibc wrappers over setreuid()/setresuid() → can implicitly modify saved IDs and allow no-op sets, but this behavior is nonportable (not in SUSv3).
setreuid()/setregid() give fine-grained control of (ruid, euid), but unprivileged processes are constrained to {ruid, euid, saved}, preserving kernel security invariants.

setreuid() behavior for unprivileged processes is partly unspecified by SUSv3, but kernel enforces that ID changes remain within {ruid, euid, saved}, while privileged processes can set arbitrary IDs.
Critical invariant: saved set-user-ID is updated when ruid changes or euid ≠ old ruid, enabling either temporary privilege regain or permanent drop via setreuid(getuid(), getuid()), and ordering (setregid before setreuid) is required to retain privilege.

getresuid()/getresgid() expose (ruid, euid, suid) explicitly, while setresuid()/setresgid() provide full 3-ID control in one atomic syscall—eliminating ambiguity of older APIs.
Kernel invariant: unprivileged changes are restricted to {ruid, euid, suid}, privileged can set arbitrary IDs, and fsuid is always synced to euid, with updates being atomic (all-or-nothing) to preserve consistency.

getgroups() exposes supplementary groups influencing access checks, with size bounded by NGROUPS_MAX and requiring careful sizing or dynamic querying via gidsetsize=0.
setgroups()/initgroups() (privileged only) redefine supplementary groups, crucial during login/session setup to establish correct access domain.
Credential syscalls form a hierarchy: setuidseteuidsetreuidsetresuid, where control granularity increases but portability decreases.
Kernel invariant: effective ID drives permission checks; fsuid shadows euid; supplementary groups extend authority set.
Credential transitions are atomic and tightly constrained, preventing partial updates or privilege escalation outside {ruid, euid, suid} unless privileged.
