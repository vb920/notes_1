### Chapter 41 - Fundamentals of Shared Libraries
1. **Static libraries** are archives of object files copied into the executable at link time.

2. Static libraries are conventionally named `libname.a`.

3. The `ar` tool creates and maintains static libraries by inserting, deleting, or listing object modules.

4. When linking with a static library, the linker extracts only the object files needed by the program.

5. Static linking wastes disk and memory when multiple programs contain duplicate copies of the same library code.

6. Static libraries also make updates painful: every executable using a changed module must be relinked.

7. **Shared libraries** solve this by keeping library code in a separate shared object loaded at run time.

8. Shared libraries are conventionally named with prefix `lib` and suffix `.so`.

9. Multiple processes can map the same shared library code pages into memory, saving RAM.

10. Although shared library code is shared, each process gets its own copy of the library’s global and static data.

11. Shared libraries reduce executable size because object code is not copied into each binary.

12. A library bug fix can often be deployed by replacing the shared library without relinking applications.

13. Shared libraries require **dynamic linking**, performed at run time by the dynamic linker.

14. On Linux ELF systems, the dynamic linker is typically `/lib/ld-linux.so.2` or an architecture-specific equivalent.

15. When a program is linked against a shared library, the executable records a dependency using an ELF `DT_NEEDED` entry.

16. At execution time, the dynamic linker resolves these dependencies, finds the libraries, maps them, and relocates symbols.

17. Shared libraries should be compiled as **position-independent code** using `-fPIC`.

18. Position-independent code allows the library to be loaded at arbitrary virtual addresses.

19. Without `-fPIC`, text relocations may reduce sharing or be unsupported on some architectures.

20. Creating a shared library typically uses:

```bash
gcc -c -fPIC mod1.c mod2.c
gcc -shared -o libfoo.so mod1.o mod2.o
```

21. A shared library may have a **real name**, **soname**, and **linker name**.

22. The **real name** identifies the actual library file, including major and minor version numbers, e.g. `libdemo.so.1.0.2`.

23. The **soname** identifies the ABI-compatible major version, e.g. `libdemo.so.1`.

24. The **linker name** is the version-independent name used at link time, e.g. `libdemo.so`.

25. Executables usually record the soname, not the real filename, allowing compatible minor upgrades.

26. Major version changes indicate ABI incompatibility and require a different soname.

27. Minor version changes should preserve function signatures, semantics, exported variables, and public structures.

28. Sonames and linker names are normally symbolic links pointing toward the latest compatible library version.

29. `ldconfig` maintains `/etc/ld.so.cache` and updates soname symbolic links for installed libraries.

30. The dynamic linker searches for libraries using `DT_RPATH`, `LD_LIBRARY_PATH`, `DT_RUNPATH`, `/etc/ld.so.cache`, then `/lib` and `/usr/lib`.

31. `LD_LIBRARY_PATH` is useful for testing but should not be relied upon in production.

32. For set-user-ID/set-group-ID programs, `LD_LIBRARY_PATH` is ignored for security.

33. The linker option `-Wl,-rpath,<dir>` embeds runtime library search paths into an executable or shared library.

34. `$ORIGIN` inside rpath means “directory containing the executable,” useful for relocatable application bundles.

35. `ldd` displays the shared libraries needed by an executable.

36. `readelf`, `objdump`, and `nm` inspect ELF metadata, dynamic dependencies, sonames, rpaths, and exported symbols.

37. Runtime symbol resolution generally gives symbols in the main executable priority over symbols in shared libraries.

38. `-Wl,-Bsymbolic` can force references inside a shared library to bind preferentially to symbols defined inside that library.

39. Static linking may still be useful for chroot environments, standalone binaries, or immunity from shared-library upgrades.

40. **Core takeaway:** shared libraries trade extra linking/loading complexity for major savings in disk, RAM, upgradeability, and runtime code sharing.

### Chapter 42 - Advanced Features of Shared Libraries

1. **Dynamically loaded libraries** allow a process to load shared objects at run time instead of only at program startup.

2. This is useful for plug-ins, optional modules, and extensible applications where code is loaded only when needed.

3. The `dlopen()` API provides runtime access to the dynamic linker.

4. Core functions are `dlopen()`, `dlsym()`, `dlclose()`, and `dlerror()`.

5. `dlopen()` loads a shared library and returns a handle used for later symbol lookup.

6. `RTLD_LAZY` resolves function symbols only when first used; `RTLD_NOW` resolves all symbols immediately.

7. `RTLD_GLOBAL` makes a library’s symbols available to later-loaded libraries; `RTLD_LOCAL` keeps them private.

8. `dlsym()` searches a loaded library for a named function or variable and returns its address.

9. Since `dlsym()` may validly return `NULL`, `dlerror()` must be used to distinguish “symbol not found” from a real NULL symbol value.

10. `dlclose()` decrements the library reference count and unloads the library only when the count reaches zero.

11. `RTLD_NODELETE` prevents unloading, preserving static data across later reloads.

12. Symbol visibility should be restricted so shared libraries export only their intended ABI.

13. Hidden symbols reduce accidental ABI exposure, avoid symbol interposition surprises, and shrink dynamic symbol tables.

14. Visibility can be controlled with `static`, GCC `visibility("hidden")`, or linker version scripts.

15. Version scripts can also implement **symbol versioning**, allowing one shared library to provide multiple ABI versions of the same function.

16. Constructor and destructor functions run automatically when a shared library is loaded and unloaded.

17. `LD_PRELOAD` forces selected libraries to load first, allowing function interposition for debugging, wrapping, or testing.

18. **Core takeaway:** advanced shared-library mechanisms let programs load code dynamically, control exported ABI, preserve compatibility through symbol versioning, and inspect or override dynamic-linker behavior.

### Chapter 43 - Interprocess Communication Overview
1. UNIX IPC facilities broadly serve three purposes: **communication**, **synchronization**, and, occasionally, signaling.

2. Communication IPC moves data between processes; synchronization IPC coordinates ordering and safe access to shared resources.

3. Data-transfer IPC uses explicit `write()`/`read()` semantics, copying data from user space to kernel space and then kernel space to another process.

4. Pipes, FIFOs, and stream sockets provide **byte-stream IPC**, where message boundaries are not preserved.

5. System V/POSIX message queues and datagram sockets provide **message-oriented IPC**, where each read receives one complete message.

6. Shared memory is the fastest communication mechanism because processes map the same physical pages and avoid kernel-mediated data copying.

7. Shared memory requires separate synchronization, usually semaphores, because all processes can concurrently observe and modify the same data.

8. Data-transfer IPC has built-in synchronization: readers block when no data is available and writers may block when buffers are full.

9. Synchronization primitives include semaphores, file locks, mutexes, and condition variables.

10. Semaphores are kernel-maintained counters used to control access to shared resources.

11. File locks coordinate access to files; `fcntl()` record locks are more flexible than whole-file `flock()` locks.

12. IPC objects differ in naming and lifetime: pipes are unnamed and process-persistent, while System V/POSIX IPC objects usually persist in the kernel until removed.

13. Sockets are the only standard IPC mechanism that naturally supports communication across a network.

14. POSIX IPC generally has cleaner UNIX-like interfaces than System V IPC, but System V IPC is often more widely portable on older systems.

15. **Core takeaway:** choose IPC based on data model, synchronization needs, lifetime, namespace, portability, and performance—not just raw speed.

### CHapter 44 - Pipes and FiFos

1. **Pipes** are one of the oldest UNIX IPC mechanisms, used to connect the output of one process to the input of another.

2. A pipe is a **kernel-maintained byte-stream buffer** with one read end and one write end.

3. Shell pipelines such as:

```bash
ls | wc -l
```

work by connecting `ls` standard output to the pipe write end and `wc` standard input to the pipe read end.

4. Processes using the pipe usually do not know a pipe exists; they simply read from `stdin` or write to `stdout`.

5. Pipes are **byte streams**, not message streams: they preserve byte order but not write boundaries.

6. A reader may read fewer or more bytes than a writer wrote in a single `write()` call.

7. `lseek()` cannot be used on pipes because pipe data is sequential and non-seekable.

8. Pipes are normally **unidirectional**: one descriptor is for reading and the other is for writing.

9. For bidirectional communication, use two pipes or a UNIX domain socket pair.

10. `pipe(filedes)` returns two file descriptors: `filedes[0]` for reading and `filedes[1]` for writing.

11. Pipes are usually created before `fork()`, so the child inherits descriptors referring to the same pipe.

12. This means pipes naturally support communication between **related processes** sharing a common ancestor.

13. After `fork()`, each process must close the pipe ends it does not use.

14. A reader must close its write end so it can observe EOF when all writers are gone.

15. A writer must close its read end so it receives `SIGPIPE`/`EPIPE` when no readers remain.

16. If unused pipe descriptors remain open, readers may block forever instead of seeing EOF.

17. A pipe is destroyed only after all file descriptors referring to it are closed.

18. Reading from an empty pipe normally blocks until data is written.

19. Reading from a pipe whose write end is fully closed returns `0`, meaning EOF.

20. Writing to a pipe whose read end is closed generates `SIGPIPE`; if ignored or caught, `write()` fails with `EPIPE`.

21. Writes of up to `PIPE_BUF` bytes are guaranteed atomic.

22. On Linux, `PIPE_BUF` is typically `4096` bytes.

23. Atomic pipe writes prevent data from multiple writers being interleaved when each write is no larger than `PIPE_BUF`.

24. Writes larger than `PIPE_BUF` may be split and interleaved with writes from other processes.

25. Pipes have limited kernel-buffer capacity; when full, further blocking writes sleep until readers consume data.

26. Pipes can be used for **process synchronization**, not just data transfer.

27. A parent can wait for multiple children by reading from a pipe until all children close their inherited write descriptors.

28. EOF on such a synchronization pipe means all writers have finished or exited.

29. This technique avoids limitations of standard signals, which are not reliably queued.

30. Shells connect filters by combining `pipe()`, `fork()`, `dup2()`, `close()`, and `exec()`.

31. `dup2()` is used to bind the pipe write end to `STDOUT_FILENO` or the pipe read end to `STDIN_FILENO`.

32. Defensive code checks whether the pipe descriptor already equals the target descriptor before closing it.

33. `popen()` simplifies executing a shell command connected to a pipe.

34. `popen(command, "r")` lets the caller read the command’s standard output.

35. `popen(command, "w")` lets the caller write to the command’s standard input.

36. `pclose()` must be used instead of `fclose()` because it closes the pipe and waits for the child shell.

37. `popen()` is convenient but inefficient because it creates a shell and possibly additional child processes.

38. `popen()` and `system()` should not be used in privileged programs, especially with user-supplied input.

39. When using stdio over pipes, buffering matters: output may not be delivered until buffers fill or are flushed.

40. `fflush()` or disabling buffering with `setbuf(fp, NULL)` may be needed for timely pipe communication.

41. A **FIFO** is a named pipe: semantically like a pipe, but represented by a pathname in the filesystem.

42. FIFOs allow communication between unrelated processes because any permitted process can open the FIFO pathname.

43. FIFOs are created with `mkfifo()` or the shell command `mkfifo`.

44. Opening a FIFO for reading normally blocks until another process opens it for writing.

45. Opening a FIFO for writing normally blocks until another process opens it for reading.

46. `O_NONBLOCK` changes FIFO open semantics: read-only open succeeds even without writers, while write-only open fails with `ENXIO` if no reader exists.

47. `O_NONBLOCK` also affects later `read()` and `write()` calls, causing operations that would block to fail with `EAGAIN`.

48. FIFOs are useful for client-server designs: clients can send requests to a well-known server FIFO and receive replies on per-client FIFOs.

49. Since pipes/FIFOs are byte streams, applications must define message framing using delimiters, fixed-size records, or length-prefixed headers.

50. **Core takeaway:** pipes provide efficient byte-stream IPC between related processes, while FIFOs extend the same model to unrelated processes via filesystem names; correctness depends heavily on closing unused ends, respecting EOF/SIGPIPE semantics, and designing message framing explicitly.

## Chapter 49 -  Memory Mappings
1. `mmap()` creates a memory mapping inside a process’s virtual address space.

2. A mapping can be **file-backed**, where memory reflects a region of a file, or **anonymous**, where memory is zero-initialized and has no real file.

3. File mappings let a process access file contents using normal memory loads and stores instead of `read()` and `write()`.

4. Anonymous mappings are commonly used for allocating zero-filled memory, especially large allocations inside `malloc()`.

5. Mappings may be **private** or **shared**.

6. `MAP_PRIVATE` creates a copy-on-write mapping: modifications are visible only to the modifying process.

7. `MAP_SHARED` creates a shared mapping: modifications are visible to other processes mapping the same region.

8. For shared file mappings, modifications are also eventually propagated to the underlying file.

9. Private file mappings are used heavily by the program loader for executable text and initialized data segments.

10. Shared file mappings support both **memory-mapped I/O** and IPC between unrelated processes.

11. Shared anonymous mappings support IPC only between related processes, such as parent and child after `fork()`.

12. A child created by `fork()` inherits the parent’s mappings.

13. Mappings are discarded across `exec()`.

14. The `mmap()` parameters define where the mapping goes, how large it is, its protections, its sharing mode, and the file region being mapped.

15. Passing `addr = NULL` lets the kernel choose a suitable virtual address and is the preferred portable style.

16. `length` is rounded up internally to a page-size multiple.

17. `offset` for file mappings must be page-aligned on Linux.

18. `prot` controls access permissions using `PROT_READ`, `PROT_WRITE`, `PROT_EXEC`, or `PROT_NONE`.

19. Accessing a mapping in a way forbidden by its protection normally raises `SIGSEGV`.

20. `PROT_NONE` is useful for guard pages that intentionally fault on accidental access.

21. `munmap()` removes a mapping from the process’s address space.

22. Unmapping may remove all or only part of a mapping; unmapping the middle can split a mapping.

23. Closing the file descriptor after `mmap()` does not destroy the mapping.

24. For `MAP_SHARED` writable mappings, `msync()` can force synchronization between memory and the underlying file.

25. `MS_SYNC` waits until modified pages reach disk.

26. `MS_ASYNC` schedules write-back and makes changes visible through the kernel’s cache.

27. Linux uses a unified page cache, so mapped memory and ordinary file I/O usually see consistent file contents.

28. Mapping past the end of a file has special behavior: partial trailing pages are zero-filled, but accessing pages wholly beyond the file may raise `SIGBUS`.

29. `MAP_ANONYMOUS` creates mappings not backed by a real file; historically, `/dev/zero` was also used for this purpose.

30. `MAP_SHARED | MAP_ANONYMOUS` is a simple way to share memory between parent and child.

31. `MAP_PRIVATE | MAP_ANONYMOUS` is a simple way to allocate private zero-filled memory.

32. `mremap()` is Linux-specific and can resize or move an existing mapping.

33. If `mremap()` moves a mapping, absolute pointers into the old region become invalid; offsets are safer.

34. `MAP_NORESERVE` controls whether swap space is reserved immediately or lazily.

35. Lazy swap reservation enables memory overcommit but risks the OOM killer if processes later touch too much memory.

36. `MAP_FIXED` forces a mapping to a specific virtual address and can replace existing mappings, so it should be used carefully.

37. `remap_file_pages()` creates nonlinear mappings where file pages appear in a different order in memory, but it is Linux-specific.

38. **Core takeaway:** `mmap()` unifies files, memory allocation, shared memory, and loader behavior under one mechanism: mapping virtual pages to either file-backed or zero-filled physical memory.

### Chapter 50 - Virtual Operations
1. `mprotect()` changes protection on already mapped virtual memory pages.

2. It works at **page granularity**, so `addr` should be page-aligned and `length` is rounded up to a page boundary.

3. Valid protections are `PROT_NONE`, `PROT_READ`, `PROT_WRITE`, and `PROT_EXEC`.

4. If a process violates memory protection, the kernel sends `SIGSEGV`.

5. `PROT_NONE` is useful for **guard pages**, helping detect accidental stack/heap overflows.

6. `mprotect()` is commonly used with `mmap()` regions to change permissions after mapping.

7. Memory locking prevents virtual memory pages from being swapped out to disk.

8. Locking memory is useful for **real-time performance**, because locked pages cannot trigger swap-in delays.

9. It is also useful for **security**, preventing sensitive data such as passwords or keys from being written to swap.

10. `RLIMIT_MEMLOCK` controls how much memory an unprivileged process may lock.

11. `mlock(addr, length)` locks a specific virtual memory range into RAM.

12. `munlock(addr, length)` removes the lock from that range.

13. Locking applies to whole pages, even if the requested address range is not page-aligned.

14. Memory locks are removed automatically on process termination, `exec()`, `munmap()`, or mapping replacement.

15. Memory locks are **not inherited across `fork()`** and are **not preserved across `exec()`**.

16. `mlockall(MCL_CURRENT)` locks all pages currently mapped in the process.

17. `mlockall(MCL_FUTURE)` locks pages that are mapped or allocated later.

18. `munlockall()` removes all memory locks held by the process.

19. `mincore()` reports which pages in a virtual memory range are currently resident in physical RAM.

20. `mincore()` is useful for observing page residency, but its result can become stale immediately unless pages are locked.

21. The residency result is returned as a byte vector, where the low bit indicates whether the page is in RAM.

22. `madvise()` lets a process tell the kernel how it expects to use a memory region.

23. `MADV_RANDOM` tells the kernel not to perform aggressive read-ahead.

24. `MADV_SEQUENTIAL` tells the kernel to expect one-time sequential access and optimize read-ahead/freeing.

25. `MADV_WILLNEED` asks the kernel to read pages ahead before future access.

26. `MADV_DONTNEED` tells the kernel the process no longer needs those pages resident.

27. On Linux, `MADV_DONTNEED` for private mappings may discard modified pages, so later access reloads from file or zeros.

28. `posix_madvise()` is the standardized variant of `madvise()`, but Linux/glibc behavior has historically differed for `DONTNEED`.

29. These APIs are not primarily IPC APIs, but they are often useful with shared memory and memory-mapped files.

30. **Core takeaway:** `mprotect()` controls page permissions, `mlock()` controls residency, `mincore()` observes residency, and `madvise()` gives the kernel performance hints about future memory access.
