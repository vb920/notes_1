### Chapter 31 -Thread Safety

1. A function is **thread-safe** if multiple threads can call it concurrently without corrupting shared state or producing undefined results.

2. The usual cause of non-thread-safety is use of **global or static storage** shared by all threads.

3. Example: a function incrementing a global variable without locking is unsafe because concurrent read-modify-write sequences race.

4. One simple way to make a function thread-safe is to serialize the whole function with a mutex.

5. Whole-function locking is correct but may reduce parallelism because only one thread can execute the function at a time.

6. A better design protects only the **critical sections** that access shared state, allowing more concurrency.

7. A **reentrant function** is stronger than thread-safe: it avoids shared static/global state and does not need mutexes.

8. Reentrant functions store state in caller-supplied buffers or automatic variables, making each invocation independent.

9. Some functions cannot easily be reentrant because they inherently manipulate global structures, such as `malloc()` managing the process heap.

10. Older UNIX APIs are often nonreentrant because they return pointers to statically allocated buffers.

11. Examples include traditional functions such as `asctime()`, `ctime()`, `strtok()`, `strerror()`, and `getpwnam()`.

12. Many reentrant replacements use the `_r` suffix, such as `strtok_r()`, `localtime_r()`, and `getpwnam_r()`.

13. In threaded programs, prefer reentrant APIs or modern thread-safe alternatives such as `getaddrinfo()` over legacy resolver functions.

14. `pthread_once()` solves the problem of **one-time initialization** in libraries used by multiple threads.

15. It guarantees that an initialization routine runs exactly once, regardless of how many threads call it concurrently.

16. The control object must be statically initialized:

```c
pthread_once_t once = PTHREAD_ONCE_INIT;
```

17. `pthread_once()` is commonly used to lazily initialize mutexes, keys, or other shared library state.

18. **Thread-specific data (TSD)** provides each thread with its own persistent copy of data associated with a library function.

19. TSD is useful for making old nonreentrant interfaces thread-safe without changing their function signatures.

20. A TSD key, created with `pthread_key_create()`, identifies one logical per-thread data slot.

21. The key is process-wide, but each thread has its own value associated with that key.

22. `pthread_getspecific()` retrieves the calling thread’s value for a key.

23. `pthread_setspecific()` stores a thread-specific pointer for the calling thread.

24. On first use, a thread usually sees `NULL`, allocates storage with `malloc()`, and stores the pointer using `pthread_setspecific()`.

25. `pthread_key_create()` can register a destructor that runs automatically when a thread exits.

26. The destructor frees that thread’s TSD buffer, preventing memory leaks in long-running threaded programs.

27. A non-thread-safe `strerror()` using a single static buffer can be fixed by giving each thread its own error-string buffer via TSD.

28. **Thread-local storage (TLS)** is a simpler nonstandard alternative using variables declared with `__thread`.

29. TLS variables are global/static in form but physically separate per thread and automatically destroyed when the thread exits.

30. **Core takeaway:** thread safety prevents concurrent corruption; reentrancy avoids shared state entirely; TSD/TLS provide per-thread state when legacy APIs require persistent internal storage.


### Chapter 32 - Thread Cancellation
1. **Thread cancellation** is a cooperative POSIX mechanism where one thread sends another thread a request to terminate.

2. `pthread_cancel(thread)` sends a cancellation request but returns immediately; it does **not wait** for the target thread to die.

3. Actual cancellation depends on the target thread’s **cancelability state** and **cancelability type**.

4. `PTHREAD_CANCEL_ENABLE` means the thread may respond to cancellation requests; this is the default for new threads.

5. `PTHREAD_CANCEL_DISABLE` means cancellation requests remain pending until cancellation is re-enabled.

6. Temporarily disabling cancellation is useful around code that must complete atomically, such as updating shared state or acquiring/releasing resources.

7. Cancelability type controls **when** an enabled thread acts on a pending cancellation request.

8. `PTHREAD_CANCEL_DEFERRED` is the default and safest mode: cancellation occurs only at defined **cancellation points**.

9. `PTHREAD_CANCEL_ASYNCHRONOUS` allows cancellation at almost any instruction, but is dangerous and rarely appropriate.

10. Cancellation points are usually blocking functions such as `read()`, `write()`, `select()`, `poll()`, `pthread_cond_wait()`, `pthread_join()`, `waitpid()`, and `sleep()`.

11. Deferred cancellation makes thread termination predictable because cleanup happens only at known library call boundaries.

12. A compute-bound thread that never calls cancellation-point functions may never respond to `pthread_cancel()`.

13. Such threads should periodically call `pthread_testcancel()`, which exists solely as an explicit cancellation point.

14. If a canceled thread is joinable, another thread must still call `pthread_join()` to reclaim its resources.

15. When joining a canceled thread, `pthread_join()` returns the special value `PTHREAD_CANCELED`.

16. Cancellation is not equivalent to `kill`; it is a structured, thread-level termination request handled inside the Pthreads runtime.

17. Cancellation can leave shared state inconsistent if a thread is terminated while holding locks or owning resources.

18. **Cleanup handlers** solve this by registering functions that run automatically when a thread is canceled.

19. Cleanup handlers are installed with `pthread_cleanup_push()` and removed with `pthread_cleanup_pop()`.

20. Each thread has a stack of cleanup handlers; on cancellation, handlers execute in LIFO order.

21. Cleanup handlers are commonly used to unlock mutexes, free heap memory, close descriptors, or restore invariants.

22. `pthread_cleanup_pop(nonzero)` removes and also executes the cleanup handler, useful for normal non-canceled completion.

23. On Linux, `pthread_cleanup_push()` and `pthread_cleanup_pop()` may be macros introducing braces, so they must appear in matched lexical pairs.

24. Deferred cancellation plus cleanup handlers gives a disciplined model: cancellation occurs only at expected points, where the program knows which resources are owned.

25. Asynchronous cancellation breaks this reasoning because the thread may be canceled before, during, or after acquiring resources.

26. With asynchronous cancellation, cleanup handlers may not know whether memory was allocated or whether a mutex was locked.

27. A thread canceled inside functions like `malloc()` may corrupt library-internal state, causing process-wide instability.

28. Therefore asynchronously cancelable threads should not hold mutexes, allocate memory, or call most library functions.

29. **Core invariant:** make cancellation deferred, place cancellation points only where invariants are stable, and protect acquired resources with cleanup handlers.

30. **Most important takeaway:** POSIX cancellation is cooperative termination; safe cancellation requires deferred mode, explicit cleanup, and careful resource ownership discipline.


### Chapter 33- Thread Details
1. Each POSIX thread has its own fixed-size stack allocated at thread creation time.

2. On Linux/x86-32, non-main threads commonly default to a **2 MB stack**, while the main thread usually has a much larger growable stack region.

3. Thread stack size matters because every thread consumes part of the process’s shared virtual address space.

4. Large per-thread stacks limit the maximum number of threads that can coexist in one process.

5. `pthread_attr_setstacksize()` lets a program tune stack size for threads created with a thread-attributes object.

6. Bigger stacks are useful for deep recursion or large automatic variables; smaller stacks allow more threads.

7. Under NPTL, `RLIMIT_STACK` influences the default stack size for new threads, but it must be set before program startup.

8. UNIX signals were designed for processes before threads existed, so signal/thread interaction is inherently subtle.

9. Signal dispositions are **process-wide**: if one thread installs a handler or ignores a signal, that disposition applies to all threads.

10. Signal masks are **per-thread**: each thread independently controls which signals it blocks.

11. Signal actions such as terminate, stop, or continue affect the entire process, not just one thread.

12. Hardware-exception signals such as `SIGSEGV`, `SIGBUS`, `SIGFPE`, and `SIGILL` are thread-directed because they arise from one thread’s execution.

13. `SIGPIPE` from writing to a broken pipe is also directed to the thread that performed the write.

14. Signals sent using `kill()` are process-directed, while signals sent using `pthread_kill()` are thread-directed.

15. For a process-directed handled signal, the kernel chooses one eligible unblocked thread to run the signal handler.

16. Because signal handlers may run in any eligible thread, shared state touched by handlers is especially dangerous in multithreaded programs.

17. A new thread inherits a copy of its creator’s signal mask.

18. `pthread_sigmask()` is the portable API for changing a thread’s signal mask in multithreaded programs.

19. `sigprocmask()` should be avoided in threaded programs because its behavior is unspecified by POSIX there.

20. The sane design for asynchronous signals in threaded programs is: block signals in all threads, then dedicate one thread to synchronously wait for them.

21. The dedicated signal thread can use `sigwait()`, `sigwaitinfo()`, or `sigtimedwait()`.

22. This converts asynchronous signal delivery into synchronous control flow, allowing safe use of mutexes, condition variables, and normal library functions.

23. No Pthreads function is async-signal-safe, so calling pthread APIs directly from a signal handler is unsafe.

24. `pthread_kill()` sends a signal to a specific thread within the same process.

25. On Linux, `pthread_kill()` is implemented using thread-group-aware kernel signaling such as `tgkill()`.

26. If any thread calls `exec()`, the entire process image is replaced and all other threads immediately vanish.

27. During `exec()`, thread-specific-data destructors and cleanup handlers for vanished threads are not run.

28. After `exec()`, old mutexes, condition variables, stacks, and thread state disappear with the replaced address space.

29. If a multithreaded process calls `fork()`, only the calling thread exists in the child.

30. This is dangerous because mutexes and global data are copied into the child, possibly while locked or inconsistent.

31. A child created by `fork()` from a multithreaded process should usually call `exec()` immediately.

32. `pthread_atfork()` allows libraries to register prepare, parent, and child handlers around `fork()`.

33. Fork handlers are mainly a mitigation for mutex/library-state consistency across `fork()`.

34. If any thread calls `exit()`, or the main thread returns from `main()`, all threads in the process terminate immediately.

35. `pthread_exit()` terminates only the calling thread, unlike `exit()`.

36. Linux threading implementations use the **1:1 model**, meaning each POSIX thread maps to one kernel scheduling entity.

37. The 1:1 model allows true parallelism on multiple CPUs and prevents one blocking syscall from stopping all threads.

38. LinuxThreads was the older implementation and had major POSIX deviations because its threads had distinct PIDs.

39. NPTL is the modern implementation; it uses `clone()` with flags such as `CLONE_VM`, `CLONE_FILES`, `CLONE_SIGHAND`, `CLONE_THREAD`, `CLONE_SETTLS`, and `CLONE_CHILD_CLEARTID`.

40. **Core takeaway:** NPTL makes Linux threads kernel-scheduled tasks sharing one process identity, but signals, `fork()`, and `exec()` remain tricky because they come from older process-centric UNIX semantics.

### Chapter 34 - Process Groups and Controls
1. **Process groups** and **sessions** form a two-level hierarchy used mainly to implement shell job control.

2. A **process group** is a set of related processes sharing the same process group ID, or **PGID**.

3. A **session** is a set of related process groups sharing the same session ID, or **SID**.

4. A process group is often equivalent to a shell **job**, such as a command or pipeline.

5. A process group leader is the process whose PID equals the process group ID.

6. A session leader is the process whose PID equals the session ID.

7. New processes inherit their parent’s process group ID and session ID across `fork()`.

8. The usual login shell is both the **session leader** and the **controlling process** for the terminal.

9. All jobs started by an interactive shell normally belong to the same session as the shell.

10. Each command or pipeline launched by the shell is placed into a separate process group.

11. A foreground command becomes the terminal’s **foreground process group**.

12. A command ending in `&` becomes a **background process group**.

13. A session may have at most one controlling terminal.

14. A terminal may be the controlling terminal of at most one session.

15. Only the foreground process group is allowed to read freely from the controlling terminal.

16. Terminal-generated signals such as `SIGINT`, `SIGQUIT`, and `SIGTSTP` are delivered to all members of the foreground process group.

17. Thus, pressing `Ctrl-C` sends `SIGINT` to the whole foreground job, not just one process.

18. Pressing `Ctrl-\` sends `SIGQUIT` to the foreground process group.

19. Pressing `Ctrl-Z` sends `SIGTSTP`, normally stopping the foreground job.

20. The controlling process receives `SIGHUP` when the controlling terminal disconnects.

21. `/proc/PID/stat` exposes kernel-maintained PGID, SID, controlling terminal, and foreground process group information.

22. `getpgrp()` returns the calling process’s process group ID.

23. If `getpgrp() == getpid()`, the process is the leader of its process group.

24. `setpgid(pid, pgid)` changes a process’s process group.

25. `setpgid(0, 0)` makes the calling process leader of a new process group.

26. A process can move only itself or one of its children into a process group.

27. A process cannot move a child into another process group after that child has already performed `exec()`.

28. This restriction prevents programs from unexpectedly changing process groups after execution has begun.

29. Job-control shells avoid races by calling `setpgid()` in both parent and child immediately after `fork()`.

30. The shell usually uses the PID of the first pipeline process as the PGID for the entire pipeline.

31. `setsid()` creates a new session and a new process group.

32. After successful `setsid()`, the caller’s PID, PGID, and SID all become equal.

33. `setsid()` also detaches the process from any controlling terminal.

34. A process group leader cannot call `setsid()`.

35. Therefore daemon-style code typically does `fork(); parent exits; child calls setsid()`.

36. This guarantees the child is not already a process group leader.

37. A new session initially has no controlling terminal.

38. On Linux/System V semantics, a session leader acquires a controlling terminal by opening a terminal device not already controlled by another session.

39. Opening with `O_NOCTTY` prevents acquiring a controlling terminal.

40. `/dev/tty` refers to the calling process’s controlling terminal.

41. If a process has no controlling terminal, opening `/dev/tty` fails with `ENXIO`.

42. `tcgetpgrp(fd)` returns the foreground process group ID of the controlling terminal.

43. `tcsetpgrp(fd, pgid)` changes the terminal’s foreground process group.

44. Job-control shells use `tcsetpgrp()` when moving jobs between foreground and background.

45. `fg` moves a stopped or background job into the foreground and gives it terminal control.

46. `bg` resumes a stopped job in the background by sending `SIGCONT`.

47. `jobs` shows the shell’s remembered background and stopped process groups.

48. `SIGCONT` is special: it can be sent to any process in the same session, even if normal UID permission checks would fail.

49. This exception is required so shells can resume stopped set-user-ID jobs.

50. A background process attempting to read from the controlling terminal receives `SIGTTIN`.

51. The default action of `SIGTTIN` is to stop the process group.

52. If `SIGTTIN` is blocked or ignored, the terminal read fails with `EIO`.

53. Background writes are usually allowed.

54. If the terminal `TOSTOP` flag is set, background writes generate `SIGTTOU`.

55. The default action of `SIGTTOU` is also to stop the process group.

56. Some terminal-control operations, such as `tcsetpgrp()` and `tcsetattr()`, may generate `SIGTTOU` for background jobs regardless of `TOSTOP`.

57. When a terminal disconnects, the kernel sends `SIGHUP` to the controlling process.

58. Interactive shells catch `SIGHUP` so they can forward it to jobs before terminating.

59. `nohup` starts a command with `SIGHUP` ignored, protecting it from terminal hangup termination.

60. `disown` removes a job from the shell’s job table, preventing the shell from sending it `SIGHUP`.

61. If the controlling process terminates, the kernel detaches the session from the terminal.

62. On Linux, this also causes `SIGHUP` and often `SIGCONT` to be sent to the foreground process group.

63. A process group becomes **orphaned** when no member has a parent in another process group within the same session.

64. If a process group becomes orphaned while containing stopped processes, the kernel sends `SIGHUP` followed by `SIGCONT` to every member.

65. This prevents stopped orphaned jobs from remaining suspended forever with no shell able to resume them.

66. For orphaned process groups, terminal stop signals such as `SIGTTIN`, `SIGTTOU`, and `SIGTSTP` are suppressed or converted into `EIO` behavior where appropriate.

67. Programs that handle `SIGTSTP` must usually restore default disposition, re-raise `SIGTSTP`, and reinstall the handler after `SIGCONT`.

68. Screen-oriented programs such as `vi` and `less` handle job-control signals to restore terminal state before stopping and redraw the screen after resuming.

69. **Core invariant:** process groups let the kernel and shell treat an entire pipeline as one controllable job.

70. **Most important takeaway:** sessions connect process groups to a controlling terminal; job control is the coordinated behavior of the shell, terminal driver, process groups, and signals.

### CHapter 35 - Process Priorities and Scheduling
1. Linux’s default CPU scheduling model is **round-robin time-sharing**, where runnable processes receive CPU time slices to balance fairness and responsiveness.

2. A process normally runs until its time slice expires, it blocks, exits, or voluntarily gives up the CPU.

3. The **nice value** is the traditional UNIX mechanism for influencing CPU scheduling priority under normal time-sharing.

4. Nice values range from **-20** high priority to **+19** low priority, with default value **0**.

5. A higher nice value means the process is being “nicer” to others by accepting less CPU preference.

6. Nice values are inherited across `fork()` and preserved across `exec()`.

7. Nice values do not create strict priority ordering; they act as scheduler weights.

8. Even a very low-priority process is not completely starved under normal scheduling.

9. `getpriority()` retrieves a process, process-group, or user’s nice value.

10. `setpriority()` changes the nice value of a process, process group, or all processes owned by a user.

11. Since `getpriority()` can validly return `-1`, callers must clear `errno` before the call and check `errno` afterward.

12. Privileged processes with `CAP_SYS_NICE` can freely raise priority or modify other processes’ priorities.

13. Unprivileged processes may generally lower their own priority but need resource-limit support to raise it.

14. Linux `RLIMIT_NICE` allows controlled priority elevation by unprivileged processes.

15. POSIX realtime scheduling gives stronger control than nice values by using explicit scheduling **policies** and **priorities**.

16. Linux supports realtime policies `SCHED_FIFO` and `SCHED_RR`.

17. Realtime processes always outrank normal `SCHED_OTHER` processes.

18. Linux realtime priorities range from **1** lowest to **99** highest.

19. A runnable higher-priority realtime process completely preempts lower-priority realtime or normal processes.

20. `SCHED_RR` is realtime round-robin scheduling among processes of the same priority.

21. Under `SCHED_RR`, equal-priority processes share CPU time using fixed time slices.

22. If a `SCHED_RR` process is preempted by a higher-priority process, it later resumes with its remaining time slice.

23. `SCHED_FIFO` is realtime first-in-first-out scheduling with no time slice.

24. A `SCHED_FIFO` process keeps the CPU until it blocks, exits, yields, or is preempted by a higher-priority process.

25. `SCHED_FIFO` is powerful but dangerous because a runaway process can monopolize the CPU.

26. Linux also provides nonstandard `SCHED_BATCH` for batch workloads and `SCHED_IDLE` for extremely low-priority work.

27. `sched_setscheduler()` changes both scheduling policy and priority.

28. `sched_setparam()` changes only scheduling priority while preserving policy.

29. `sched_getscheduler()` and `sched_getparam()` retrieve a process’s current scheduling policy and priority.

30. Realtime priority ranges should be queried using `sched_get_priority_min()` and `sched_get_priority_max()` rather than hard-coded.

31. `RLIMIT_RTPRIO` controls how far unprivileged processes may raise realtime priority.

32. `RLIMIT_RTTIME` limits how much CPU time a realtime process may consume in one burst before receiving `SIGXCPU`.

33. `SCHED_RESET_ON_FORK` prevents children from inheriting privileged realtime policies or negative nice values.

34. `sched_yield()` lets a realtime process voluntarily move to the back of its priority queue.

35. `sched_rr_get_interval()` returns the `SCHED_RR` time slice for a process.

36. CPU affinity controls which CPUs a process or thread may run on.

37. Linux’s scheduler already tries to preserve **soft affinity** to avoid cache penalties from moving tasks between CPUs.

38. `sched_setaffinity()` imposes hard CPU affinity, restricting execution to a chosen CPU set.

39. CPU affinity is actually **per-thread** on Linux, so individual threads can be pinned separately.

40. **Core takeaway:** nice values provide weighted fairness, realtime policies provide strict priority control, and CPU affinity constrains where kernel scheduling may place a task.

### CHapter 36 - Process Resources
1. `getrusage()` reports kernel-maintained resource-consumption statistics for a process, its waited-for children, or, on Linux, the calling thread.

2. `RUSAGE_SELF` measures the calling process; `RUSAGE_CHILDREN` accumulates usage of children that have terminated **and been waited for**.

3. Child resource usage is transferred upward only when `wait()`/`waitpid()` reaps the child.

4. `RUSAGE_THREAD` is Linux-specific and reports usage for the calling thread.

5. Important `struct rusage` fields include user CPU time, system CPU time, page faults, block I/O counts, and voluntary/involuntary context switches.

6. `ru_utime` measures time spent executing user-space code; `ru_stime` measures time spent executing in kernel mode.

7. Minor page faults (`ru_minflt`) do not require disk I/O; major page faults (`ru_majflt`) require I/O and are much more expensive.

8. Voluntary context switches occur when a process blocks or yields before its time slice expires.

9. Involuntary context switches occur when the scheduler preempts the process, usually due to time-slice expiry or higher-priority runnable work.

10. Resource limits are maintained per process and restrict consumption of kernel-managed resources.

11. `getrlimit()` retrieves a resource limit; `setrlimit()` modifies one.

12. Each resource has a **soft limit** and a **hard limit**.

13. The soft limit is the active enforcement boundary.

14. The hard limit is the ceiling to which an unprivileged process may raise its soft limit.

15. Privileged processes with `CAP_SYS_RESOURCE` may raise hard limits.

16. Unprivileged processes may lower hard limits, but this is effectively irreversible.

17. Resource limits are inherited across `fork()` and preserved across `exec()`.

18. `RLIM_INFINITY` means no enforced limit for that resource.

19. `RLIMIT_AS` limits total virtual address space; exceeding it makes `brk()`, `mmap()`, or allocator calls fail with `ENOMEM`.

20. `RLIMIT_DATA` limits the data segment and heap growth, mainly affecting `brk()`/`sbrk()`-based allocation.

21. `RLIMIT_STACK` limits stack growth; exceeding it typically triggers `SIGSEGV`.

22. Handling stack-limit `SIGSEGV` safely may require an alternate signal stack.

23. `RLIMIT_CORE` caps core dump size; setting it to `0` disables core file generation.

24. `RLIMIT_CPU` limits consumed CPU seconds; the soft limit sends `SIGXCPU`, while the hard limit causes `SIGKILL`.

25. `RLIMIT_FSIZE` limits file growth; exceeding it sends `SIGXFSZ` and causes writes/truncates to fail with `EFBIG`.

26. `RLIMIT_NOFILE` limits the maximum file descriptor number plus one, controlling how many descriptors a process may allocate.

27. `RLIMIT_NPROC` limits the number of processes/threads for a real user ID; `fork()`, `vfork()`, or `clone()` may fail with `EAGAIN`.

28. Linux-specific limits such as `RLIMIT_MSGQUEUE`, `RLIMIT_SIGPENDING`, `RLIMIT_RTPRIO`, and `RLIMIT_RTTIME` constrain message queues, queued signals, realtime priority, and realtime CPU bursts.

29. Some limits are measured per process, while others are charged against all processes with the same real user ID.

30. **Core takeaway:** `getrusage()` is for observing resource consumption, while `setrlimit()`/`getrlimit()` define kernel-enforced boundaries that prevent processes or users from exhausting system resources.

### CHapter 37 - Daemons
1. A **daemon** is a long-lived background process that normally starts at boot and runs until system shutdown.

2. A daemon has **no controlling terminal**, so it is immune to terminal-generated job-control signals such as `SIGINT`, `SIGTSTP`, and terminal-originated `SIGHUP`.

3. Common daemons include `cron`, `sshd`, `httpd`, and `inetd`; by convention many daemon names end in `d`.

4. Some Linux daemons are actually **kernel threads**, shown by `ps` with names in square brackets, e.g. `[kworker]`.

5. To create a daemon, the process first calls `fork()` and the parent exits, allowing the shell to regain control.

6. The child is guaranteed not to be a process group leader, enabling a successful `setsid()` call.

7. `setsid()` creates a new session, makes the child a session leader, creates a new process group, and detaches it from any controlling terminal.

8. A second `fork()` is commonly performed after `setsid()` so the final daemon is **not a session leader** and therefore cannot reacquire a controlling terminal.

9. A daemon usually calls `umask(0)` so future file creations are controlled explicitly by requested permissions.

10. A daemon should usually `chdir("/")` to avoid keeping some other mounted filesystem busy forever.

11. Long-lived daemons must close inherited file descriptors to avoid descriptor leaks and accidental filesystem references.

12. File descriptors `0`, `1`, and `2` are normally reopened on `/dev/null`.

13. Reopening standard descriptors prevents library code from accidentally failing or corrupting unrelated files later opened as stdout/stderr.

14. Daemonization is therefore mainly about detaching from shell/session/terminal state and sanitizing process resources.

15. During shutdown, daemons typically receive `SIGTERM`; if cleanup is needed, they should handle it quickly before `SIGKILL` follows.

16. Since daemons are long-lived, memory leaks and file descriptor leaks are especially dangerous.

17. Many daemons must enforce a **single active instance**, often using lock files or advisory file locking.

18. Because daemons have no controlling terminal, the kernel normally does not send them `SIGHUP`.

19. Daemons conventionally reuse `SIGHUP` as an administrative signal meaning “reload configuration” or “reopen log files.”

20. A daemon’s `SIGHUP` handler should usually just set a `volatile sig_atomic_t` flag, leaving real reinitialization to the main loop.

21. Reopening logs after `SIGHUP` is important for log rotation, because renaming a log file does not affect an already-open file descriptor.

22. `syslog` provides centralized logging for daemons that cannot write diagnostics to a terminal.

23. User processes log messages via `syslog(3)`, which sends formatted records to `/dev/log`.

24. `syslogd` receives messages and routes them according to facility and severity rules in `/etc/syslog.conf`.

25. A syslog **facility** identifies the source class, such as `LOG_DAEMON`, `LOG_AUTH`, `LOG_MAIL`, or `LOG_LOCAL0`.

26. A syslog **level** identifies severity, ranging from `LOG_EMERG` down to `LOG_DEBUG`.

27. `openlog()` sets default logging identity, options, and facility; `syslog()` writes messages; `closelog()` closes the logging connection.

28. Use `LOG_PID` when multiple daemon worker processes may log messages, so individual PIDs appear in logs.

29. Never pass user input directly as the syslog format string; use `syslog(priority, "%s", user_string)` to avoid format-string vulnerabilities.

30. **Core takeaway:** daemon creation detaches a process from terminal/session inheritance, while `SIGHUP` and `syslog` provide the standard control and observability mechanisms for long-running background services.

### Chapter 38 - Writing  Secure Privileged Programs
1. Privileged programs are dangerous because bugs execute with elevated authority, potentially compromising files, devices, credentials, or the whole system.

2. Prefer **not** to write set-user-ID/set-group-ID programs; if privilege is unavoidable, isolate the privileged operation into the smallest possible helper.

3. Avoid set-user-ID-root when a dedicated unprivileged user/group can perform the task with narrower authority.

4. Apply the **least privilege principle**: drop effective privileges immediately, reacquire them only around privileged operations, and permanently drop them once no longer needed.

5. Temporary privilege dropping uses saved set-user-ID/set-group-ID; permanent dropping must reset real, effective, and saved IDs.

6. Credential-changing calls have subtle platform differences, so secure Linux code should prefer `setresuid()`/`setresgid()` and verify the resulting IDs.

7. Never execute shells or interpreters with privilege; avoid `system()`, `popen()`, `execlp()`, and `execvp()` in privileged code.

8. Before `exec()`, permanently drop privileges and close all unnecessary file descriptors, or mark them `FD_CLOEXEC`.

9. Sensitive data such as passwords should be erased from memory immediately after use and protected from core dumps via `RLIMIT_CORE = 0`.

10. Consider confinement mechanisms such as Linux capabilities, `chroot()` jails, containers, or virtual machines to limit blast radius.

11. Signals can create TOCTOU windows; privileged programs must consider being stopped, resumed, or interrupted at hostile points.

12. For file operations, avoid pathname races: use `open()` followed by `fstat()`, not `stat()` followed by `open()`.

13. Use `O_EXCL`, safe permissions, careful `umask`, `fchown()`, and `fchmod()` to avoid momentarily exposing writable privileged files.

14. Do not trust environment variables such as `PATH`, `IFS`, or loader-related variables; sanitize or rebuild the environment before privileged work or `exec()`.

15. Validate all untrusted inputs: command-line arguments, files, IPC data, network packets, CGI values, and environment strings.

16. Avoid buffer overruns by rejecting unsafe functions like `gets()` and carefully using bounded alternatives such as `snprintf()`.

17. Defensive servers should resist denial-of-service by using timeouts, throttling, resource limits, bounded data structures, and controlled logging.

18. **Core takeaway:** secure privileged programming means minimizing privilege, eliminating races, distrusting all inputs/environment, closing privilege leaks, and failing safely on every unexpected condition.

### Chapter 39 - Capabilities

1. Linux capabilities replace the traditional **all-or-nothing root privilege model** with fine-grained privilege bits.

2. Instead of checking only whether `euid == 0`, the kernel checks whether the process has the specific capability required for an operation.

3. Capabilities reduce privilege blast radius: a program can be allowed to change system time without also gaining unrestricted filesystem access.

4. Capabilities are mostly invisible to legacy programs because processes running as root are usually granted the full capability set.

5. Each process/thread has three main capability sets: **permitted**, **effective**, and **inheritable**.

6. The **permitted set** is the upper bound of capabilities the process may use.

7. Dropping a capability from the permitted set is permanent unless a later `exec()` grants it again.

8. The **effective set** contains the capabilities currently active for kernel privilege checks.

9. A process may temporarily drop a capability from effective and later restore it if it remains in permitted.

10. The **inheritable set** controls which capabilities may survive across `exec()`.

11. Capabilities are technically **per-thread** attributes, not purely per-process attributes.

12. A child created by `fork()` inherits copies of its parent’s capability sets.

13. Executable files can also carry capability sets stored in the `security.capability` extended attribute.

14. File capabilities allow privileged execution without making the file set-user-ID-root.

15. The file **permitted set** grants capabilities into the process’s permitted set during `exec()`.

16. The file **effective bit** decides whether new permitted capabilities also become immediately effective.

17. Capability-dumb programs need the file effective bit set because they do not manually raise capabilities.

18. Capability-aware programs should start with effective capabilities disabled and raise only what they need.

19. During `exec()`, Linux computes new capabilities roughly from process inheritable, file inheritable, file permitted, and the bounding set.

20. The **capability bounding set** limits which capabilities can ever be gained through `exec()`.

21. Dropping a capability from the bounding set is irreversible and prevents later regain through file capabilities.

22. `setcap` assigns file capabilities; `getcap` displays them.

23. Example: assigning `cap_sys_time=ep` to `date` allows it to set system time without full root privilege.

24. UID transitions still interact with capabilities for compatibility with traditional UNIX root semantics.

25. Changing all user IDs from root to non-root normally clears permitted and effective capabilities.

26. Changing effective UID from `0` to nonzero clears effective capabilities but may preserve permitted ones.

27. The preferred API for manipulating capabilities is **libcap**, not raw `capget()`/`capset()`.

28. Secure capability-aware programs should raise a capability only around the privileged operation, then drop it immediately.

29. `securebits` can disable special root semantics, creating a more purely capability-based environment.

30. **Core takeaway:** capabilities split root privilege into independent kernel-checked powers, enabling least-privilege execution far safer than set-user-ID-root.

### CHapter 40 - Login Accounting
1. **Login accounting** records both currently logged-in users and historical login/logout activity.

2. Linux maintains two main accounting files: **utmp** and **wtmp**.

3. `/var/run/utmp` records users currently logged in; tools like `who(1)` read this file.

4. `/var/log/wtmp` is an append-only audit trail of logins and logouts; tools like `last(1)` read it.

5. On login, a `USER_PROCESS` record is written to `utmp` and appended to `wtmp`.

6. On logout, the `utmp` record is overwritten with a `DEAD_PROCESS` record, and a logout record is appended to `wtmp`.

7. Each accounting entry is represented by `struct utmpx`, which stores fields such as username, PID, terminal line, host, session ID, timestamp, and record type.

8. Important record types include `BOOT_TIME`, `INIT_PROCESS`, `LOGIN_PROCESS`, `USER_PROCESS`, and `DEAD_PROCESS`.

9. Login records are keyed mainly by terminal identity using fields such as `ut_line` and `ut_id`.

10. The `utmpx` API provides functions such as `setutxent()`, `getutxent()`, `getutxid()`, `getutxline()`, and `endutxent()` for scanning login accounting files.

11. By default, these functions read `utmp`, but `utmpxname()` can redirect them to another file such as `wtmp`.

12. The `getutx*()` functions return pointers to static storage, so they are not reentrant.

13. `getlogin()` retrieves the login name associated with the controlling terminal by consulting `utmp`.

14. `getlogin()` may fail for daemons or processes without a terminal, because they have no controlling terminal entry.

15. `LOGNAME` is not a secure substitute for `getlogin()` because users can modify environment variables.

16. Programs implementing login services, such as `login`, `sshd`, or terminal emulators, must update `utmp`, `wtmp`, and often `lastlog`.

17. The `lastlog` file records only the most recent login for each user, unlike `wtmp`, which records all login history.

18. `lastlog` is indexed directly by UID, so the record for UID `N` is located at `N * sizeof(struct lastlog)`.

19. These files are normally writable only by privileged programs, because incorrect or malicious updates would corrupt login auditing.

20. **Core takeaway:** `utmp` tracks current login state, `wtmp` tracks login history, and `lastlog` tracks each user’s most recent login; login-capable programs must update all three consistently and securely.
