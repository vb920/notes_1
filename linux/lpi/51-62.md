### Chapter 52 - POSIX Message Queues
1. **POSIX message queues** allow processes to exchange data as discrete messages, unlike pipes/FIFOs, which are byte streams.

2. They are similar to System V message queues, but POSIX queues use **priorities**, not message types.

3. Messages are received in **descending priority order**; among equal priorities, they are received FIFO.

4. POSIX message queues are **reference counted**: unlinking removes the name, but the queue object remains until all open descriptors are closed.

5. Core APIs are:

```c
mq_open()
mq_send()
mq_receive()
mq_close()
mq_unlink()
```

6. `mq_open()` creates or opens a message queue and returns a message queue descriptor of type `mqd_t`.

7. Queue names follow POSIX IPC naming rules, typically like:

```text
/mq_name
```

8. `mq_open()` flags include `O_CREAT`, `O_EXCL`, `O_RDONLY`, `O_WRONLY`, `O_RDWR`, and `O_NONBLOCK`.

9. `O_NONBLOCK` makes `mq_send()` or `mq_receive()` fail with `EAGAIN` instead of blocking.

10. When creating a queue, the caller can specify permissions and attributes using `mode` and `struct mq_attr`.

11. Important queue attributes are:

```c
mq_maxmsg    // maximum number of messages
mq_msgsize   // maximum size of each message
mq_curmsgs   // current number of queued messages
mq_flags     // descriptor flags such as O_NONBLOCK
```

12. `mq_maxmsg` and `mq_msgsize` are fixed when the queue is created and cannot later be changed.

13. `mq_getattr()` retrieves queue attributes.

14. `mq_setattr()` can change only descriptor-level flags, mainly `O_NONBLOCK`.

15. `mq_send()` adds a message to the queue with an associated priority.

16. If the queue is full, `mq_send()` blocks unless `O_NONBLOCK` is enabled.

17. `mq_receive()` removes the oldest message with the highest priority.

18. The receive buffer must be at least as large as the queue’s `mq_msgsize`, or `mq_receive()` fails with `EMSGSIZE`.

19. If the queue is empty, `mq_receive()` blocks unless nonblocking mode is enabled.

20. Unlike pipes, an empty POSIX message queue does **not** produce EOF when no writer exists.

21. `mq_timedsend()` and `mq_timedreceive()` add absolute-time timeout support.

22. Timeout expiry causes failure with `ETIMEDOUT`.

23. A major POSIX message queue feature is **asynchronous notification** via `mq_notify()`.

24. Notification occurs when a queue transitions from **empty** to **nonempty**.

25. Only one process at a time may register for notification on a queue.

26. After one notification is delivered, the registration is automatically removed.

27. Therefore, a process must re-register with `mq_notify()` after every notification if it wants continued notifications.

28. Notification can be delivered by signal using `SIGEV_SIGNAL`.

29. Notification can also invoke a function in a new thread using `SIGEV_THREAD`.

30. Correct notification handling usually requires opening the queue in nonblocking mode and draining all messages after notification.

31. Re-register for notification **before** draining the queue to avoid missing the next empty-to-nonempty transition.

32. On Linux, POSIX message queues are implemented using a virtual filesystem and can be viewed under a mounted `mqueue` filesystem.

33. Linux message queue descriptors are implemented as file descriptors, so they can be monitored with `select()`, `poll()`, or `epoll`.

34. This `select()`/`poll()`/`epoll()` support is Linux-specific and not portable POSIX behavior.

35. Linux exposes limits through `/proc/sys/fs/mqueue`, including `msg_max`, `msgsize_max`, and `queues_max`.

36. `RLIMIT_MSGQUEUE` limits total POSIX message queue memory usage for a real user ID.

37. Compared with System V message queues, POSIX queues have cleaner APIs, reference counting, and notification support.

38. System V queues are sometimes more portable and offer more flexible message selection by type.

39. **Core takeaway:** POSIX message queues provide structured, priority-ordered message passing with clean lifecycle semantics and optional asynchronous notification, making them more modern than System V queues but somewhat less portable.


### Chapter 53 - POSIX Semaphores
1. **POSIX semaphores** are synchronization objects used by processes or threads to coordinate access to shared resources.

2. A POSIX semaphore is an integer value that the system never allows to fall below `0`.

3. If a thread/process tries to decrement a semaphore whose value is `0`, it either blocks or fails, depending on the function used.

4. POSIX supports two types of semaphores: **named semaphores** and **unnamed semaphores**.

5. A **named semaphore** has a system-wide name and can be accessed by unrelated processes using `sem_open()`.

6. An **unnamed semaphore** has no name and lives directly in shared memory or thread-shared memory.

7. Named semaphores are opened or created with:

```c
sem_open()
```

8. `sem_open()` can use flags such as `O_CREAT` and `O_EXCL`, similar to file creation.

9. When creating a named semaphore, the caller supplies permissions and an initial semaphore value.

10. Named semaphore creation and initialization are atomic, avoiding the initialization race problems of System V semaphores.

11. `sem_close()` closes a named semaphore for the calling process.

12. Closing a named semaphore does **not** remove it from the system.

13. `sem_unlink()` removes the semaphore name and marks the semaphore for deletion after all processes close it.

14. `sem_wait()` decrements a semaphore by `1`.

15. If the semaphore value is greater than `0`, `sem_wait()` succeeds immediately.

16. If the semaphore value is `0`, `sem_wait()` blocks until another process or thread increments it.

17. `sem_trywait()` is the nonblocking version of `sem_wait()` and fails with `EAGAIN` if decrement is impossible.

18. `sem_timedwait()` waits only until a specified absolute timeout and fails with `ETIMEDOUT` if the semaphore cannot be decremented.

19. `sem_post()` increments a semaphore by `1`.

20. If another process or thread is blocked in `sem_wait()`, `sem_post()` may wake one waiter.

21. POSIX semaphores do not provide strict FIFO wakeup ordering under normal time-sharing scheduling.

22. `sem_getvalue()` retrieves the current semaphore value, but the returned value may become stale immediately.

23. For unnamed semaphores, `sem_init()` initializes the semaphore.

24. The `pshared` argument of `sem_init()` decides whether the semaphore is shared between threads or between processes.

25. Thread-shared unnamed semaphores can live in global variables or heap memory.

26. Process-shared unnamed semaphores must live in shared memory, such as POSIX shared memory, System V shared memory, or an `mmap()` shared mapping.

27. `sem_destroy()` destroys an unnamed semaphore and should be called before its underlying memory is deallocated.

28. POSIX semaphores are simpler and usually faster than System V semaphores, especially when contention is low.

29. For thread synchronization, mutexes are usually preferred because mutex ownership enforces cleaner structure.

30. **Core takeaway:** POSIX semaphores provide a clean, efficient synchronization mechanism; use named semaphores for unrelated processes, unnamed semaphores for threads or shared-memory-based process synchronization.


### Chapter 54 - POSIX Shared Memory
1. **POSIX shared memory** lets unrelated processes share memory without creating an ordinary disk file.

2. It avoids the awkward key/identifier model of System V shared memory.

3. It also avoids the need for a real backing file required by ordinary shared file mappings.

4. POSIX shared memory objects are identified by names such as:

```text
/demo_shm
```

5. On Linux, these objects are usually implemented under the tmpfs filesystem mounted at:

```text
/dev/shm
```

6. POSIX shared memory has **kernel persistence**: objects remain until explicitly removed or until system shutdown.

7. To use POSIX shared memory, first create or open an object using `shm_open()`.

8. `shm_open()` returns a normal file descriptor, making the API fit naturally with UNIX file-descriptor operations.

9. Common `shm_open()` flags include `O_CREAT`, `O_EXCL`, `O_RDONLY`, `O_RDWR`, and `O_TRUNC`.

10. A newly created shared memory object initially has size `0`.

11. Therefore, after creation, programs normally call `ftruncate()` to set the object’s size.

12. The object is then mapped into memory using:

```c
mmap(..., MAP_SHARED, fd, 0)
```

13. After `mmap()`, the file descriptor may be closed without destroying the mapping.

14. Processes can use `fstat()` to determine the shared memory object’s size.

15. Processes can use `fchmod()` and `fchown()` to change permissions and ownership.

16. `shm_unlink()` removes the shared memory object’s name and prevents future opens.

17. Existing mappings remain valid after `shm_unlink()` until processes unmap them or terminate.

18. Like all shared memory techniques, POSIX shared memory needs synchronization, usually semaphores or mutexes.

19. Compared with System V shared memory, POSIX shared memory is cleaner because it uses names, descriptors, `mmap()`, `ftruncate()`, and ordinary UNIX-style operations.

20. **Core takeaway:** POSIX shared memory combines the speed of shared memory with a cleaner file-descriptor-based API, making it the preferred modern shared-memory IPC mechanism when portability allows.

### Chapter 55 - File Locking

1. **File locking** is used to synchronize multiple processes that access or update the same file.

2. Without locking, two processes may read the same old value, both update it, and overwrite each other’s changes.

3. File locks are preferable to generic semaphores when the shared resource is a file, because the kernel naturally associates locks with files.

4. Linux provides two main file-locking APIs: `flock()` and `fcntl()`.

5. `flock()` locks an entire file.

6. `fcntl()` supports **record locking**, meaning locks on byte ranges within a file.

7. File locks are normally **advisory**, meaning processes must voluntarily cooperate by checking and acquiring locks.

8. Advisory locks do not stop a noncooperating process from reading or writing the file.

9. When using stdio with file locks, buffering can cause problems because data may be read before locking or flushed after unlocking.

10. Safer approaches are to use `read()`/`write()`, flush buffers carefully, or disable stdio buffering.

11. `flock(fd, LOCK_SH)` places a shared lock.

12. `flock(fd, LOCK_EX)` places an exclusive lock.

13. `flock(fd, LOCK_UN)` unlocks the file.

14. Multiple processes may hold shared locks simultaneously.

15. Only one process may hold an exclusive lock, and it excludes both shared and exclusive locks.

16. `LOCK_NB` makes `flock()` nonblocking; if the lock cannot be acquired, it fails with `EWOULDBLOCK`.

17. `flock()` locks are associated with the **open file description**, not merely the file descriptor.

18. Duplicated descriptors created by `dup()` or inherited through `fork()` refer to the same `flock()` lock.

19. A child can release a `flock()` lock inherited from its parent.

20. `flock()` locks are normally preserved across `exec()` unless the descriptor is closed by close-on-exec.

21. A limitation of `flock()` is that it locks only whole files, reducing concurrency.

22. `fcntl()` record locks are more flexible because they can lock arbitrary byte ranges.

23. `fcntl()` supports read locks `F_RDLCK`, write locks `F_WRLCK`, and unlock operations `F_UNLCK`.

24. A read lock is like a shared lock; a write lock is like an exclusive lock.

25. To place a read lock, the file must be open for reading.

26. To place a write lock, the file must be open for writing.

27. The locked byte range is described with `struct flock`.

28. `l_whence`, `l_start`, and `l_len` define the region to lock, similar to `lseek()`.

29. If `l_len` is `0`, the lock extends from the start position to EOF, including future file growth.

30. `F_SETLK` tries to acquire or release a lock without blocking.

31. `F_SETLKW` waits until the requested lock becomes available.

32. `F_GETLK` checks whether a lock could be acquired, but the result may become stale immediately.

33. `fcntl()` lock conversion is atomic, unlike `flock()` conversion.

34. A process cannot lock itself out using `fcntl()` locks.

35. `fcntl()` locks are associated with both the process and the file inode.

36. Closing **any** descriptor for that file by the process releases all of that process’s record locks on the file.

37. Record locks are not inherited by children after `fork()`.

38. Record locks are preserved across `exec()` unless descriptors are closed.

39. The kernel can detect deadlocks for blocking `fcntl()` lock requests and fail one request with `EDEADLK`.

40. Linux can support mandatory locking, but it requires special filesystem mount options and special file permission bits.

41. Mandatory locking forces I/O operations to obey locks, but it is rarely recommended.

42. Mandatory locking can cause denial-of-service risks, performance overhead, and awkward error handling.

43. `/proc/locks` shows current system locks, including `flock()` and `fcntl()` locks.

44. `/proc/locks` can also show blocked lock requests.

45. Daemons often use a PID lock file in `/var/run` to ensure only one instance runs.

46. The daemon opens a PID file, places a write lock on it, writes its PID, and keeps the descriptor open.

47. If another instance cannot acquire the lock, it assumes the daemon is already running.

48. Older locking tricks used `open(O_CREAT | O_EXCL)`, `link()`, or permission tricks, but these are weaker than `fcntl()` locks.

49. **Core takeaway:** use `fcntl()` record locks for precise file-region synchronization; use PID lock files for single-instance daemons; avoid mandatory locking unless absolutely necessary.


### Chapter 56 - Sockets
1. **Sockets** are IPC objects that allow communication between applications on the same host or across a network.

2. A socket is created with:

```c
socket(domain, type, protocol)
```

3. The returned socket is represented by a normal file descriptor.

4. The **domain** defines the address format and communication scope.

5. `AF_UNIX` is used for communication between processes on the same machine.

6. `AF_INET` is used for IPv4 network communication.

7. `AF_INET6` is used for IPv6 network communication.

8. The two main socket types are `SOCK_STREAM` and `SOCK_DGRAM`.

9. `SOCK_STREAM` provides reliable, bidirectional, byte-stream communication.

10. Stream sockets preserve byte order but do **not** preserve message boundaries.

11. Internet stream sockets normally use TCP.

12. `SOCK_DGRAM` provides message-oriented datagram communication.

13. Datagram sockets preserve message boundaries, but delivery is unreliable.

14. Datagrams may be lost, duplicated, or received out of order.

15. Internet datagram sockets normally use UDP.

16. Servers usually call `bind()` to attach their socket to a well-known address.

17. For stream sockets, a server calls `listen()` to mark the socket as passive/listening.

18. The `backlog` argument to `listen()` limits the queue of pending connection requests.

19. A client uses `connect()` to initiate a connection to the server’s listening socket.

20. The server uses `accept()` to accept a pending connection.

21. `accept()` returns a **new connected socket**, while the original listening socket remains open for future clients.

22. After a stream connection is established, both sides can use `read()`/`write()` or socket-specific calls like `send()`/`recv()`.

23. Closing one side of a stream socket causes the peer to see EOF on read after buffered data is consumed.

24. Writing to a closed stream socket normally generates `SIGPIPE` and fails with `EPIPE`.

25. Datagram sockets normally use `sendto()` and `recvfrom()` because each message may specify or reveal a peer address.

26. `recvfrom()` reads exactly one datagram; if the buffer is too small, the datagram may be truncated.

27. `connect()` can also be used on datagram sockets to fix a default peer address.

28. A connected datagram socket can use `write()`/`send()` and receives only datagrams from its connected peer.

29. Socket addresses are passed through generic `struct sockaddr *`, but each domain has its own real address structure.

30. **Core takeaway:** stream sockets model reliable connected byte streams, while datagram sockets model independent unreliable messages; `bind()`, `listen()`, `connect()`, `accept()`, `sendto()`, and `recvfrom()` are the core building blocks.

### Chapter 57 - Sockets - UNiX DOMAIN
1. **UNIX domain sockets** allow IPC between processes on the same host.

2. They use the socket domain `AF_UNIX`.

3. Unlike Internet sockets, UNIX domain sockets use **pathnames** as socket addresses.

4. The address structure is:

```c
struct sockaddr_un {
    sa_family_t sun_family;   // AF_UNIX
    char sun_path[108];       // socket pathname
};
```

5. `sun_path` size is not fully portable, so programs should avoid long socket pathnames.

6. Before filling `sockaddr_un`, it is good practice to zero the structure with `memset()`.

7. Binding a UNIX domain socket creates a socket entry in the filesystem.

8. Such socket files appear in `ls -l` output with file type `s`.

9. A UNIX domain socket pathname cannot already exist when `bind()` is called.

10. Therefore, servers often call `unlink()` or `remove()` before `bind()`.

11. After the server exits, the socket pathname usually remains and should be removed.

12. Real programs should avoid creating UNIX socket files in `/tmp` unless security is carefully handled.

13. A safer design is to place socket files in a protected directory with controlled permissions.

14. UNIX domain **stream sockets** use `SOCK_STREAM`.

15. They provide reliable, bidirectional, byte-stream communication between local processes.

16. A stream server creates a socket, binds it to a pathname, calls `listen()`, and then accepts clients with `accept()`.

17. `accept()` returns a new connected socket for client communication.

18. The original listening socket remains open for accepting more clients.

19. A stream client creates a socket and calls `connect()` using the server’s pathname.

20. After connection, both sides can use `read()`/`write()` just like with other file descriptors.

21. UNIX domain **datagram sockets** use `SOCK_DGRAM`.

22. Unlike Internet datagrams, UNIX domain datagrams are reliable inside the kernel.

23. UNIX domain datagram messages are delivered in order and without duplication.

24. Datagram servers usually bind to a well-known pathname.

25. Datagram clients may also need to bind to a unique pathname so the server can send replies.

26. A common client technique is to include the client PID in the socket pathname.

27. `sendto()` sends a datagram to a specified UNIX socket pathname.

28. `recvfrom()` receives one datagram and can also return the sender’s address.

29. If the receive buffer is smaller than the datagram, the datagram may be truncated.

30. UNIX domain socket access can be controlled using filesystem permissions.

31. To connect to a UNIX stream socket or send to a UNIX datagram socket, write permission on the socket file is normally required.

32. Execute/search permission is also required on every directory component in the socket path.

33. Some systems ignore socket file permissions, so directory permissions are the more portable protection mechanism.

34. `socketpair()` creates two already-connected UNIX domain sockets.

35. `socketpair(AF_UNIX, SOCK_STREAM, 0, fd)` is like a bidirectional pipe.

36. It is useful between parent and child after `fork()`.

37. `socketpair()` avoids filesystem socket names, reducing cleanup and security problems.

38. Linux also provides an **abstract UNIX socket namespace**.

39. Abstract sockets start with a null byte in `sun_path` and do not create filesystem entries.

40. **Core takeaway:** UNIX domain sockets are local IPC sockets; use stream sockets for reliable byte streams, datagram sockets for local message passing, `socketpair()` for related processes, and protected directories or abstract sockets for safer naming.

### CHapter 58 - Sockets - TCP /IP Networks
1. An **internet** is a network of networks; the global **Internet** is the worldwide TCP/IP internet.

2. A **subnet** is one network inside an internet.

3. A **router** connects multiple networks and forwards packets between them.

4. A **multihomed host** has multiple network interfaces and therefore multiple network addresses.

5. TCP/IP is organized as layered protocols: application layer, transport layer, network layer, and data-link layer.

6. The socket API hides most lower-layer complexity from applications.

7. **Encapsulation** means each lower layer wraps higher-layer data inside its own packet format and header.

8. The **data-link layer** moves frames across a physical network such as Ethernet.

9. Each data-link layer has an **MTU**, the maximum frame size it can carry.

10. The **network layer** is mainly IP, which routes packets from source host to destination host.

11. IP transmits independent packets called **datagrams**.

12. IP is **connectionless**: it does not establish a virtual circuit between hosts.

13. IP is **unreliable**: packets may be lost, duplicated, reordered, or discarded.

14. IPv4 uses **32-bit addresses**, usually written in dotted-decimal form, such as `192.168.1.10`.

15. IPv6 uses **128-bit addresses**, usually written as colon-separated hexadecimal groups.

16. `127.0.0.1` is the IPv4 loopback address; IPv6 loopback is `::1`.

17. `INADDR_ANY` lets a server bind to all local IPv4 interfaces on a multihomed host.

18. IP may fragment packets if a datagram is larger than the path MTU.

19. Fragmentation is undesirable because losing one fragment makes the whole datagram unusable.

20. The transport layer mainly provides **UDP** and **TCP**.

21. UDP adds port numbers and checksums to IP, but remains connectionless and unreliable.

22. UDP applications should keep datagrams small, often around 512 bytes, to avoid IP fragmentation.

23. TCP provides reliable, bidirectional, connection-oriented byte-stream communication.

24. TCP achieves reliability using sequence numbers, acknowledgements, retransmissions, checksums, flow control, and congestion control.

25. **Core takeaway:** IP provides best-effort packet delivery between hosts; UDP exposes that lightweight unreliable service to applications, while TCP builds a reliable ordered byte stream on top of IP.

### Chapter 59 - Sockets -
1. **Internet domain sockets** use IP addresses plus port numbers as socket addresses.

2. `AF_INET` represents IPv4 sockets, while `AF_INET6` represents IPv6 sockets.

3. Internet stream sockets use **TCP** and provide reliable, bidirectional byte-stream communication.

4. Internet datagram sockets use **UDP** and provide message-oriented but unreliable communication.

5. UDP datagrams may be lost, duplicated, reordered, or silently dropped if receiver buffers overflow.

6. Network programs must handle **byte order** differences between architectures.

7. Network byte order is **big-endian**.

8. Use `htons()` and `htonl()` to convert host-order integers to network byte order.

9. Use `ntohs()` and `ntohl()` to convert network-order integers back to host byte order.

10. Port numbers are 16-bit values and must be stored in network byte order inside socket address structures.

11. IPv4 addresses are stored in `struct sockaddr_in`.

12. `sockaddr_in` contains `sin_family`, `sin_port`, and `sin_addr`.

13. `sin_family` is set to `AF_INET`.

14. IPv6 addresses are stored in `struct sockaddr_in6`.

15. `sockaddr_in6` contains `sin6_family`, `sin6_port`, `sin6_addr`, and extra IPv6-specific fields.

16. `sin6_family` is set to `AF_INET6`.

17. `sockaddr_storage` is a generic structure large enough to hold any socket address type.

18. It is useful for IP-version-independent code.

19. `inet_pton()` converts printable IPv4/IPv6 addresses into binary network form.

20. `inet_ntop()` converts binary IPv4/IPv6 addresses into printable strings.

21. `INET_ADDRSTRLEN` and `INET6_ADDRSTRLEN` help size address-string buffers safely.

22. For structured network data, raw C structs should not be sent directly across heterogeneous machines.

23. Different machines may use different integer sizes, byte order, alignment, and padding.

24. Portable applications should use marshalling formats or simple text protocols.

25. Newline-delimited text protocols are easy to debug with tools like `telnet`.

26. DNS maps hostnames to IP addresses and IP addresses back to hostnames.

27. DNS is hierarchical, distributed, and avoids centralized management of all Internet names.

28. A fully qualified domain name identifies a host within the DNS hierarchy.

29. `/etc/services` maps service names such as `http` or `ssh` to port numbers and protocols.

30. `getaddrinfo()` is the modern function for converting host/service names into socket address structures.

31. `getaddrinfo()` replaces older functions like `gethostbyname()` and `getservbyname()`.

32. It supports both IPv4 and IPv6 and enables protocol-independent programs.

33. `getaddrinfo()` returns a linked list because a host/service may map to multiple possible addresses.

34. Programs usually loop through the returned list until `socket()`, `bind()`, or `connect()` succeeds.

35. `freeaddrinfo()` releases the dynamically allocated result list.

36. `gai_strerror()` converts `getaddrinfo()` or `getnameinfo()` error codes into readable strings.

37. `getnameinfo()` performs the reverse conversion: socket address to hostname/service name.

38. `AI_PASSIVE` tells `getaddrinfo()` to return wildcard addresses suitable for server sockets.

39. `AI_NUMERICHOST` and `AI_NUMERICSERV` avoid name resolution and force numeric interpretation.

40. TCP servers usually create a socket, set options like `SO_REUSEADDR`, bind, listen, and accept clients.

41. TCP clients usually call `getaddrinfo()`, create a socket, and connect to one returned address.

42. If a client does not bind explicitly, the kernel assigns an ephemeral port automatically.

43. TCP protocols often need application-level message framing because TCP is a byte stream.

44. A common framing method is newline-terminated text messages read using a helper like `readLine()`.

45. Obsolete APIs include `inet_aton()`, `inet_ntoa()`, `gethostbyname()`, `gethostbyaddr()`, `getservbyname()`, and `getservbyport()`.

46. New code should prefer `inet_pton()`, `inet_ntop()`, `getaddrinfo()`, and `getnameinfo()`.

47. UNIX domain sockets are often faster and support filesystem-based access control.

48. Internet domain sockets are necessary for communication across hosts and are more general for networked applications.

49. **Core takeaway:** Internet socket programming combines socket lifecycle calls with byte-order conversion, address structures, DNS/service resolution, and protocol-aware design for TCP or UDP communication.

### Chapter - 60 - Server Design
1. Servers are commonly designed as either **iterative** or **concurrent**.

2. An **iterative server** handles one client completely before accepting the next client.

3. Iterative servers are suitable when each request is short and quick, such as a single request-response exchange.

4. A **concurrent server** handles multiple clients at the same time.

5. Concurrent servers are needed when client handling is slow, long-lived, or involves extended conversations.

6. The traditional concurrent TCP server creates one child process per client using `fork()`.

7. The parent server keeps the listening socket open and repeatedly calls `accept()`.

8. For each accepted connection, the parent forks a child to handle that client.

9. The child closes the unused listening socket, services the connected socket, and then exits.

10. The parent closes its copy of the connected socket and continues accepting new connections.

11. A concurrent server must reap dead children, usually with a `SIGCHLD` handler and `waitpid(..., WNOHANG)`.

12. TCP echo service benefits from concurrency because a client may keep the connection open and send arbitrary amounts of data.

13. UDP echo service can often be iterative because each datagram is handled independently and quickly.

14. High-load servers may avoid per-client `fork()` overhead using **preforked** or **prethreaded** server pools.

15. Another scalable design is a single-process event-driven server using `select()`, `poll()`, or `epoll`.

16. Server farms and load balancers distribute high client load across multiple machines.

17. `inetd` is an Internet superserver that listens on many service ports and starts server programs on demand.

18. `inetd` simplifies server code by doing `socket()`, `bind()`, `listen()`, `accept()`, `fork()`, descriptor setup, and `exec()`.

19. Servers started by `inetd` commonly use standard input/output/error as the connected socket.

20. **Core takeaway:** use iterative servers for simple quick requests, concurrent or event-driven designs for long/high-load clients, and `inetd` for infrequent lightweight network services.


### Chapter 61 - Advanced Sockets
1. Stream sockets can produce **partial reads** and **partial writes**, so one `read()` or `write()` may transfer fewer bytes than requested.

2. A partial read occurs when fewer bytes are currently available in the socket receive buffer.

3. A partial write may occur because of signal interruption, nonblocking mode, or asynchronous socket errors.

4. Robust stream-socket code must loop until the desired number of bytes has been transferred.

5. Helper functions like `readn()` and `writen()` repeatedly call `read()`/`write()` to complete fixed-size transfers.

6. TCP is a byte stream, so even if the sender writes 100 bytes once, the receiver may receive them across multiple reads.

7. Application protocols over TCP must define their own framing, such as fixed-size records, length prefixes, or newline-delimited messages.

8. `close()` closes both directions of a socket connection.

9. `shutdown()` gives finer control by closing only one or both halves of the connection.

10. `shutdown(sockfd, SHUT_WR)` performs a **half-close**: the local side stops writing but can still read.

11. `SHUT_WR` lets the peer observe EOF after reading all outstanding data.

12. Half-close is useful when a client wants to say “I am done sending” but still wants to receive the server’s reply.

13. `shutdown()` affects the underlying open file description, not just one file descriptor.

14. Therefore, duplicated descriptors or descriptors inherited across `fork()` are affected by `shutdown()`.

15. `shutdown()` does not close the file descriptor itself; `close()` is still needed.

16. `recv()` and `send()` are socket-specific alternatives to `read()` and `write()`.

17. They take an extra `flags` argument for per-call socket behavior.

18. `MSG_DONTWAIT` performs nonblocking I/O for a single `recv()` or `send()` call.

19. `MSG_PEEK` reads data without removing it from the socket receive buffer.

20. `MSG_WAITALL` asks `recv()` to wait until the requested number of bytes is available, though short reads can still occur.

21. `MSG_NOSIGNAL` prevents `SIGPIPE` on send failure and makes `send()` return `EPIPE` instead.

22. `MSG_MORE` tells Linux to hold output briefly because more data is coming.

23. `sendfile()` efficiently transfers file contents directly to a socket.

24. It avoids copying file data from kernel space to user space and then back to kernel space.

25. This is useful for web servers and file servers sending large files unchanged.

26. `sendfile()` is often called a **zero-copy** transfer mechanism.

27. `TCP_CORK` can combine small headers and file payload into fewer TCP segments.

28. This is useful when sending something like HTTP headers followed by file data.

29. `getsockname()` returns the local address of a socket.

30. It is useful for discovering a kernel-assigned ephemeral port.

31. `getpeername()` returns the address of the connected peer socket.

32. It is useful when a process inherited a connected socket, such as from `inetd`.

33. TCP segments contain fields such as source port, destination port, sequence number, acknowledgement number, flags, window size, checksum, and data.

34. TCP uses sequence numbers to order bytes and detect duplicates.

35. TCP acknowledgements indicate the next byte expected from the peer.

36. TCP connection setup uses the **three-way handshake**: `SYN`, `SYN+ACK`, `ACK`.

37. TCP connection termination usually uses four segments: `FIN`, `ACK`, `FIN`, `ACK`.

38. The side that first closes performs the **active close**.

39. The active closer normally enters the `TIME_WAIT` state.

40. `TIME_WAIT` ensures reliable connection termination if the final ACK is lost.

41. `TIME_WAIT` also prevents old duplicate TCP segments from being accepted by a new connection incarnation.

42. On Linux, `TIME_WAIT` commonly lasts about 60 seconds.

43. Do not disable or bypass `TIME_WAIT` casually; it protects TCP correctness.

44. `SO_REUSEADDR` helps servers restart while old TCP endpoints still exist.

45. Most TCP servers should set `SO_REUSEADDR` before `bind()`.

46. `SO_REUSEADDR` avoids common `EADDRINUSE` failures after server restart.

47. `netstat` displays current socket state, listening sockets, established connections, queues, and TCP states.

48. `tcpdump` monitors network packets and is useful for seeing SYNs, ACKs, FINs, retransmissions, and data flow.

49. `getsockopt()` and `setsockopt()` retrieve and modify socket options.

50. Socket options may exist at different protocol levels, such as `SOL_SOCKET`, `IPPROTO_TCP`, or `IPPROTO_IP`.

51. `SO_TYPE` is a read-only option that reveals whether a socket is stream, datagram, etc.

52. Some socket options are inherited by sockets returned from `accept()`, but inheritance rules vary across systems.

53. For portability, explicitly reset important flags such as `O_NONBLOCK` or `FD_CLOEXEC` on accepted sockets.

54. TCP is reliable, ordered, connection-oriented, and byte-stream based.

55. UDP is connectionless, message-oriented, and unreliable.

56. UDP has lower overhead and can be better for simple request-response protocols, broadcasting, multicasting, and real-time media.

57. If UDP applications need reliability, they must implement sequencing, acknowledgements, retransmission, and duplicate detection.

58. If flow control and congestion control are needed, TCP is usually the better choice.

59. Out-of-band data lets stream sockets mark urgent data, but it is limited and generally discouraged.

60. `sendmsg()` and `recvmsg()` are the most general socket I/O calls.

61. They support scatter-gather I/O and ancillary data.

62. Ancillary data over UNIX domain sockets can pass file descriptors between processes.

63. File descriptor passing really passes a reference to the same open file description.

64. UNIX domain sockets can also transmit sender credentials for local authentication.

65. `SOCK_SEQPACKET` sockets combine connection orientation, reliable delivery, and preserved message boundaries.

66. SCTP is a reliable message-preserving transport protocol and can support multistreaming.

67. DCCP provides unreliable datagram-style delivery with congestion control.

68. **Core takeaway:** advanced socket programming requires handling partial I/O, understanding TCP lifecycle states, using `shutdown()` and socket options correctly, and choosing TCP, UDP, or specialized mechanisms based on reliability, framing, latency, and scalability needs.

### CHapter 62 - Terminals
1. Terminals historically were physical devices connected via serial lines; today they are mostly **terminal emulators** such as `xterm`.

2. The abbreviation **tty** comes from “teletype” and is still used for terminal devices.

3. Terminal devices are represented as character devices, such as `/dev/tty1` or pseudoterminals under `/dev/pts`.

4. Each terminal has a **terminal driver** that manages input, output, echoing, special characters, and terminal modes.

5. Terminal input can operate in **canonical mode** or **noncanonical mode**.

6. In **canonical mode**, input is line-oriented and line editing is enabled.

7. A `read()` from a canonical terminal normally returns only after a complete line is entered.

8. In canonical mode, characters such as erase, kill-line, EOF, and word-erase are interpreted by the terminal driver.

9. In **noncanonical mode**, input is available character-by-character rather than line-by-line.

10. Programs like `vi`, `less`, and `more` use noncanonical mode to respond immediately to keystrokes.

11. The terminal driver maintains an **input queue** and an **output queue**.

12. If echoing is enabled, typed input characters are automatically copied to the output queue and displayed.

13. Terminal attributes are retrieved with `tcgetattr()`.

14. Terminal attributes are changed with `tcsetattr()`.

15. Terminal settings are stored in `struct termios`.

16. `c_iflag` controls input processing.

17. `c_oflag` controls output processing.

18. `c_cflag` controls hardware/line settings.

19. `c_lflag` controls local terminal behavior such as echoing, canonical mode, and signal characters.

20. The `c_cc` array stores terminal special characters such as `VINTR`, `VEOF`, `VERASE`, and `VSUSP`.

21. `tcsetattr()` can apply changes immediately with `TCSANOW`.

22. `TCSADRAIN` applies changes after pending output is transmitted.

23. `TCSAFLUSH` applies changes after output drains and also discards pending input.

24. The `stty` command is the shell-level tool for viewing and changing terminal attributes.

25. `stty -a` displays current terminal flags, special characters, line speed, rows, and columns.

26. `stty sane` is useful for restoring a damaged terminal after a buggy program leaves it in raw or no-echo mode.

27. Special characters include `INTR` usually `Control-C`, `EOF` usually `Control-D`, `SUSP` usually `Control-Z`, and `QUIT` usually `Control-\`.

28. `INTR`, `QUIT`, and `SUSP` can generate signals for the foreground process group if `ISIG` is enabled.

29. `ECHO` controls whether typed characters are displayed.

30. `ICANON` enables canonical line-based input.

31. `IEXTEN` enables extended input processing such as word erase and literal-next.

32. In noncanonical mode, `VMIN` and `VTIME` control when `read()` returns.

33. `VMIN = 1` and `VTIME = 0` gives blocking single-character input.

34. **Raw mode** disables canonical processing, signal characters, echoing, and most input/output transformations.

35. **Cbreak mode** gives character-at-a-time input but usually still allows signal-generating characters.

36. Programs that enter raw or cbreak mode must restore the original terminal settings before exit.

37. `SIGWINCH` is sent when terminal window size changes.

38. `ioctl(TIOCGWINSZ)` retrieves terminal rows and columns.

39. `isatty()` checks whether a file descriptor refers to a terminal, and `ttyname()` returns the terminal device name.

40. **Core takeaway:** terminal programming is about controlling the terminal driver—input mode, echoing, special characters, signals, window size, and safe restoration of terminal state.
