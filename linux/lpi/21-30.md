### Chapter 21 — Signal Handlers (Deep Kernel Notes)

1. A signal handler executes as an **asynchronous kernel-triggered control path** that can interrupt user code at almost any instruction, making correctness the primary design concern.

2. The preferred handler design is to **set a flag of type `volatile sig_atomic_t` and return**, deferring all substantial work to normal execution flow.

3. Linux blocks delivery of the currently executing signal while its handler runs (unless `SA_NODEFER` is used).

4. Traditional UNIX signals are **pending-state based, not event-count based**; the kernel records only whether a signal is pending.

5. If the same signal arrives multiple times while blocked, only a single pending instance is retained.

6. Therefore, standard signals cannot be used as reliable event counters; delivered signals may represent multiple underlying events.

7. A signal handler effectively creates a second execution context inside the process, similar to a lightweight asynchronous thread.

8. Reentrancy is critical because a handler may interrupt code that is already executing the same function.

9. A function is reentrant only if concurrent/interrupted executions cannot corrupt shared state.

10. Functions depending solely on stack-local variables are naturally reentrant.

11. Functions manipulating global variables, static storage, or shared internal structures are generally nonreentrant.

12. Heap allocators (`malloc`, `free`) are unsafe because interruption during allocator metadata updates can corrupt heap structures.

13. Functions returning pointers to static buffers (e.g., `crypt()`) become unsafe because handler execution can overwrite main-program data.

14. `stdio` functions (`printf`, `scanf`, etc.) are particularly dangerous because they modify shared buffered-I/O state.

15. The classic failure mode is: main code enters a nonreentrant function, a signal arrives, the handler re-enters the same function, and shared state becomes inconsistent.

16. POSIX addresses this through **async-signal-safe functions**, which are guaranteed safe inside handlers.

17. Async-signal-safe functions are either reentrant or implemented in a manner that prevents unsafe interruption effects.

18. The practical kernel rule is simple: **signal handlers should call only async-signal-safe functions**.

19. Even async-signal-safe calls may modify `errno`; handlers should save and restore `errno` around their work.

20. Shared handler flags should use `volatile sig_atomic_t` because reads and writes are guaranteed atomic with respect to signal delivery.

21. The guarantee applies only to atomic load/store operations, not compound operations such as `++`, `--`, or read-modify-write sequences.

22. A handler need not return normally; it may terminate execution via `_exit()`, `abort()`, `kill()`, or a nonlocal jump.

23. `exit()` is unsafe inside handlers because it performs complex library cleanup and stdio flushing.

24. `sigsetjmp()` and `siglongjmp()` provide controlled recovery paths from asynchronous signal delivery.

25. Linux follows System V semantics: ordinary `longjmp()` does **not restore the signal mask**.

26. Failure to restore the signal mask may leave the triggering signal permanently blocked after handler exit.

27. `sigsetjmp(..., 1)` captures the current signal mask, and `siglongjmp()` restores it during recovery.

28. Nonlocal jumps are powerful but dangerous because they can bypass partially completed critical updates and leave data structures inconsistent.

29. Signals may arrive before a jump target is initialized; robust programs therefore use guard variables before performing `siglongjmp()`.

30. `abort()` works by generating `SIGABRT`, usually producing both process termination and a core dump for postmortem analysis.

31. By default, the kernel builds a signal frame on the process's current user stack when invoking a handler.

32. Stack exhaustion creates a special problem: a `SIGSEGV` handler cannot run if the kernel cannot allocate its stack frame.

33. `sigaltstack()` solves this by registering a separate user-space stack dedicated to signal handling.

34. `SA_ONSTACK` instructs the kernel to switch execution to the alternate signal stack before entering the handler.

35. Alternate stacks are primarily intended for handling catastrophic failures such as stack-overflow-driven `SIGSEGV`.

36. The kernel never resizes the alternate stack; overflow of that region results in undefined memory corruption.

37. `SA_SIGINFO` upgrades signal delivery from "signal number only" to a richer kernel-to-user information transfer mechanism.

38. With `SA_SIGINFO`, the handler receives a `siginfo_t` structure describing the exact source of the signal.

39. `siginfo_t` can reveal sender PID, sender UID, queued data, child status changes, timer expirations, and hardware fault information.

40. For hardware exceptions (`SIGSEGV`, `SIGBUS`, `SIGILL`, `SIGFPE`), the kernel supplies the faulting address through `si_addr`.

41. The `si_code` field identifies the precise origin of delivery—kernel-generated, `kill()`, `sigqueue()`, timer expiration, child event, or hardware fault.

42. `ucontext_t` provides a snapshot of interrupted execution state, including saved registers, stack pointer, program counter, and signal mask.

43. This context is effectively the kernel's saved CPU state for the thread at the moment of signal delivery.

44. Blocking system calls can be interrupted when a signal handler executes.

45. By default, such interruptions often cause the syscall to fail with `EINTR`.

46. `EINTR` is not a true failure; it indicates that kernel execution was interrupted by signal processing.

47. Applications commonly restart interrupted syscalls using retry loops around operations such as `read()`.

48. `SA_RESTART` asks the kernel to transparently restart many interrupted blocking syscalls after handler completion.

49. Restart behavior is syscall-dependent; some kernel interfaces intentionally never restart automatically.

50. Multiplexing and waiting APIs (`select`, `poll`, `epoll_wait`, `pselect`, `ppoll`, `pause`, `sigsuspend`, `nanosleep`) frequently require explicit `EINTR` handling.

***

### Core Kernel Insight

**A signal is fundamentally a kernel-mediated asynchronous control transfer into user space. The entire chapter revolves around preserving process correctness despite arbitrary interruption—through reentrancy, async-signal-safety, atomic communication (`sig_atomic_t`), proper signal-mask restoration (`sigsetjmp/siglongjmp`), alternate stacks (`sigaltstack`), rich delivery metadata (`SA_SIGINFO`), and robust handling of interrupted system calls (`EINTR`/`SA_RESTART`).**

### Chapter 22 — Advanced Signals (Kernel-Focused Notes)

1. Core dumps are kernel-generated snapshots of a process's address space, registers, and execution state at the moment of fatal signal delivery.

2. Signals such as `SIGQUIT`, `SIGSEGV`, `SIGABRT`, and `SIGILL` can trigger core dump creation before process termination.

3. Core dump generation is controlled by multiple kernel checks including permissions, filesystem state, resource limits (`RLIMIT_CORE`, `RLIMIT_FSIZE`), and dumpability flags.

4. Set-UID and Set-GID programs normally do not generate core files to prevent disclosure of privileged process memory.

5. Linux exposes core dump configuration through `/proc/sys/kernel/core_pattern`, allowing custom filenames, directories, or piping dumps to helper programs.

6. `SIGKILL` and `SIGSTOP` are special kernel-enforced signals whose dispositions cannot be changed, ignored, caught, or blocked.

7. These signals guarantee that administrators always retain the ability to stop or terminate runaway processes.

8. `SIGCONT` has unique semantics: a stopped task resumes execution even if `SIGCONT` is blocked or ignored.

9. Delivery of `SIGCONT` automatically discards pending stop signals; delivery of a stop signal discards pending `SIGCONT`.

10. Signal delivery is ultimately constrained by the kernel scheduler and process execution state.

11. Linux distinguishes between `TASK_INTERRUPTIBLE` and `TASK_UNINTERRUPTIBLE` sleep states.

12. A task in `TASK_INTERRUPTIBLE` can be awakened by signal delivery, causing blocking operations to terminate or restart.

13. A task in `TASK_UNINTERRUPTIBLE` ignores signals until the awaited kernel event completes.

14. Consequently, even `SIGKILL` may appear ineffective while a process remains stuck in uninterruptible sleep (`D` state).

15. Linux introduced `TASK_KILLABLE` to allow fatal signals to terminate certain previously unkillable waits (e.g., NFS operations).

16. Hardware-generated signals (`SIGSEGV`, `SIGBUS`, `SIGILL`, `SIGFPE`) originate from CPU exceptions rather than user-space requests.

17. These signals are fundamentally synchronous because they arise from execution of a specific machine instruction.

18. Returning normally from a handler for a hardware fault usually re-executes the faulting instruction, resulting in repeated signal generation.

19. Linux therefore expects handlers for hardware faults to terminate the process or escape via `siglongjmp()`.

20. Ignoring hardware-generated faults is meaningless; Linux forces delivery even when applications attempt to ignore them.

21. Signal generation is classified as either **synchronous** or **asynchronous**.

22. Synchronous signals originate directly from the executing thread (CPU fault, `raise()`, self-`kill()`), and are delivered immediately.

23. Asynchronous signals originate externally (another process, terminal driver, timer expiration, child state changes).

24. Asynchronous signals become pending and are delivered when the kernel next transitions the target task from kernel mode to user mode.

25. Standard signal delivery order is implementation-defined; Linux currently delivers pending standard signals in ascending signal-number order.

26. Multiple pending handlers may nest because handler execution itself can trigger kernel-user transitions that allow delivery of additional signals.

27. Historical UNIX signal implementations were unreliable: handlers reset automatically, signals were not masked during handler execution, and syscalls were not restarted.

28. Modern Linux implements reliable signals through `sigaction()`, which should always be preferred over `signal()` for portable programs.

29. Realtime signals were added to address major deficiencies of traditional signals.

30. Unlike standard signals, realtime signals are **queued**, meaning every generated occurrence is preserved by the kernel.

31. Realtime signals can carry additional user-defined payload data through `sigqueue()`.

32. The sender supplies data using a `sigval` union (`sival_int` or `sival_ptr`).

33. The kernel delivers this payload to the receiver through the `siginfo_t.si_value` field.

34. Realtime signals obey strict delivery ordering: lower-numbered realtime signals have higher delivery priority.

35. Multiple queued instances of the same realtime signal are delivered FIFO in the order generated.

36. Linux reserves some realtime signals internally for thread-library implementation, making `SIGRTMIN` runtime-dependent.

37. Queued realtime signals consume kernel memory and are limited by `RLIMIT_SIGPENDING`.

38. `sigsuspend()` provides an atomic operation that replaces the signal mask and immediately suspends execution.

39. Its main purpose is eliminating the classic race condition between `sigprocmask()` and `pause()`.

40. Internally, `sigsuspend()` behaves like an atomic **unblock-and-sleep** operation that cannot miss a signal arriving between those two actions.

41. `sigwaitinfo()` provides synchronous signal acceptance without requiring asynchronous handlers.

42. The calling thread sleeps until one signal from a specified set becomes pending.

43. The accepted signal is removed from the pending set and detailed metadata is returned via `siginfo_t`.

44. `sigwaitinfo()` is generally cleaner and easier than managing asynchronous handlers for control-flow-type signals.

45. `sigtimedwait()` extends this model by allowing bounded waiting with timeout support.

46. Linux further extends synchronous signal handling through `signalfd()`.

47. `signalfd()` converts pending signals into readable file-descriptor events.

48. Signals can then be processed using ordinary `read()` operations and integrated with `select()`, `poll()`, or `epoll()` event loops.

49. Reading from a `signalfd` consumes the signal, removing it from the pending set.

50. Although signals provide kernel-mediated interprocess notifications, they are a low-bandwidth IPC mechanism because they carry very little data and introduce synchronization complexity.

***

### Core Kernel Insight

**Chapter 22 shifts the perspective from basic signal handling to the kernel's signal-delivery machinery: task sleep states control when signals can act, hardware exceptions generate synchronous signals tied to CPU execution, realtime signals transform signals into a queued event system with payloads, and interfaces like `sigsuspend()`, `sigwaitinfo()`, and `signalfd()` provide increasingly sophisticated ways to synchronize user space with the kernel's pending-signal infrastructure.**

### Chapter 23 — Timers and Sleeping (Kernel-Focused Notes)

1. Timers are kernel-managed mechanisms that schedule future notifications, while sleep APIs suspend execution until a specified time interval elapses.

2. Classical UNIX interval timers (`setitimer`) are implemented as per-process kernel timers that generate signals upon expiration.

3. Linux provides three interval timer classes, each tied to a different kernel time accounting source.

4. `ITIMER_REAL` counts wall-clock (real) time and generates `SIGALRM`.

5. `ITIMER_VIRTUAL` counts only user-mode CPU execution time and generates `SIGVTALRM`.

6. `ITIMER_PROF` counts total CPU consumption (user + kernel time) and generates `SIGPROF`.

7. A process owns only one timer instance of each type; setting a new timer replaces the existing one.

8. Interval timers contain two values: an initial expiration (`it_value`) and a reload interval (`it_interval`).

9. When a periodic timer expires, the kernel automatically reloads it from `it_interval` and continues counting.

10. Timer expiration is asynchronous and delivered through the signal subsystem.

11. Since timer signals are ordinary signals, standard-signal limitations apply (e.g., non-queued delivery and potential signal loss).

12. `getitimer()` allows inspection of the current countdown value maintained by the kernel.

13. Interval timers survive `exec()` but are not inherited across `fork()`.

14. `alarm()` is a simplified wrapper around the real-time interval timer mechanism.

15. `alarm()` provides only one-shot wall-clock timers and always delivers `SIGALRM`.

16. On Linux, `alarm()` and `setitimer(ITIMER_REAL)` share the same underlying kernel timer.

17. Timer expiration time and actual process wakeup are distinct events; scheduler latency may delay execution after expiry.

18. Linux maintains timer periodicity even under scheduling delays, preventing cumulative drift ("creeping error").

19. Traditional timer accuracy was limited by jiffy granularity and software clock frequency.

20. High-resolution timers (hrtimers) remove this limitation and allow microsecond or better precision.

21. A common timer application is implementing timeouts on blocking system calls.

22. A timeout pattern uses `SIGALRM` to interrupt a blocked syscall, causing it to fail with `EINTR`.

23. Such designs depend on disabling `SA_RESTART`, otherwise the kernel transparently restarts interrupted syscalls.

24. `sleep()` suspends execution for a fixed duration and returns unslept time when interrupted.

25. Historically, some implementations built `sleep()` on top of `alarm()`, creating portability concerns.

26. `nanosleep()` improves upon `sleep()` by providing nanosecond-resolution interval specification.

27. Unlike `sleep()`, POSIX explicitly requires that `nanosleep()` not rely on signal-generated timers.

28. If interrupted, `nanosleep()` returns `EINTR` and supplies the remaining unslept interval.

29. Repeated restart of `nanosleep()` can accumulate rounding errors under heavy signal load.

30. `clock_nanosleep()` solves this by supporting absolute-time sleeps through `TIMER_ABSTIME`.

31. Absolute sleeping eliminates oversleep drift because the target wakeup time never changes.

32. POSIX clocks provide nanosecond-resolution kernel time sources accessible through `clock_gettime()`.

33. `CLOCK_REALTIME` represents the system wall clock and can be modified by privileged users.

34. `CLOCK_MONOTONIC` measures continuously increasing time and is immune to system clock adjustments.

35. `CLOCK_PROCESS_CPUTIME_ID` measures CPU time consumed by an entire process.

36. `CLOCK_THREAD_CPUTIME_ID` measures CPU consumption of an individual thread.

37. `clock_getres()` reveals the effective resolution of a clock and indirectly indicates hrtimer availability.

38. `clock_settime()` can modify only settable clocks such as `CLOCK_REALTIME`.

39. POSIX timers were introduced to overcome limitations of classical interval timers.

40. Unlike `setitimer()`, POSIX timers allow creation of multiple independent timer instances per process.

41. POSIX timers support arbitrary clock sources rather than being restricted to three predefined timer classes.

42. `timer_create()` allocates a kernel timer object and defines its notification mechanism.

43. `timer_settime()` arms or disarms a timer and configures one-shot or periodic operation.

44. `timer_gettime()` retrieves the remaining interval before expiration.

45. `timer_delete()` releases the kernel resources associated with a timer.

46. POSIX timers support nanosecond precision through the `itimerspec` structure.

47. Timer notification can occur via signals, threads, or no notification at all (`SIGEV_NONE`).

48. `SIGEV_SIGNAL` integrates timer expiration into the signal-delivery subsystem.

49. Realtime signals are commonly used because they can carry timer-specific metadata.

50. `siginfo_t` identifies timer-generated signals using `SI_TIMER`.

51. Applications commonly place the timer ID in `sigev_value` so handlers can identify which timer expired.

52. POSIX timers solve a major deficiency of traditional interval timers through **overrun accounting**.

53. If multiple expirations occur before notification is processed, the kernel tracks the number of missed expirations.

54. `timer_getoverrun()` returns the count of additional expirations occurring beyond the delivered notification.

55. Even when realtime signals are used, timer expirations themselves are not queued indefinitely.

56. Instead of queuing unlimited signals, Linux records expirations using the overrun counter.

57. `SIGEV_THREAD` notification moves expiration handling out of signal context and into a newly created thread.

58. This avoids many async-signal-safety problems associated with signal handlers.

59. Linux additionally provides the `timerfd` API, exposing timers as file descriptors.

60. `timerfd_create()` creates a timer object that integrates naturally with the file-descriptor model.

61. `timerfd_settime()` and `timerfd_gettime()` mirror the semantics of POSIX timer operations.

62. Timer expiration is retrieved via `read()` rather than signal delivery.

63. Each read returns a 64-bit expiration count indicating how many expirations occurred since the previous read.

64. Therefore `timerfd` naturally handles overruns without requiring separate bookkeeping.

65. Because timerfd uses file descriptors, timers can be monitored alongside sockets, pipes, and files using `select()`, `poll()`, and `epoll()`.

66. This makes timerfd highly suitable for modern event-driven reactor architectures.

67. Unlike POSIX timers, timerfd notifications do not require signal handlers, signal masks, or realtime-signal management.

***

### Core Kernel Insight

**Chapter 23 describes Linux timing infrastructure evolving from signal-based interval timers (`setitimer`, `alarm`) to high-resolution clocks (`clock_gettime`), POSIX timers (`timer_create`), and finally file-descriptor-based timers (`timerfd`). The key kernel progression is from scarce per-process signal timers to scalable kernel timer objects that support nanosecond precision, overrun accounting, multiple clock sources, and seamless integration with event-driven I/O frameworks.**

### Chapter 24 — Process Creation (`fork`, `vfork`, Process Model) — Deep Kernel Notes

1. UNIX process management revolves around four fundamental operations: **fork → exec → wait → exit**.

2. `fork()` creates a new process by duplicating the calling process.

3. The original process is called the **parent**, while the newly created process is the **child**.

4. After a successful `fork()`, both parent and child continue execution from the instruction immediately following the `fork()` call.

5. The only reliable way to distinguish parent from child is the return value of `fork()`.

6. Parent receives the child's PID; child receives `0`.

7. `fork()` is one of the most elegant UNIX design decisions because process creation is separated from program execution.

8. Unlike "spawn" systems, UNIX allows arbitrary preparation work between `fork()` and `exec()`.

9. A child may either continue running the same program or replace itself using `execve()`.

10. Conceptually, `fork()` duplicates the parent's text, data, heap, stack, and process attributes.

11. From the programmer's perspective, the child initially appears as an exact clone of the parent.

12. Parent and child execute the same code but become independent processes.

13. After `fork()`, both processes have separate virtual address spaces.

14. Changes to variables in one process are invisible to the other.

15. Global variables are copied into the child.

16. Stack variables are copied into the child.

17. Heap allocations are copied into the child.

18. Both processes see identical memory contents immediately after the fork.

19. Modern Linux does **not** physically copy all memory during `fork()`.

20. Instead, Linux relies on **Copy-On-Write (COW)** memory management.

21. Parent and child initially share the same physical pages.

22. Shared pages are marked read-only by the kernel.

23. When either process attempts to modify a shared page, a page fault occurs.

24. The kernel allocates a new physical page and copies the original contents.

25. The faulting process receives the new private page.

26. This mechanism delays copying until modification actually occurs.

27. Copy-On-Write makes `fork()` efficient even for very large processes.

28. The optimization is particularly valuable because many forks are immediately followed by `exec()`.

29. If `exec()` occurs immediately, most shared pages are never copied.

30. Program text (code pages) is naturally shareable because it is read-only.

31. Parent and child therefore commonly share the same executable code pages.

32. `fork()` duplicates memory mappings but not necessarily the underlying physical memory.

33. The kernel must duplicate page tables even though pages themselves are shared.

34. Process creation therefore remains nontrivial despite Copy-On-Write.

35. Every successful `fork()` creates a unique PID for the child.

36. The child can discover its parent's PID using `getppid()`.

37. The parent can manage specific children using the PID returned by `fork()`.

38. Process creation may fail if system-wide process limits are exhausted.

39. User process limits (`RLIMIT_NPROC`) may also prevent further forks.

40. Open file descriptors are inherited across `fork()`.

41. The child receives duplicates of all file descriptor table entries.

42. These descriptors refer to the same underlying **open file descriptions**.

43. Consequently, parent and child share a common file offset.

44. Changes to file position by one process are immediately visible to the other.

45. Open file status flags (`O_APPEND`, etc.) are also shared.

46. This sharing frequently explains surprising file I/O behavior after fork.

47. Shared file offsets prevent processes from unknowingly overwriting one another.

48. However, they do not prevent interleaving of output.

49. Coordinated output still requires synchronization.

50. The shell uses `wait()` to prevent prompts and command output from mixing.

51. When descriptor sharing is undesirable, each process should close unused descriptors after the fork.

52. Process memory footprint can be controlled by forking before expensive computations.

53. The child may allocate memory freely and then terminate.

54. When the child exits, all memory allocated by the child disappears automatically.

55. This technique can eliminate memory leaks and heap fragmentation in isolated computations.

56. `vfork()` is a historical optimization predating efficient Copy-On-Write implementations.

57. `vfork()` creates a child without duplicating address-space metadata.

58. Parent and child temporarily share the exact same memory.

59. During this interval, the parent is suspended.

60. Parent execution resumes only after the child calls `exec()` or `_exit()`.

61. Modifying memory between `vfork()` and `exec()` modifies the parent's memory.

62. Returning from the calling function after `vfork()` can corrupt the parent's stack.

63. Calling ordinary library functions after `vfork()` is generally unsafe.

64. POSIX treats most behavior between `vfork()` and `exec()` as undefined.

65. The intended pattern is: `vfork() → exec()` or `vfork() → _exit()`.

66. Modern Copy-On-Write has largely eliminated the practical need for `vfork()`.

67. Consequently, `fork()` is usually preferred.

68. After a regular `fork()`, scheduling order is undefined.

69. Neither parent-first nor child-first execution is guaranteed.

70. Linux kernel policies regarding which process runs first have changed multiple times historically.

71. Programs that depend on a specific execution order contain race conditions.

72. Race conditions after `fork()` often appear extremely rarely and are difficult to debug.

73. Correct programs never assume which process executes first.

74. Synchronization primitives must be used whenever ordering matters.

75. One simple synchronization strategy uses signals.

76. A process blocks a synchronization signal before `fork()`.

77. One process later sends the signal after completing required work.

78. The other process waits using `sigsuspend()`.

79. Blocking the signal before `fork()` prevents missed notifications.

80. This pattern eliminates timing races between parent and child initialization.

81. More sophisticated synchronization mechanisms include pipes, semaphores, shared memory, and file locks.

82. Process creation is therefore not merely duplication—it is the foundation of UNIX concurrency.

***

### Core Kernel Insight

**`fork()` creates the illusion of process duplication while the kernel aggressively avoids real copying using Copy-On-Write. The essential challenge after process creation is no longer memory duplication but coordination: parent and child now execute independently, share some kernel-managed resources (open file descriptions), possess separate address spaces, and must use synchronization mechanisms whenever execution order matters.**
### Chapter 25 — Process Termination (`exit`, `_exit`, Exit Handlers) — Deep Kernel Notes

1. Process termination occurs either **normally** (`exit()`, `_exit()`, return from `main()`) or **abnormally** (fatal signal delivery).

2. The kernel's actual process-destruction mechanism is `_exit()`.

3. `_exit()` immediately terminates the process and never returns.

4. Process termination status is passed to the parent and later retrieved through `wait()`.

5. Although `_exit()` accepts an `int`, only the lowest 8 bits are preserved as the exit status.

6. By convention, exit status `0` means success.

7. Nonzero status values indicate application-defined failure conditions.

8. Shells reserve values above 128 for processes terminated by signals, making such values potentially confusing.

9. Most applications call `exit()` rather than `_exit()` directly.

10. `exit()` is a library-layer wrapper built on top of `_exit()`.

11. Before invoking `_exit()`, `exit()` performs user-space cleanup activities.

12. Exit handlers registered with `atexit()` and `on_exit()` execute before process destruction.

13. Exit handlers run in **reverse registration order** (LIFO order).

14. After exit handlers complete, all stdio output buffers are flushed.

15. Only after these user-space activities does `exit()` invoke `_exit()`.

16. Returning from `main()` is generally equivalent to calling `exit(return_value)`.

17. Falling off the end of `main()` under modern C standards is equivalent to `exit(0)`.

18. Once `_exit()` begins execution, process termination is guaranteed.

19. The kernel then performs extensive cleanup of process resources.

20. All open file descriptors are automatically closed.

21. Closing descriptors automatically releases any advisory file locks owned by the process.

22. Open directory streams are closed.

23. Open message catalog descriptors are closed.

24. Open character-conversion descriptors (`iconv`) are closed.

25. All attached System V shared-memory segments are detached.

26. Corresponding shared-memory attachment counts are decremented.

27. System V semaphore adjustments (`semadj`) are applied before the process disappears.

28. POSIX named semaphores are automatically closed.

29. POSIX message queues are automatically closed.

30. Memory locks created with `mlock()` or `mlockall()` are removed.

31. All memory mappings established via `mmap()` are destroyed.

32. The kernel releases all virtual-memory structures associated with the process.

33. If the process was controlling a terminal session, terminal-related cleanup occurs.

34. Session and process-group state may generate additional signals (`SIGHUP`, `SIGCONT`).

35. Resource reclamation occurs regardless of whether termination was normal or abnormal.

36. Exit handlers provide a controlled mechanism for application-specific cleanup.

37. Exit handlers execute only when termination occurs through `exit()`.

38. Exit handlers are skipped entirely if the process directly calls `_exit()`.

39. Exit handlers are also skipped when a fatal signal kills the process.

40. Therefore exit handlers cannot be relied upon for all cleanup scenarios.

41. `atexit()` registers a function taking no arguments and returning no value.

42. Multiple exit handlers may be registered.

43. The same handler may be registered multiple times.

44. Newly registered handlers are placed at the front of the execution stack.

45. Exit handlers may themselves register new handlers.

46. Newly registered handlers execute before remaining previously registered handlers.

47. If an exit handler never returns, subsequent handlers are never executed.

48. Calling `_exit()` inside an exit handler aborts further exit processing.

49. Abnormal termination inside an exit handler also prevents execution of remaining handlers.

50. Child processes created via `fork()` inherit copies of the parent's exit-handler registrations.

51. Successful `exec()` removes all registered exit handlers.

52. This occurs because the entire process image is replaced.

53. `on_exit()` extends the exit-handler model with arguments.

54. `on_exit()` handlers receive both the process exit status and a user-supplied pointer.

55. This permits context-sensitive cleanup behavior.

56. Unlike `atexit()`, `on_exit()` is nonstandard and therefore less portable.

57. Exit-handler registration is effectively unlimited under modern glibc implementations.

58. Exit handlers are frequently used by libraries that must perform automatic cleanup.

59. Signal handlers generally should not directly call `exit()` because `exit()` is not async-signal-safe.

60. Instead, signal handlers normally set flags that later cause normal program termination.

61. Interactions between `fork()` and stdio introduce a subtle but important correctness issue.

62. Stdio buffers reside entirely in user-space memory.

63. Therefore forked children inherit copies of all pending stdio buffers.

64. If buffered output exists at the time of `fork()`, both parent and child contain identical copies.

65. If both later call `exit()`, both flush their copies of the buffers.

66. The result is duplicated output.

67. This problem is most visible when stdout is block-buffered (e.g., redirected to a file).

68. Terminal output often avoids the problem because line buffering flushes data immediately.

69. Raw system calls such as `write()` are unaffected because they bypass stdio buffers.

70. `write()` places data directly into kernel-managed I/O buffers.

71. Kernel buffers are shared resources and are not duplicated during `fork()`.

72. One solution is calling `fflush()` before executing `fork()`.

73. Another solution is disabling stdio buffering entirely.

74. The most common solution is having the child terminate with `_exit()` instead of `exit()`.

75. `_exit()` prevents duplicate flushing because stdio cleanup is skipped.

76. This principle extends beyond buffering issues to exit handlers as well.

77. In fork-based programs, typically only one process should invoke `exit()`.

78. Additional child processes often terminate via `_exit()`.

79. This avoids duplicate execution of exit handlers.

80. It also prevents duplicate flushing of inherited stdio buffers.

81. Correct termination behavior therefore depends on understanding the distinction between user-space cleanup (`exit`) and kernel-level process destruction (`_exit`).

***

### Core Kernel Insight

**`exit()` and `_exit()` are not equivalent. `_exit()` is the kernel-level process termination primitive, while `exit()` is a user-space cleanup wrapper that executes exit handlers and flushes stdio buffers before invoking `_exit()`. Most subtle bugs in process termination—especially after `fork()`—arise from forgetting that user-space state (stdio buffers, exit-handler registrations) is duplicated into the child, whereas kernel-managed resources are ultimately cleaned up by `_exit()`.**
### Chapter 26 _ Monitoring CHild Processes

1. `wait()` is the fundamental kernel mechanism that allows a parent to collect state-change information from children and reap zombie processes.

2. A terminated child is not immediately destroyed; the kernel retains its PID, exit status, and resource-usage accounting until the parent performs a wait.

3. `wait()` blocks until any unwaited-for child terminates and returns that child's PID.

4. If no unwaited-for children exist, `wait()` fails with `ECHILD`, which is commonly used to detect that all children have been reaped.

5. During child reaping, the kernel transfers the child's CPU-time and resource-usage statistics to the parent's cumulative child accounting.

6. `waitpid()` extends `wait()` by allowing selection of a specific child or process group.

7. `waitpid(pid > 0)` waits for one specific child; `waitpid(-1, ...)` behaves like `wait()`.

8. `WNOHANG` enables nonblocking polling, allowing event-driven designs without parent suspension.

9. `WUNTRACED` reports children stopped by signals, while `WCONTINUED` reports children resumed via `SIGCONT`.

10. Child state changes visible to the kernel include: normal exit, signal termination, signal stop, and continuation after stop.

11. The wait status value is a compact kernel-encoded state object describing why a child changed state.

12. Applications must never inspect wait-status bits directly; portable code uses the `WIF*` and `W*STATUS` macros.

13. `WIFEXITED()` identifies normal termination and `WEXITSTATUS()` retrieves the exit code.

14. `WIFSIGNALED()` detects signal-caused termination and `WTERMSIG()` identifies the fatal signal.

15. `WCOREDUMP()` indicates that the kernel generated a core file during abnormal termination.

16. `WIFSTOPPED()` and `WSTOPSIG()` expose signal-driven process suspension.

17. `WIFCONTINUED()` indicates that a previously stopped process resumed execution.

18. If a signal handler wants cleanup while preserving true signal termination semantics, it should restore default disposition and re-raise the signal.

19. `waitid()` is a more modern interface that provides precise event filtering and richer child-state information via `siginfo_t`.

20. Unlike `waitpid()`, `waitid()` returns structured information (`si_pid`, `si_code`, `si_status`) instead of a packed wait status.

21. `WNOWAIT` allows inspection of child status without consuming the event, leaving the child waitable.

22. `wait3()` and `wait4()` additionally return kernel resource-usage statistics through `struct rusage`.

23. A child whose parent terminates becomes an orphan and is adopted by PID 1 (`init`/subreaper).

24. A child that exits before being waited upon becomes a zombie.

25. A zombie retains only minimal kernel metadata—PID, exit status, and accounting information.

26. Zombies cannot be killed by signals because the live execution context and address space no longer exist.

27. Zombie cleanup requires either a parent `wait()` call or adoption and reaping by `init`.

28. Excessive unreaped zombies can exhaust entries in the kernel process table and block creation of new processes.

29. Long-running daemons and servers must actively reap children to prevent zombie accumulation.

30. `SIGCHLD` is the kernel's asynchronous notification mechanism for child state changes.

31. The kernel sends `SIGCHLD` to a parent whenever a child terminates and optionally when it stops or continues.

32. Since standard signals are not queued, multiple child exits may collapse into a single pending `SIGCHLD`.

33. Therefore a `SIGCHLD` handler must loop:

    ```c
    while (waitpid(-1, NULL, WNOHANG) > 0)
        ;
    ```

34. Reaping only one child per signal delivery can leave zombies behind.

35. Correct `SIGCHLD` handlers must preserve and restore `errno` because `waitpid()` may overwrite it.

36. To avoid race conditions, `SIGCHLD` should often be blocked before child creation and later atomically unblocked via `sigsuspend()`.

37. `SA_NOCLDSTOP` suppresses `SIGCHLD` delivery when children are merely stopped instead of terminated.

38. Explicitly setting `SIGCHLD` disposition to `SIG_IGN` causes future child exits to be automatically reaped without zombification.

39. Under `SIG_IGN`, child termination status is discarded and later `wait()` calls cannot retrieve it.

40. `SIGCHLD` is unusual because explicitly setting `SIG_IGN` changes behavior compared to simply inheriting the default ignored disposition.

41. `SA_NOCLDWAIT` provides zombie-free behavior similar to `SIG_IGN` while still allowing implementation-defined `SIGCHLD` delivery.

42. Linux may still generate `SIGCHLD` when `SA_NOCLDWAIT` is used, but the child status is already discarded.

43. The parent's two design choices are: synchronous waiting (`wait*()`) or asynchronous reaping (`SIGCHLD` handler).

44. High-performance UNIX programs typically rely on `SIGCHLD` plus `waitpid(WNOHANG)` for scalable child management.

45. The kernel guarantees that a child's termination information persists until explicitly consumed, ensuring reliable parent-child synchronization.

46. **Core invariant:** every child must eventually be reaped—either by its parent or by `init`; otherwise zombies accumulate.

47. **Kernel design principle:** process termination is a two-phase operation—(1) execution ends, (2) parent collects status via `wait()`.

48. **Most important takeaway:** `fork()` creates children, `wait()/waitpid()` reclaim them, and `SIGCHLD` is the kernel notification path connecting the two.


### Chapter 27 — Program Execution (`execve`, `exec`, `system`) Deep Notes

1. `execve()` is the kernel primitive for **replacing a process's entire user-space image**; it does **not create a new process**, preserving the PID while destroying old code, heap, data, and stack.

2. After loading the executable, the kernel builds a new process image, sets up `argv`, `envp`, the initial stack, and transfers control to runtime startup code before `main()` executes.

3. The common process creation model is **fork() + execve()**: fork creates a new process, exec transforms that process into another program.

4. A successful `execve()` never returns because the calling program no longer exists; any return from `execve()` indicates failure.

5. `execve(pathname, argv, envp)` gives complete control over executable path, argument vector, and environment inherited by the new image.

6. `argv[0]` is purely convention; it usually contains the executable basename but can be any string supplied by the caller.

7. `/proc/PID/exe` is a kernel-maintained symbolic link exposing the executable currently associated with a process.

8. `execve()` preserves process identity (PID) but fundamentally rebuilds its address space and execution context.

9. During exec, if the executable has the **setuid/setgid bits**, the kernel updates effective UID/GID to match file ownership, enabling controlled privilege elevation.

10. After credential changes, effective IDs are copied into the saved-set IDs, enabling later privilege drop/restore mechanisms.

11. Common exec failures include: `EACCES`, `ENOENT`, `ENOEXEC`, `ETXTBSY`, and `E2BIG`.

12. `ENOEXEC` often indicates an invalid executable format or a script missing a valid `#!` interpreter line.

13. The kernel's exec path must validate executable format before constructing a new address space.

14. Linux primarily executes binaries using the **ELF format**, which defines program segments and loading behavior.

15. ELF supports a `PT_INTERP` program header that specifies a dynamic loader (e.g., `ld-linux.so`) responsible for loading shared libraries.

16. For dynamically linked binaries, the kernel may actually load the interpreter first, delegating subsequent program loading to it.

17. All other exec-family APIs are user-space wrappers built on top of the single kernel syscall `execve()`.

18. Naming convention: `v = argv vector`, `l = argument list`, `p = PATH search`, `e = explicit environment`.

19. `execvp()` and `execlp()` implement PATH-based executable lookup, mimicking shell command resolution.

20. PATH searching is performed in user space before invoking `execve()`.

21. If a filename contains `/`, PATH searching is bypassed and the path is used directly.

22. Using `execvp()` or `execlp()` in privileged (setuid/setgid) programs is dangerous because attackers may manipulate PATH.

23. Secure privileged programs should use explicit absolute paths and avoid PATH-based exec variants.

24. `execle()` and `execve()` allow callers to supply a fully controlled environment, improving security and reproducibility.

25. `fexecve()` executes a program from an already-open file descriptor rather than a pathname.

26. `fexecve()` eliminates the TOCTOU (time-of-check to time-of-use) race where a file may be replaced after verification but before execution.

27. Script execution is handled inside the kernel through special processing of files beginning with `#!`.

28. When the kernel detects `#!`, it extracts the interpreter pathname and optional argument from the first line.

29. The kernel internally converts script execution into execution of the interpreter.

30. Effective interpreter invocation becomes:

    ```
    interpreter-path [optional-arg] script-path arg...
    ```

31. The interpreter pathname from `#!` is not resolved using PATH; an absolute path is typically required.

32. Linux limits the `#!` line length to 127 characters.

33. The optional argument after `#!` is passed as one argument to the interpreter and is commonly used for flags such as `awk -f`.

34. If `execvp()`/`execlp()` encounter an executable text file without `#!`, they often fall back to launching `/bin/sh`.

35. Open file descriptors normally survive across exec, making file-descriptor inheritance a fundamental UNIX design feature.

36. Shell I/O redirection works because file descriptors remain open through exec.

37. `dup2()` is commonly used before exec to redirect standard input/output/error onto desired files or pipes.

38. The kernel supports per-descriptor `FD_CLOEXEC`, which automatically closes descriptors during successful exec.

39. `FD_CLOEXEC` prevents descriptor leaks into unrelated executables and is an important security mechanism.

40. Duplicating a descriptor via `dup()` or `dup2()` clears the close-on-exec bit on the duplicate.

41. During exec, user-installed signal handlers are discarded because the code implementing them disappears.

42. Signals with custom handlers are reset to `SIG_DFL`; signals already at `SIG_IGN` generally remain ignored.

43. Pending signals and the signal mask survive exec, allowing signals to remain queued for the new program.

44. Alternate signal stacks established via `sigaltstack()` are destroyed because the old memory image vanishes.

45. `system()` provides a high-level API that executes commands through a shell using internally managed fork/exec/wait logic.

46. Typical implementation is:

    ```
    fork()
      -> execl("/bin/sh", "sh", "-c", command)
    waitpid()
    ```

47. `system()` is convenient but expensive because it requires creating a shell process and often additional child processes.

48. Correct `system()` implementations block `SIGCHLD` and carefully manage `SIGINT`/`SIGQUIT` to avoid race conditions and improper signal delivery.

49. `waitpid()` must be used instead of `wait()` so `system()` collects the status of its own child rather than another child process.

50. Never use `system()`, `execvp()`, or `execlp()` in privileged programs; use `fork()` + explicit-path `execve()` for secure process creation.

### Chapter 28 - Program Creation and Execution in Detail
1. **Process accounting** is a kernel facility that records information about every process when it terminates, allowing post-mortem analysis even if the parent never monitored the process.

2. An accounting record is written **only at process exit**, making process accounting a termination-time auditing mechanism.

3. Historically, process accounting was used for billing users for CPU and resource consumption on multi-user UNIX systems.

4. The accounting subsystem preserves metadata such as command name, UID/GID, CPU usage, memory consumption, start time, and termination status.

5. Process accounting is useful for reconstructing process history after execution has completed.

6. Linux enables process accounting through the privileged `acct()` system call.

7. `acct(path)` activates accounting to the specified file; `acct(NULL)` disables accounting.

8. Accounting is typically enabled during system boot rather than by application programs.

9. Each terminated process generates exactly one accounting record in the accounting file.

10. Since Linux 2.6.10, multithreaded programs generate a single accounting record when the final thread exits.

11. Accounting records are written in **termination order**, not process creation order.

12. A system crash prevents accounting records from being written for processes still running.

13. Linux stores accounting records using the `struct acct` format defined in `<sys/acct.h>`.

14. The `ac_flag` field records special execution events such as fork-without-exec, superuser usage, signal termination, and core dumping.

15. `AFORK` indicates the process terminated without performing an `exec()` after a `fork()`.

16. `ASU` indicates the process exercised privileged (superuser) capabilities.

17. `AXSIG` indicates signal-caused termination.

18. `ACORE` indicates kernel generation of a core dump.

19. The kernel updates the `ac_comm` field on every `execve()`, so it stores the basename of the last executed program.

20. CPU usage fields (`ac_utime`, `ac_stime`) are stored as compressed clock ticks (`comp_t`) rather than plain integers.

21. `comp_t` uses a compact floating-point representation consisting of a 13-bit mantissa and a 3-bit base-8 exponent.

22. Process CPU times must be divided by `sysconf(_SC_CLK_TCK)` to obtain seconds.

23. Linux uniquely records full process termination status in `ac_exitcode`.

24. Process accounting captures resource-consumption history even after the process has completely disappeared from the process table.

25. The kernel can automatically suspend accounting when free disk space becomes scarce.

26. `/proc/sys/kernel/acct` controls accounting behavior through high-water, low-water, and check-frequency thresholds.

27. Accounting resumes automatically when free space rises above the configured high-water threshold.

28. Linux Version-3 accounting records extend the traditional format to overcome scalability limitations.

29. Version-3 records add explicit `PID` and `PPID` fields.

30. Version-3 widens UID and GID storage from 16 bits to 32 bits.

31. Version-3 stores elapsed execution time as a floating-point value rather than compressed ticks.

32. `clone()` is the fundamental Linux process-creation primitive underlying modern thread implementations.

33. Internally, `fork()`, `vfork()`, and `clone()` are all built on the same kernel process-creation framework (`do_fork()`).

34. Unlike `fork()`, a cloned child begins execution at a caller-supplied function instead of returning from the cloning point.

35. Because cloned tasks may share memory with the parent, clone requires an explicitly allocated child stack.

36. The `flags` argument determines which kernel resources are shared between parent and child.

37. `CLONE_VM` causes parent and child to share the same virtual address space, which is the defining property of threads.

38. `CLONE_FILES` causes both tasks to share a single file descriptor table.

39. `CLONE_FS` shares filesystem attributes such as current working directory, root directory, and umask.

40. `CLONE_SIGHAND` shares signal disposition tables while still keeping signal masks separate.

41. `CLONE_THREAD` places the child in the same thread group, allowing multiple kernel scheduling entities to appear as one process.

42. Linux thread groups implement the POSIX requirement that all threads report the same PID via `getpid()`.

43. Internally, each thread still has a unique TID obtainable via `gettid()`.

44. NPTL threads are implemented primarily through `clone()` with sharing flags such as `CLONE_VM`, `CLONE_THREAD`, `CLONE_FILES`, and `CLONE_SIGHAND`.

45. **Conceptually:** `fork()` creates an independent process, `vfork()` creates a temporary memory-sharing child for immediate exec, and `clone()` allows selective sharing of kernel-managed resources, making it the foundation of Linux threading.

### Chapter 29 - Threads (Introduction)

1. A **thread** is an execution stream within a process; multiple threads execute concurrently while sharing the same address space.

2. A traditional UNIX process is simply a process containing **one thread**.

3. All threads share the process-wide **text, data, bss, heap, and mapped memory regions**.

4. Each thread has its own **stack**, containing local variables, return addresses, and function call state.

5. On multiprocessor systems, threads can execute truly in parallel on different CPUs.

6. If one thread blocks in I/O, other threads in the same process can continue running.

7. Threads were introduced to overcome two limitations of process-based concurrency: expensive process creation and difficult data sharing.

8. Memory sharing between threads is essentially free because all threads operate in the same address space.

9. Sharing data between processes requires explicit IPC mechanisms such as pipes, sockets, message queues, or shared memory.

10. Creating a thread is typically much faster than creating a process because page tables and address spaces are not duplicated.

11. Process-wide attributes shared by all threads include PID, PPID, credentials, file descriptors, signal dispositions, timers, resource limits, working directory, and open files.

12. Thread-specific attributes include thread ID, stack, signal mask, `errno`, scheduling parameters, thread-local storage, and alternate signal stack.

13. Since thread stacks reside in the same address space, one thread can access another thread's stack through pointers, though this is dangerous and requires careful lifetime management.

14. The POSIX threading API (Pthreads) became the standard threading interface in POSIX.1c and SUS.

15. Pthreads defines opaque types such as `pthread_t`, `pthread_mutex_t`, `pthread_cond_t`, and `pthread_attr_t`.

16. In multithreaded programs, `errno` is no longer truly global; each thread has its own independent copy.

17. Unlike traditional UNIX system calls, Pthreads functions return **0 on success** and a positive error number on failure.

18. Pthreads programs on Linux must be compiled using the `-pthread` option.

19. Every process starts with a single thread called the **main (initial) thread**.

20. `pthread_create()` creates a new thread, which begins execution at a caller-specified start routine.

21. The creator thread and the newly created thread execute concurrently after `pthread_create()` returns.

22. Thread scheduling order after creation is unspecified; applications must use synchronization when ordering matters.

23. Arguments are passed to a thread start routine via a single `void *` parameter, commonly pointing to a structure.

24. A thread terminates by returning from its start function, calling `pthread_exit()`, being canceled, or when the process exits.

25. `pthread_exit()` terminates only the calling thread and allows it to return a value to another thread.

26. If the main thread calls `pthread_exit()`, the process remains alive while other threads continue executing.

27. Thread return values must not point to data on the terminating thread's stack because that memory becomes invalid.

28. Every thread possesses a unique POSIX thread ID represented by type `pthread_t`.

29. `pthread_self()` returns the ID of the calling thread.

30. Because `pthread_t` is opaque, thread IDs should be compared only using `pthread_equal()` and not using `==`.

31. POSIX thread IDs are distinct from Linux kernel thread IDs returned by `gettid()`.

32. `pthread_join()` waits for a specific thread to terminate and optionally retrieves its return value.

33. Joining a thread is conceptually similar to `waitpid()` for processes.

34. Unlike processes, threads have no parent-child hierarchy; any thread may join with any known thread.

35. POSIX intentionally provides no "join any thread" operation to preserve modular software design.

36. Threads are joinable by default and must eventually be joined to reclaim resources.

37. An unjoined terminated thread becomes a thread-level equivalent of a zombie.

38. Excessive thread zombies can exhaust thread resources and prevent creation of additional threads.

39. `pthread_detach()` marks a thread as detached, causing automatic cleanup on termination.

40. Detached threads cannot later be joined and their return values are discarded.

41. Thread attributes (`pthread_attr_t`) allow customization of stack size, detach state, scheduling policy, and related properties.

42. Threads can be created detached from the start using thread attributes.

43. From a design perspective, threads excel when fast communication and shared-state access are required.

44. Processes provide stronger isolation because memory corruption in one process typically cannot damage another process.

45. A bug in one thread can corrupt memory used by all other threads because they share the same address space.

46. Multithreaded programming requires synchronization primitives (mutexes, condition variables, etc.) to avoid race conditions.

47. Processes are easier to isolate and debug, while threads are generally more efficient and lightweight.

48. **Core takeaway:** threads are lightweight execution units within the same process that trade increased efficiency and easy data sharing for greater synchronization complexity and reduced fault isolation.

### Chapter 30 - Threads Sychnronization

1. **Thread synchronization** is required because multiple threads share the same address space and can concurrently access the same data.

2. A **critical section** is code that accesses shared resources and must execute atomically.

3. Unsynchronized access to shared variables causes **race conditions**, where program output depends on unpredictable thread scheduling.

4. The classic read-modify-write sequence:
   ```c
   loc = glob;
   loc++;
   glob = loc;
   ```
   is not atomic and can lose updates when executed concurrently.

5. Even seemingly simple operations such as `glob++` are often implemented as multiple machine instructions and are therefore not inherently thread-safe.

6. The result of a race condition is **nondeterministic behavior**—the same program may produce different outputs on different runs.

7. A **mutex** (mutual exclusion lock) is the primary mechanism for protecting shared resources.

8. A mutex has exactly two states: **locked** and **unlocked**.

9. At most one thread can own a mutex at any instant.

10. The standard mutex protocol is:
    ```text
    lock mutex
    access shared resource
    unlock mutex
    ```

11. Mutex locking is **advisory**, meaning correctness depends on all threads cooperating and following the locking rules.

12. A mutex is represented by the type `pthread_mutex_t`.

13. Statically allocated mutexes can be initialized using:
    ```c
    pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;
    ```

14. `pthread_mutex_lock()` acquires a mutex, blocking if another thread currently owns it.

15. `pthread_mutex_unlock()` releases a mutex previously locked by the calling thread.

16. Only the thread that owns a mutex may unlock it.

17. Protecting shared variables with mutexes transforms unsafe concurrent access into a serialized, deterministic operation.

18. `pthread_mutex_trylock()` performs a nonblocking lock attempt and returns `EBUSY` if the mutex is already locked.

19. `pthread_mutex_timedlock()` waits only until a specified absolute timeout and returns `ETIMEDOUT` if acquisition fails.

20. On Linux, uncontended mutex operations usually execute entirely in user space without system calls.

21. Mutexes are typically implemented using **futexes** (fast user-space mutexes).

22. System calls occur only when lock contention requires kernel involvement.

23. Mutexes are therefore significantly faster than file locks or System V semaphores in most cases.

24. A **deadlock** occurs when two or more threads wait forever for locks held by each other.

25. The classic deadlock pattern arises when threads acquire the same set of mutexes in different orders.

26. The simplest deadlock-prevention strategy is a **global lock hierarchy** in which all threads acquire mutexes in a fixed order.

27. Another strategy is **try-and-back-off**, which uses `pthread_mutex_trylock()` and retries when lock acquisition fails.

28. Dynamically allocated mutexes must be initialized with:
    ```c
    pthread_mutex_init()
    ```

29. Unneeded dynamically initialized mutexes should be destroyed using:
    ```c
    pthread_mutex_destroy()
    ```

30. Destroying a locked mutex or one that may still be used by other threads results in undefined behavior.

31. Mutex behavior is controlled by mutex attributes, particularly the **mutex type**.

32. `PTHREAD_MUTEX_NORMAL` provides no error checking and may deadlock on self-locking.

33. `PTHREAD_MUTEX_ERRORCHECK` detects misuse such as relocking an already owned mutex and returns errors instead of deadlocking.

34. `PTHREAD_MUTEX_RECURSIVE` allows the same thread to lock the mutex multiple times using an internal lock count.

35. Condition variables solve a different problem from mutexes: **state notification** rather than mutual exclusion.

36. A **condition variable** allows one thread to sleep until another thread reports that some shared state has changed.

37. Condition variables are represented by type:
    ```c
    pthread_cond_t
    ```

38. Statically initialized condition variables use:
    ```c
    pthread_cond_t cond = PTHREAD_COND_INITIALIZER;
    ```

39. A condition variable is always associated with a mutex protecting the shared state being monitored.

40. `pthread_cond_signal()` wakes at least one waiting thread, while `pthread_cond_broadcast()` wakes all waiting threads.

41. Condition variables themselves store **no state**; they merely communicate that state may have changed.

42. `pthread_cond_wait()` atomically:
    ```text
    unlocks mutex
    sleeps on condition variable
    wakes when signaled
    re-locks mutex
    ```

43. This atomic unlock-and-sleep operation prevents lost wake-up races between waiter and signaler.

44. Every condition variable has an associated **predicate** (a Boolean condition over shared data) that determines whether progress can be made.

45. Waiting must always be written as:
    ```c
    while (!predicate)
        pthread_cond_wait(...);
    ```
    not with an `if` statement.

46. The predicate must be rechecked because other threads may alter the shared state before the awakened thread reacquires the mutex.

47. The loop also protects against **spurious wakeups**, which POSIX explicitly permits.

48. `pthread_cond_timedwait()` adds a timeout and returns `ETIMEDOUT` if the signal never arrives.

49. In producer-consumer designs, producers update shared state under a mutex and then signal a condition variable.

50. Consumers wait on the condition variable whenever the shared state indicates no work is available.

51. Condition variables eliminate inefficient polling loops that repeatedly check a shared variable.

52. The producer-consumer pattern demonstrates the complementary roles of synchronization primitives:
    * **Mutex:** protects shared data.
    * **Condition variable:** notifies about state changes.

53. Multiple threads waiting on the same condition variable must use the same associated mutex.

54. Applications can emulate a "join any thread" operation by combining thread state variables, a mutex, and a condition variable.

55. Dynamically allocated condition variables require:
    ```c
    pthread_cond_init()
    ```

56. They should be cleaned up with:
    ```c
    pthread_cond_destroy()
    ```

57. A condition variable may be destroyed only when no threads are waiting on it.

58. **Core mutex invariant:** every access to a shared mutable resource must occur while holding its protecting mutex.

59. **Core condition-variable invariant:** wait in a loop that checks a predicate protected by the associated mutex.

60. **Most important takeaway:** mutexes provide **mutual exclusion**, while condition variables provide **event notification**; together they form the foundation of safe thread synchronization in POSIX threads.
